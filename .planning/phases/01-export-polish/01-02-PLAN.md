<objective>
Extend the v5 JSON contract (top-level `custom_properties` dict) and the Blender-side consumer so that Flame-side callers can stamp caller-supplied metadata onto the baked Blender camera. This is the schema-level scaffolding that Plan 04's handler rework will populate with `forge_bake_action_name` + `forge_bake_camera_name`.

Scope (per CONTEXT.md D-10, D-11, D-12):
- Add an `Optional[dict]` passthrough to `fbx_to_v5_json` (Option A from PATTERNS.md §1.7 — 4-line additive change; keeps JSON write atomic; honors "extend the v5 contract, do not bypass").
- Extend `_stamp_metadata` in `bake_camera.py` to apply the dict onto the camera data-block.
- Regression test proving round-trip.

Explicitly OUT of scope here:
- Calling sites in the Flame hook (Plan 04 does that).
- Version stamping, frame-range stamping, any property beyond what the caller passes in (POLISH-02 is deferred).
- Changing existing metadata keys stamped by `_stamp_metadata` (`forge_bake_version`, `forge_bake_source`, `forge_bake_scale`, `forge_bake_input_path` stay).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller Python → fbx_to_v5_json | Caller supplies a dict that gets serialized to JSON on disk. |
| v5 JSON on disk → bake_camera.py (bpy subprocess) | Blender subprocess reads the JSON and assigns values onto bpy ID-block custom properties. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | v5 JSON file on disk between writer (Flame side) and reader (Blender side) | accept | Temp dir is per-invocation under `tempfile.mkdtemp()` (created by Plan 04); file is short-lived, cleaned on success. Attacker with write access to `~/forge-bakes/` or the temp dir already has the local user's privileges (internal tool, PROJECT.md security posture). |
| T-02-02 | Injection | custom_properties dict containing unexpected types | mitigate | `_stamp_metadata` assigns `cam_data[key] = value` which bpy validates; a key like `"../../evil"` is inert as a dict key on an ID-block (not a filesystem path). Value types outside str/int/float/list/dict would raise a bpy TypeError, surfacing as a bake CalledProcessError in Plan 04 — fails loud. |
| T-02-03 | Information Disclosure | stamped custom properties leak Flame node names into .blend | accept | The whole point of EXP-04 is to carry Flame Action + camera names into the .blend for the return trip. No PII beyond user-chosen node names. |
| T-02-04 | Denial of Service | maliciously large dict | accept | Internal tool, user-owned invocation; caller controls what gets stamped. Not a realistic threat model for this phase. |
</threat_model>
