    ```python
    _SANITIZE_NAME_RE = re.compile(r"[^A-Za-z0-9._-]")


    def _sanitize_name_component(name: str) -> str:
        """Make a Flame node name safe to embed in a filesystem path.

        The export handler builds `~/forge-bakes/{action}_{cam}.blend` from
        Flame Action and camera names. Flame permits characters in node names
        that are hostile to paths: `/`, `\\`, spaces, shell metacharacters
        (`$`, `` ` ``, `;`, `&`, `|`, `>`, `<`, `*`, `?`, quotes), and
        control bytes. Embedding those raw into a path risks path traversal
        (`..`-chains), shell confusion if the path is ever logged into a
        shell-expanded context, and plain OS-level write failures.

        This helper is ONLY for the filesystem path component. The original,
        unsanitized name is still stamped into the v5 JSON's
        `custom_properties` (per D-11) so the Blender-side "Send to Flame"
        return trip can look up the correct Flame Action.

        Rules:
          - Whitelist `[A-Za-z0-9._-]`; replace all other runes with `_`.
          - Truncate to 64 chars (keeps combined filename < 255-char FS limit).
          - If the result is empty, return `"unnamed"` so the caller never
            builds a degenerate `_{cam}.blend` or `{action}_.blend` path.

        Never raises; `None` and non-str inputs coerce via `str()`.
        """
        safe = _SANITIZE_NAME_RE.sub("_", str(name))[:64]
        return safe if safe else "unnamed"
    ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Flame's Python runtime (trusted) → local filesystem (`~/forge-bakes/`, `tempfile.mkdtemp` dir) | The hook writes two filesystem artifacts based on Flame node names. Node names come from the user (trusted) but may contain characters hostile to paths. |
| Flame's Python runtime → Blender subprocess (trusted, user-installed) | `subprocess.Popen` passes argv lists; no shell interpolation. Blender is a trusted local binary (verified via `blender_bridge.resolve_blender_bin()`). |
| `.planning/config.json` on disk → Python `json.load` (via Plan 03 helper) | User-owned repo file; attacker with write access already has local-user privileges (internal VFX tool, per PROJECT.md security posture). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering / Injection | Action/camera names as filesystem path components (`~/forge-bakes/{action}_{cam}.blend`) | mitigate | Task 2's `_sanitize_name_component` whitelists `[A-Za-z0-9._-]`, truncates to 64 chars, and falls back to `"unnamed"`. `/`, `\`, `..`, spaces, shell metacharacters all become `_`. Pre-mitigation severity: medium (a Flame node literally named `../../etc/passwd` could write outside `~/forge-bakes/`). Post-mitigation: low. |
| T-04-02 | Tampering / Injection | subprocess argument handling for Blender launch | mitigate | All subprocess calls in `_launch_blender_on_blend` and elsewhere use argv lists (`subprocess.Popen([...])`); `shell=True` is never used. Post-mitigation: low. |
| T-04-03 | Information Disclosure | Temp dir path preserved on failure and embedded in error dialogs | accept | Local-only tool, single-user workstation, temp dir lives under the OS temp root (`/tmp` or `TMPDIR`) with user-only permissions. Disclosing a `/tmp/forge_bake_xyz` path to the user via a dialog is intentional (D-14: user needs it to triage the failure). Severity: low. |
| T-04-04 | Denial of Service | Disk exhaustion from repeated exports | accept | Overwrite-on-collision policy (D-05) keeps `~/forge-bakes/` bounded at one .blend per `(Action, camera)` pair. On success the temp dir is deleted; on failure the user is expected to clean up. Severity: low. |
| T-04-05 | Tampering (TOCTOU / symlink) | `os.makedirs("~/forge-bakes", exist_ok=True)` does not resolve or verify symlinks | accept | Trust model is single local user with ownership of their home directory. A symlink at `~/forge-bakes/` could redirect the write, but only the user (or something running as the user) could plant it. PROJECT.md security posture explicitly treats artifacts as living on trusted storage. Severity: low. Documented here so a future security review has prior art. |
| T-04-06 | Information Disclosure | Custom properties (`forge_bake_action_name`, `forge_bake_camera_name`) leak Flame node names into .blend files | accept | The whole point of EXP-04 is carrying the names across the boundary. Node names are not PII; they are artist-chosen identifiers. Phase 2's return-trip logic depends on this disclosure. Severity: low. |
| T-04-07 | Denial of Service | Blender launch subprocess fails or hangs | mitigate | Launch failure is caught and surfaced via a warning dialog + `reveal_in_file_manager` fallback (D-03). `subprocess.Popen` does not wait on the child (no `.communicate()` call), so a hung Blender does not block Flame. Severity: low. |
| T-04-08 | Elevation of Privilege | Blender binary spoofing via `blender_bridge.resolve_blender_bin()` | accept | The resolver honors `FORGE_BLENDER_BIN` env var and PATH. Attacker would need to modify the user's env or PATH — same privilege level as the victim. Internal tool, trusted workstation. Severity: low. |

**Post-mitigation check:** no threats remain at HIGH severity. Task 2 (sanitize) and Task 4 (argv-only subprocess) address the two realistic medium-severity threats. All other threats are LOW and documented as accepted under the internal-VFX-tool security posture.
</threat_model>
