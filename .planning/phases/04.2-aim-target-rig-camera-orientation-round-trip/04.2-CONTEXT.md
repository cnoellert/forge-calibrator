# Phase 4.2: Aim/Target-rig camera orientation round-trip - Context

**Gathered:** 2026-04-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Aim/Target-rig Flame cameras must round-trip through Blender with their world-space orientation preserved. Today the orientation is lost: a live probe against the user's reference camera (`Camera1` in `action9`) confirmed that an aim-rig camera has `rotation=(0,0,0)` and its orientation is expressed via `aim`, `up`, `roll` PyAttributes instead. Flame's `action.export_fbx(bake_animation=True)` DOES preserve this in the FBX — it writes a separate `Model::<Cam>_aim` Null with the aim target's translation, a `C: "OP", <aim_null_id>, <cam_id>, "LookAtProperty"` connection, and the camera's `UpVector` + `Roll` as animatable properties. Our ASCII FBX parser (`forge_flame/fbx_ascii.py:_extract_cameras`) reads only `Model::...,Camera` nodes + their Lcl Translation / Lcl Rotation / FieldOfView AnimCurves, so the aim semantics are silently discarded. By the time the v5 JSON reaches `bake_camera.py`, the camera has position + focal but no usable rotation, and the Blender camera lands pointing down -Z from its position regardless of where the Flame camera was actually aimed.

Phase 4.2 fixes this at the parser layer. We teach `fbx_ascii` to detect the LookAtProperty connection, resolve the aim Null's per-frame translation, read the camera's per-frame UpVector + Roll, and compose a per-frame Euler (rx, ry, rz) in Flame's ZYX-with-sign-flip convention — which is already nailed by `forge_core.math.rotations.compute_flame_euler_zyx`. The resolved Euler lands in the existing `rotation_flame_euler` field of the v5 JSON. Blender side and return-trip side both stay unchanged — the existing import path already forces Free-rig on landing (`target_mode.set_value(False)` attempts), matching the user's principle "rig type doesn't matter, transform does".

Scope anchor: parser read-side change only. No Flame-state mutation on export, no new v5 JSON schema fields, no Blender-side changes, no return-trip changes. One adjacent latent bug is in scope (the broken `cam.target_mode.set_value(False)` calls that would raise on aim-rig cameras) because it's the same code-area and would surface during live UAT of this phase.

</domain>

<decisions>
## Implementation Decisions

### Conversion strategy (core)
- **D-01:** Approach **B** — read the aim-rig semantics out of the FBX Flame already writes, and resolve them to Euler rotation inside the FBX→JSON converter. No Flame-side rig-toggle, no pre-export state mutation, no Blender-side constraint machinery.
- **D-02:** Implementation site is `forge_flame/fbx_ascii.py` — specifically the read path that leads from `fbx_to_v5_json` through `_extract_cameras`. The resolver must run before the per-frame `rotation_flame_euler` list is composed so the JSON carries the resolved Euler, not stale (0,0,0).
- **D-03:** Approach A (toggle `cam.target_mode=False` in Flame) is **rejected as infeasible** on Flame 2026.2.1 — live probe of `Camera1.attributes` returned a 39-element list with no `target_mode` entry, and `dir(cam)` shows no rig-toggle method on `PyCoNode`. The existing `target_mode.set_value(False)` calls in the codebase are latent bugs (see D-10); they are NOT a working escape hatch.

### Math pipeline
- **D-04:** Build the 3×3 camera-to-world rotation matrix from `(position, aim, up, roll)` using the standard look-at construction:
  - `forward = normalize(aim - position)` (view direction, world-space)
  - `right = normalize(cross(forward, up_world))` (world-space right, orthogonalised)
  - `up_cam = cross(right, forward)` (orthonormal up in camera frame)
  - Apply roll about the forward axis via `axis_angle_rotation(forward, -roll_deg)` to `right` and `up_cam` (sign convention verified against probe during planning — see D-07).
  - Assemble `R = [right | up_cam | -forward]` (or Flame's actual column convention — **to be verified during planning**, see D-07).
- **D-05:** Feed the resulting matrix to `forge_core.math.rotations.compute_flame_euler_zyx` to produce (rx, ry, rz) degrees in Flame's ZYX-with-X,Y-negated convention. This is the same function the 2VP solver uses for Free-rig solves — reusing it keeps the rotation convention in one place.
- **D-06:** The new math helper lives in `forge_core/math/rotations.py` alongside `compute_flame_euler_zyx` (not in `forge_flame/fbx_ascii.py`) — it's pure numpy, reusable from Blender scripts, and belongs with the other rotation-convention code per the module's stated purpose ("Other hosts will need these same helpers to move camera data into or out of Flame's coordinate frame").

### Verification of math convention
- **D-07:** Before landing the resolver, the planner must verify the exact column convention (right/up/forward or right/up/-forward) and Roll sign against **two known-answer cases** from the live probe:
  - Camera1's current static aim: position=(0, 57.77, 2113.30), aim=(0.355, 57.13, 2093.32), up=(0, 30, 0), roll=-1.25°. The resolved rotation must produce a Blender camera pointing at the world-space aim point when round-tripped.
  - A second free-rig camera (e.g., any existing Camera Match solve with known rotation) — run it through the new path in "force aim-rig resolution" mode to confirm matrix math doesn't contradict the proven `compute_flame_euler_zyx` output for a camera whose rotation was solved directly.
  - If the acceptance test (round-trip orientation within 0.1° on all three Euler axes) fails, the column convention or Roll sign is wrong — the planner must nail these via a one-frame probe before committing the resolver.

### Animation scope
- **D-08:** Support **fully animated** aim-rig — per-frame aim target + per-frame up + per-frame roll. The FBX emits these as animatable properties (`A` / `A+` markers on the probe), and the Null's `Lcl Translation` is keyframable via standard AnimCurveNodes. Per-frame composition uses the same mechanism `_extract_cameras` already uses for per-frame position/rotation/fov (follow AnimCurveNode → d|X / d|Y / d|Z KeyTime/KeyValueFloat).
- **D-09:** If any of (aim, up, roll) is static (single keyframe, or Properties70 direct value with no AnimCurve), treat it as a constant across the whole frame range — don't require artifical keyframes to be synthesised.

### Latent `target_mode` bug (scope expansion flag)
- **D-10:** Fix the broken `cam.target_mode.set_value(False)` calls in the same phase:
  - `flame/camera_match_hook.py:1511` (in the solve-apply path)
  - `forge_flame/camera_io.py:234` (in `import_json_to_flame_camera`)
  These currently resolve `cam.target_mode` to `None` on Flame 2026.2.1 (confirmed via live probe — no `target_mode` in the 39-element `cam.attributes` list), then call `.set_value(False)` on None, which would raise `AttributeError`. The only reason these lines haven't caused observed bugs is that the solve-apply path typically runs on newly-created cameras that may carry a Python attribute of `None` that swallows the call silently in some code paths, or these paths simply haven't been exercised on an aim-rig camera. The fix is to remove the calls entirely (the rest of the solve/import flow sets position + rotation + fov explicitly, which is sufficient to define a Free-rig pose) — or replace with a conditional that checks `hasattr` on the real `cam.attributes` list.
- **D-11:** The fix lands as a separate commit within phase 4.2's plan (not a separate phase) — it's the same code area, same root cause (Flame API assumption mismatch), and will surface during live UAT of the aim-rig round-trip when the test camera is applied back.

### v5 JSON schema
- **D-12:** **No schema change.** The resolved Euler lands in the existing `frames[].rotation_flame_euler` field. No aim/up/roll metadata is preserved in the JSON — the user's principle "rig type doesn't matter, transform does" makes metadata-preservation optionality unnecessary. Return-trip continues to land as a Free-rig camera via the existing import path.
- **D-13:** If we ever need aim-rig reconstruction on return (not planned), the JSON schema can be extended non-breakingly by adding optional `aim_rig_metadata` fields — that's a future phase, not now.

### Fallback behavior
- **D-14:** **Fail loud** when the aim-rig math can't resolve — raise `ValueError` with a specific message identifying which invariant failed (view vector near-zero, up parallel to view, missing LookAtProperty connection on a camera that has no rotation curve either). This matches the fidelity-discipline pattern 04.1 established (WR-01, WR-02 hotfixes: unknown fps label raises; unsupported filmback raises) and upholds the core value — never silently ship wrong orientation.
- **D-15:** The loud-failure threshold is: `|forward| < 1e-6` (aim coincident with position), `|cross(forward, up)| < 1e-6` (up parallel to view). These values are chosen to match the existing `cb <= 1e-6` gimbal-lock threshold in `compute_flame_euler_zyx`; using the same tolerance keeps the numeric regime consistent across the rotation pipeline.

### Claude's Discretion
- **D-07 verification mechanics** — whether the verify step is a one-shot manual probe via forge-bridge, a unit test fixture, a `tests/conftest.py` helper, or a CLI harness. Planner picks.
- **Parser internal structure** — whether the aim-rig resolver is a new module (`forge_flame/fbx_aim_resolver.py`), a private helper inside `fbx_ascii.py`, or an extension of `_extract_cameras`. Planner picks based on test-footprint and readability.
- **Error message wording** for D-14 — as long as the message names the failing invariant.
- **D-10 fix shape** — delete the broken lines, or replace with a guarded `hasattr` check. Both are acceptable; planner picks based on a grep of all call sites and whether any non-obvious future extension would want the guard.

## Existing Code Insights

### Reusable Assets
- **`forge_core.math.rotations.compute_flame_euler_zyx`** — 3×3 matrix → Flame Euler (rx, ry, rz) degrees. Handles gimbal lock (ry = ±90°) via a pinned-rx branch with `cb <= 1e-6` threshold. This is the final stage of the aim-rig resolver; any look-at matrix we build lands here.
- **`forge_core.math.rotations.flame_euler_to_cam_rot`** — inverse (Euler → matrix). Not directly needed for 4.2, but proves the convention is fully invertible (the 264-test suite validates this).
- **`forge_flame.fbx_ascii` Connections graph** — the parser already walks `C: "OP", <src>, <dst>, ...` connections for AnimCurveNode → Model bindings. Walking `LookAtProperty` connections uses the same machinery — we just add another property name to the dispatch.
- **`forge_flame.fbx_ascii` AnimCurveNode read path** — existing code reads `KeyTime` + `KeyValueFloat` on `d|X`/`d|Y`/`d|Z` sub-nodes. The aim Null's translation curves have identical structure (probe line 649: `C: "OP", 105553322837504, 5155773440, "Lcl Translation"`). Same reader, applied to a different Model ID.
- **`tests/fixtures/forge_fbx_probe.fbx`** — existing fixture with a Perspective camera + aim-Null + LookAtProperty connection; can be used as a secondary unit-test fixture.

### Established Patterns
- **Pure-math helpers in `forge_core/math/`** — no numpy aliasing, no cv2, no Flame imports. The new `rotation_matrix_from_look_at` helper follows this pattern (numpy-only, testable standalone).
- **Duck-type node identification** (`fbx_io.py:65`) — `Model::...,"Camera"` headers identify camera Models; analogous lookup for `Model::..._aim` / `Null` type headers plus the LookAtProperty connection source. Same shape of code.
- **Fail-loud on missing invariants** (04.1 WR-01/WR-02 pattern, committed 2026-04-22 in d68b77c / 094c3b3) — raises `ValueError` with an invariant-named message when a required structural element is missing. D-14 continues this pattern.
- **Extensive module docstrings explaining "Why this module exists"** — the new look-at helper should land with a docstring explaining why it lives in `forge_core.math.rotations` (per D-06) and reference memory/flame_rotation_convention.md.
- **Per-frame composition in `fbx_to_v5_json`** — existing code already resolves per-frame values from AnimCurveNodes for position, rotation, focal. The aim-rig resolver plugs in at the same stage.

### Integration Points
- **Parser entry point:** `fbx_to_v5_json` in `forge_flame/fbx_ascii.py` — currently calls `_extract_cameras`, then composes per-frame position + rotation + focal into `frames[]`. The aim-rig branch fires when:
  1. The target Model::Camera has a `LookAtProperty` connection as destination, AND
  2. No Lcl Rotation AnimCurves exist for it (or they're identically zero) — per the probe, aim-rig cameras emit `Lcl Rotation` = 0,0,0 effectively.
  When both conditions hold, resolve rotation via the look-at math instead of reading the (zero) Lcl Rotation curve.
- **Test fixture for verification:** `.planning/phases/04.2-.../04.2-aimrig-probe-camera1.fbx` (copied from the live probe) is the known-answer input. A matching expected-output JSON (with the resolved Euler) becomes the regression fixture.
- **Hook-level change:** none — `flame/camera_match_hook.py:2244-2249` already calls `fbx_io.export_action_cameras_to_fbx` without rig awareness; the FBX it produces already contains the aim-rig structure; all that changes is what the downstream FBX→JSON converter does with it.
- **Blender-side change:** none — the v5 JSON schema is unchanged.

<deferred>
## Deferred Ideas

- **Aim-rig preservation on return trip.** If a user wants their returned Blender-edited camera to land as an aim-rig in Flame (rather than Free-rig), that's a separate semantic round-trip. Would require (a) the return path detecting a designated "this was originally aim-rig" marker, (b) Blender-side extraction of aim/up/roll (presumably via a Track-To constraint or Empty target), (c) forge-bridge code to set the returned camera's aim/up/roll PyAttributes and arrange the Null in the Action node graph. Not scoped for 4.2 per D-12. Route to v2 milestone if ever requested.
- **Stereo-rig aim semantics.** `Camera1.attributes` includes `interaxial`, `zero_parallax`, `toe_in_adjust`, `film_offset_left`, `film_offset_right` — stereo-rig cameras have additional state. Phase 4.2 deliberately scopes to the per-camera aim-rig resolution; stereo-rig aim interplay (if any) is a separate investigation. Not surfaced by the original 4.2 problem statement and not touched by B.
- **Fixing `target_mode` calls elsewhere.** If future code (v2 scope: aim-rig preservation on return) needs a reliable "set Free-rig" primitive, the right approach is to probe Flame 2026.2.1 for whatever the real mechanism is (whether it's setting `rotation` explicitly, setting `aim`/`up`/`roll` to sentinel values, or something else) and centralise it in `forge_flame`. Out of scope for 4.2's parser-only fix per D-10.
- **Runtime closure of DOC-01 (Phase 4 deferral).** Phase 4's DOC-01 smoke-test runtime closure was deferred pending 4.1's empty-camera-bake fix (per `PROJECT.md` last-updated note 2026-04-22). 4.1 has shipped; re-running the smoke test with a static aim-rig camera as one of the test inputs would close DOC-01 runtime-wise. Not 4.2's job, but the planner should flag this so the next Phase 4 resume knows the prerequisite is ready.
