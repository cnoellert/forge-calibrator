# Phase 04.2: Code Review Report

**Reviewed:** 2026-04-24

## Summary

This phase delivers the Flame Aim/Target-rig camera orientation round-trip,
which is core to the project's stated value: geometric fidelity end-to-end.
The numerical core — `rotation_matrix_from_look_at` — is well-tested with
a Camera1 live-probe known-answer gate, a free-rig cross-check against the
already-proven `flame_euler_to_cam_rot` inverse, and explicit fail-loud
coverage for degenerate forward/up inputs. The full test suite (380 + 2
skipped) passes, and the 107 Phase-4.2-relevant tests all pass.

Two latent issues warrant attention before Blender round-trip verification:

1. **UpVector / Roll precedence mismatch in `_extract_cameras`** — the
   inline comment says NodeAttribute is authoritative ("live-probe values")
   but the code unconditionally overwrites with the Model's value when
   present. The current fixture doesn't exercise this (Model never carries
   UpVector / Roll), but any future Flame export that stamps both would
   silently pick the wrong value. This is a correctness hazard for the
   project's stated core value.

2. **UpVector extraction is untested in `TestAimRigFixture`** — the
   fixture doesn't carry a `UpVector` in the Model or NodeAttribute
   Properties70 (only the template default in `ObjectType` definitions),
   so the parser's UpVector-fetch code path is never exercised against a
   non-default value. The test's `CAMERA1_UP = (0, 30, 0)` cross-check
   succeeds only because it's parallel to the default `(0, 1, 0)`.

The `target_mode.set_value()` / `target_mode.get_value()` removals in
`camera_match_hook.py` and `camera_io.py` are clean — no orphaned
references remain in the reviewed files (the only remaining reference is
in `flame/rotation_diagnostic.py` which is already guarded by
`hasattr(cam, "target_mode")` and is outside this review's scope).

## Warnings

### WR-01: UpVector / Roll precedence contradicts docstring comment

**File:** `forge_flame/fbx_ascii.py:625-635` (UpVector), `forge_flame/fbx_ascii.py:646-654` (Roll)

**Issue:** The inline comment states:

> Static up from Properties70 "UpVector" — NodeAttribute first
> (authoritative for live-probe values), then Model (template
> default). Later assignment wins; we check NA first then
> overwrite with Model only if Model has a non-template value.

But the code unconditionally overwrites `static_up_vector` with the Model
value whenever `_get_property70(model, "UpVector")` returns a non-None
result — there is no "non-template" check. If Flame (or another DCC) ever
emits `UpVector` on both the NodeAttribute and the Model, the Model value
would win, directly contradicting the stated "NA is authoritative" intent.

The current fixture does NOT exercise this: the Model's Properties70
(lines 364-370 of `forge_fbx_aimrig.fbx`) contain `RotationActive`,
`ScalingMax`, `LookAtProperty`, `DefaultAttributeIndex`, `Lcl Translation`
— but no `UpVector` or `Roll`. So the test passes by accident: Model lookup
returns None, NA value survives. The Roll precedence (lines 646-654) has
the same structural issue.

**Fix:** Either tighten the code to honour the stated "NA is authoritative"
precedence by reading the Model only as a fallback when NA didn't yield a
value, or update the comment to match the current behaviour. Given the
stated "live-probe values" semantics, the former matches intent:

```python
# Static up from Properties70 "UpVector" — NodeAttribute is
# authoritative (live-probe values), Model is fallback (template default).
up_found = False
if na is not None:
    p = _get_property70(na, "UpVector")
    if p and len(p) >= 3:
        static_up_vector = (_as_float(p[0]), _as_float(p[1]), _as_float(p[2]))
        up_found = True
if not up_found:
    p = _get_property70(model, "UpVector")
    if p and len(p) >= 3:
        static_up_vector = (_as_float(p[0]), _as_float(p[1]), _as_float(p[2]))
```

Apply the same pattern to the Roll block at lines 646-654. Add a test
fixture (or mutate the existing one in-memory) that exercises both
NA-only, Model-only, and both-present cases to lock the precedence.

### WR-02: Test fixture does not exercise UpVector extraction

**File:** `tests/test_fbx_ascii.py:481-586`, `tests/fixtures/forge_fbx_aimrig.fbx`

**Issue:** The aim-rig fixture carries `Roll` on `NodeAttribute::Camera1`
(line 343) but does NOT carry `UpVector` anywhere except the FBX template
definitions (line 217, inside `ObjectType: "NodeAttribute" { PropertyTemplate
... }`). The parser's `_get_property70(na, "UpVector")` returns None for
the actual instance, so the parser falls through to the hardcoded default
`(0, 1, 0)`. The test's `CAMERA1_UP = (0.0, 30.0, 0.0)` round-trips only
because it normalizes to the same direction as `(0, 1, 0)` — the
`rotation_matrix_from_look_at` helper re-orthogonalizes `up` against
`forward`, so any up parallel to `(0,1,0)` yields the same `right`/`up_cam`
basis.

This means:
- The UpVector extraction path (`static_up_vector` assignment at lines
  632 and 635) is effectively dead code in the test suite.
- A regression like `p[2]` being read as `p[0]` (typo, index mix-up) would
  not be caught.
- The Model-vs-NodeAttribute precedence (see WR-01) is also untested.

**Fix:** Add a synthetic test that constructs an FBX tree with
`UpVector = (1.0, 0.5, 0.3)` on the NodeAttribute and verifies the parsed
`static_up_vector` matches. Parallel tests for Model-only and both-present
(NA wins) would cover WR-01's precedence fix.

```python
def test_up_vector_from_node_attribute(self):
    from forge_flame.fbx_ascii import _extract_cameras, FBXNode
    # Minimal tree: 1 Camera Model, 1 NodeAttribute (UpVector set),
    # 1 Null Model, OO connection NA->cam, OP connection null->cam
    # LookAtProperty. Assert cameras[0].static_up_vector == (1.0, 0.5, 0.3).
    ...
```

## Info

### IN-01: Duplicate Model iteration in `_extract_cameras`

**File:** `forge_flame/fbx_ascii.py:482-494`

**Issue:** `_iter_objects(root, "Model")` is walked twice — once for
cameras (line 483), once for Nulls (line 492). For FBXs with many objects
this pays the top-level walk cost twice. It's not a correctness bug and
the out-of-scope performance guideline applies, but one pass with branch
dispatch on `values[2]` would be cleaner and matches the style of the
surrounding code.

**Fix:**
```python
camera_models: dict[int, FBXNode] = {}
null_models: dict[int, FBXNode] = {}
for m in _iter_objects(root, "Model"):
    if len(m.values) < 3:
        continue
    kind = m.values[2]
    if kind == "Camera":
        camera_models[_object_id(m)] = m
    elif kind == "Null":
        null_models[_object_id(m)] = m
```

### IN-02: `test_roll_sign_convention` does not lock direction

**File:** `tests/test_rotations.py:130-143`

**Issue:** The test asserts that +90° roll, −90° roll, and 0° roll all
produce distinct matrices, but does not verify the actual direction of
rotation (e.g., for +90° roll, does `right` rotate toward `up_cam` or
toward `-up_cam`?). A sign flip in `theta = np.radians(-float(roll_deg))`
would still pass this test as long as the resulting matrices remained
distinct. The roll-sign convention is decisively locked by
`test_camera1_known_answer` (which asserts `abs(rz - (-1.25)) < 1.0`),
so the overall coverage is adequate, but this test's name overstates
what it proves.

**Fix:** Strengthen the assertion to lock direction, or rename to
`test_roll_produces_distinct_matrices` so the coverage is described
accurately:

```python
def test_roll_plus_90_rotates_right_toward_up_cam(self):
    # At roll=+90° (with the sign-flip convention in the helper),
    # the `right` column should rotate toward -up_cam (clockwise
    # through the lens). Locks direction, not just distinctness.
    R0 = rotation_matrix_from_look_at((0,0,0),(0,0,-100),(0,1,0),  0.0)
    Rp = rotation_matrix_from_look_at((0,0,0),(0,0,-100),(0,1,0), 90.0)
    # `right` was (1,0,0); after +90° roll it should point toward -up_cam = (0,-1,0).
    np.testing.assert_allclose(Rp[:, 0], (0.0, -1.0, 0.0), atol=1e-12)
```

### IN-03: Zero-length up vector produces misleading error message

**File:** `forge_core/math/rotations.py:146-153`

**Issue:** When `up = (0, 0, 0)`, `np.cross(forward, up) = (0, 0, 0)` and
`right_norm = 0.0`, triggering the "up vector parallel to view direction"
error. Technically parallel-and-zero share the same symptom (cross product
degenerate) but the error message misdirects a future reader who passes
`(0,0,0)` by accident — they'll go hunting for a direction collinear with
their forward vector instead of noticing the zero magnitude.

**Fix:** Either add a pre-check for `|up| <= 1e-6` with its own error
message, or broaden the existing message to cover both cases:

```python
if right_norm <= 1e-6:
    raise ValueError(
        f"aim-rig resolve: up vector degenerate (zero) or parallel to "
        f"view direction — |cross(forward, up)| = {right_norm!r} fails "
        f">= 1e-6 threshold"
    )
```

### IN-04: Local import of `compute_flame_euler_zyx` unused in non-aim-rig path

**File:** `forge_flame/fbx_ascii.py:807-810`

**Issue:** Both `rotation_matrix_from_look_at` and
`compute_flame_euler_zyx` are imported unconditionally at the top of
`_merge_curves`, but `compute_flame_euler_zyx` is only used inside the
`if is_aim_rig:` branches. The comment on line 804 says "Imported up-front
(not conditionally) to keep the static_fallback and per-frame branches
below straightforward", which is a valid readability choice — but the
numpy import cost is paid unconditionally for any FBX reader call, even
those with no aim-rig cameras. Given `numpy` is already a `forge_core`
module dependency, the cost is cache-hit cheap in practice, so this is
informational only.

**Fix:** No action required, unless an audit of module-scope imports
surfaces numpy-purity as a constraint for the reader. Then move the
import fully inside the `if is_aim_rig:` block at line 834.
