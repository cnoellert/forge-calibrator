# Phase 4.2 — where it landed, where it needs to go

## What works (as of commit cbfd185)

- **End-to-end round-trip is functional.** Flame aim-rig → Blender → back → Flame Free-rig. No crashes, no hook errors. Item 2 of HUMAN-UAT passes with a known limitation.
- **Install path is correct.** tools/blender lives at `/opt/Autodesk/shared/forge_blender_scripts/` (not under Flame's Python path). See commit `64a96cd`.
- **Roll sign is handled.** Parser passes `roll_deg=+cam.static_roll` (FBX-stored sign). See commit `c05516e`.

## Known limitation

- **ry residual ≈ 0.087°** on the Camera1 fixture — rendered orientation is within ROADMAP's 0.1° tolerance but **not tight enough for production CG registration** (user reported visible registration slip). See `04.2-HUMAN-UAT.md` item 2 for the user's exact feedback.

## Root cause identified (but fix not landed)

Flame's free-rig rotation uses convention `R = Rz(-rz)·Ry(-ry)·Rx(-rx)` — **XYZ matrix order with all three signs negated**. Our `compute_flame_euler_zyx` implements `R = Rz(+rz)·Ry(-ry)·Rx(-rx)` — only rx/ry sign-flipped. Plan 01's tests never caught this because they used `flame_euler_to_cam_rot` as their own oracle (circular).

Verified via forge-bridge live probe + user viewport manual-match 2026-04-24:

- **Our reader output (current, reverted state):** `(1.858, 0.977, +1.252)` for Camera1 fixture
- **Flame viewport ground truth (user manual match):** `(1.8193, 1.0639, 1.2529)`
- **Decomposing our look-at matrix with the correct XYZ-signflip convention:** `(1.814, 1.058, +1.252)` — matches Flame truth within 0.005°/0.006°/0.001°

## Why the fix was reverted (commit cbfd185 reverts 3e25b5e)

Landing the convention fix in the reader alone (`compute_flame_euler_xyz` in `forge_flame/fbx_ascii.py` aim-rig branch) caused Flame to **hard-crash** inside native `action.import_fbx()` on the return trip. The crash symptom was:

```
14:25:16.708 Entering importToActionFBX
14:25:16.821 Executing command: tar cbf 20 ifffsLog.tar ...   # crash-dump tarball
14:25:16.861 Executing command: tar cbf 20 swLog.tar ...      # crash-dump tarball
(log ends — process died)
```

**Reason:** the fix was half-applied. The reader emitted Euler in the new XYZ-signflip convention, but `tools/blender/bake_camera.py`'s `_flame_euler_to_rot_matrix` (the Blender-subprocess parallel implementation) still used the OLD ZYX-signflip. That broke internal pipeline symmetry — the Blender scene carried a matrix that wasn't the aim-rig orientation, Blender's FBX exporter decomposed it with Blender's native XYZ, and Flame's importer hit an edge case it couldn't handle.

## The correct fix (for the next session)

Update the convention in **every** place that touches the Euler math, in sync:

1. `forge_core/math/rotations.py` — add `compute_flame_euler_xyz` + `flame_euler_xyz_to_cam_rot` that implement `R = Rz(-rz)·Ry(-ry)·Rx(-rx)`. Keep the existing `_zyx` functions untouched (Free-rig solve/adapter.py relies on their symmetric-pass-through behaviour; don't risk breaking that).
2. `forge_flame/fbx_ascii.py` — aim-rig branch (static + per-frame) uses the new `_xyz` functions.
3. `tools/blender/bake_camera.py` — `_flame_euler_to_rot_matrix` must match the reader's convention. Parallel-implement `flame_euler_xyz_to_cam_rot` using mathutils.
4. `tools/blender/forge_sender/flame_math.py` — same story; addon-side math must match.
5. `tools/blender/extract_camera.py` — its decomposition (via `_rot3_to_flame_euler_deg` in `flame_math.py`) must also use the new convention.

**Do not touch `adapter.py`** or anything else in the Free-rig solve path — those use the old convention symmetrically and work in production. Changing them without careful Free-rig regression testing is out of scope.

### Verification criteria

- Aim-rig fixture produces Euler within 0.01° of Flame viewport ground truth `(1.8193, 1.0639, 1.2529)` on all three axes.
- Free-rig round-trip unchanged (rotation values preserved through bake→extract).

- Existing 380-test suite stays green.

## Open question worth exploring before the precision fix

Is the 0.087° residual *entirely* from the Euler convention mismatch, or does `rotation_matrix_from_look_at` also have a bug? Matrix diff showed our look-at's `right` has Y=0 pre-roll (as expected from `cross(forward, (0,1,0))`), and Flame's ground-truth `right` has Y=+0.022. The Rodrigues roll application closes that gap when the roll sign is right — but the residual in the +X component suggests either Up handling or a subtle convention difference in the basis construction itself. Worth a second look before committing to the full pipeline update.
