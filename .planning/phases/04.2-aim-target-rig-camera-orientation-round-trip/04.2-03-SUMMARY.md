## Decisions Made

### Roll sign convention (critical discovery, anticipated)

The FBX fixture stores `P: "Roll", "Roll", "", "A+",1.25252056121826` (POSITIVE), but:
1. The live forge-bridge probe reports `roll = -1.252521` (NEGATIVE) — see CONTEXT.md §specifics.
2. Plan 01's Camera1 known-answer test input uses `CAMERA1_ROLL_DEG = -1.252521` (NEGATIVE).
3. Plan 01's reference output is `rz = -1.2520907…` (NEGATIVE).

If the FBX parser fed the raw stored Roll (+1.25) to `rotation_matrix_from_look_at`, it would produce `rz ≈ +1.25°` — **opposite sign** from Plan 01's locked answer.

**Resolution (Task 3, feat commit 18b9f1b):** Parser negates Roll before feeding the look-at helper, exactly mirroring the sign-flip Flame applies to Lcl Rotation at FBX emission time:

```python
# Aim-rig branch (static-fallback and per-frame):
R = rotation_matrix_from_look_at(
    position=sp,
    aim=cam.static_aim_position,
    up=cam.static_up_vector,
    roll_deg=-cam.static_roll,  # ← NEGATE
)
```

This is the same shape as the existing free-rig negation `rx_deg = -_sample_at(rx, ktime, cam.static_rotation[0])` — Flame's ZYX-with-X,Y-negated convention applies to Roll too.

Verified by the Camera1 integration test passing at abs=1e-3° tolerance. This locks the roll-sign convention at the parser boundary, not in `forge_core.math.rotations.rotation_matrix_from_look_at` itself (Plan 01's helper takes roll as-given, a pure-math primitive).

### AnimCurveNode target duck-typing (Model vs NodeAttribute)

The Camera1 probe fixture attaches `AnimCurveNode::Roll` to `NodeAttribute::Camera1` (id 5155769856), NOT directly to `Model::Camera1` (id 5156816384) — see fixture lines 642-643:

```
;AnimCurveNode::Roll, NodeAttribute::Camera1
C: "OP",105553322838464,5155769856, "Roll"
```

The plan spec originally said `target_id == model_id and channel == "Roll"`, but that would miss this shape. The implementation checks BOTH the camera Model and its NodeAttribute as valid targets for the "UpVector" and "Roll" channels (`target_id in (model_id, na_id)`). This makes the parser tolerant to either attachment shape without requiring a follow-up patch.

### Local numpy import placement

The plan called for `from forge_core.math.rotations import ...` inside `_merge_curves` body (rather than at module scope) so the module stays numpy-free for callers that don't exercise the reader path (e.g. `v5_json_to_fbx` on the writer side). This is now in place; `grep -cE "^import numpy|^from numpy" forge_flame/fbx_ascii.py` returns 0, matching the plan's acceptance criterion.
