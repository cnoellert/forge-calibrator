## Decisions Made

### Option A (delete) vs option B (hasattr guard)

Plan called the choice out explicitly under CONTEXT.md's "Claude's Discretion" note (D-14 footer). Picked **A (delete)** and the outcome confirmed the rationale:

- **Redundancy:** All three sites were structurally redundant. The surrounding `set_value` calls for position/rotation/fov unambiguously define the Free-rig pose — per D-10 analysis and the user's principle "rig type doesn't matter, transform does", there is no state the deleted `target_mode` line was contributing that isn't already set explicitly below it.
- **Guard-path value is low:** Flame 2026.2.1 is pinned per PROJECT.md Compatibility constraint; there is no evidence future Flame versions will re-expose a usable `target_mode`, and if they do, the right fix is a centralised `forge_flame` primitive (not ad-hoc guards at two call sites). The deferred-ideas section of CONTEXT.md explicitly calls this out as v2 scope.
- **CLAUDE.md alignment:** "Raise on truly exceptional conditions only" + "Guard clauses return early... rather than raising" — a guard that silently skips is strictly worse than removing dead code. A hasattr-guard at these sites would always evaluate to False on the pinned Flame version, producing cargo-cult code with no runtime purpose.
- **Trace readback specifically:** Line 1578 is pure diagnostic metadata written to `/tmp/forge_camera_match_trace.json`. Dropping the key removes a field that was never readable anyway — this shrinks the Flame-state surface leaked to the diagnostic artifact rather than expanding it. T-04.2-04 threat register entry explicitly notes this tradeoff falls on the correct side.

### No replacement comment left behind

Plan explicitly instructed: do not leave a `# target_mode omitted — attribute missing on PyCoNode` comment in either file. Followed this — the commit message and this SUMMARY carry the "why" context; comments would clutter tight code blocks for future readers who don't need the history.

### rotation_diagnostic.py:99-103 intentionally untouched

```python
if hasattr(cam, "target_mode"):
    try:
        cam.target_mode.set_value(False)
    except Exception as e:
        print(f"target_mode set failed: {e}")
```

This is the correct defensive shape for a diagnostic tool that runs under forge-bridge against arbitrary Flame versions. Leaving it in place:

- Matches the CLAUDE.md "duck-type Flame API access" architecture note — hasattr is the established idiom.
- Does not share production code paths; doesn't cause user-facing errors; no fix needed.
- If a future Flame version exposes `target_mode` again, this diagnostic line will begin to function without any code change — that's the point of the guard.
