<objective>
Fix the three broken `cam.target_mode.*` call sites that assume Flame 2026.2.1 PyCoNode exposes a `target_mode` attribute — live probe confirmed it does NOT (39-element `cam.attributes` list returned no `target_mode` entry, and `dir(cam)` showed no rig-toggle method). These lines resolve `cam.target_mode` to `None`, then call `.set_value(False)` / `.get_value()` on None, which is an AttributeError-in-waiting. The reason they haven't crashed yet is the code paths haven't been exercised on an aim-rig camera during solve-apply — Phase 4.2's UAT of the aim-rig round-trip is exactly that exercise, so fix this here per D-10/D-11.

Purpose: Per D-10/D-11, the fix lands in the same phase as the aim-rig work because (a) it's the same code area, (b) same root cause (Flame API assumption mismatch — the "rig type doesn't matter, transform does" principle is what makes removing these calls safe; position+rotation+fov set_value already defines the Free-rig pose unambiguously), and (c) it would surface during live UAT of this phase's aim-rig round-trip.

Output: Two files modified (three call sites fixed). Choice of A (delete) or B (hasattr guard) per CONTEXT.md "Claude's Discretion" note — this plan picks **option A (delete)** because: (1) all three sites are redundant — the surrounding explicit position/rotation/fov set_value calls override whatever rig state the camera had; (2) the guard-path value is low since Flame 2026.2.1 is pinned and we have no evidence future Flame versions will re-expose a usable target_mode; (3) deletion is more aligned with CLAUDE.md's "Raise on truly exceptional conditions only" + "Guard clauses return early... rather than raising" posture (a guard that silently skips is strictly worse than removing dead code); (4) the trace readback at line 1578 is pure diagnostic metadata — dropping it removes a field that was never readable anyway.
</objective>
