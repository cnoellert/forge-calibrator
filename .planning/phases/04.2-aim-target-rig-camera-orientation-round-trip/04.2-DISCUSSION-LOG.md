## Pre-discussion: Premise reframing

The ROADMAP 4.2 problem statement asserted "Flame writes rotation=(0,0,0) for every frame on aim-rig cameras, discarding the aim+up+roll". User raised a gating question before committing to an approach: "The issue is that the camera that came out of Flame for export in this instance was a target cam. I just am not sure that we exported it as such." — i.e. is the orientation actually lost, or is it in the FBX in a form our pipeline ignores?

This triggered a static analysis of the FBX parser + a live forge-bridge probe of Camera1 in action9.

### Static analysis findings

- `flame/camera_match_hook.py:2244-2249` calls `fbx_io.export_action_cameras_to_fbx` with no aim-rig detection — no rig-toggle, no pre-export state mutation.
- `forge_flame/templates/camera_baked.fbx` contains `LookAtProperty`, `UpVectorProperty`, and `ViewCameraToLookAt` property definitions — Flame's FBX schema has native aim-rig support.
- `tests/fixtures/forge_fbx_probe.fbx` line 644–645 shows the shape: `;Model::Perspective.Target, Model::Perspective` + `C: "OP", 5496823296, 5496816128, "LookAtProperty"` — a separate Null node + a typed connection.
- `forge_flame/fbx_ascii.py:_extract_cameras` (line 449) only pulls `Model::...,Camera` nodes + Lcl Translation / Lcl Rotation / FieldOfView AnimCurves. It does not follow LookAtProperty connections, does not read Null nodes, does not read UpVector or Roll properties.

This confirmed: aim-rig info IS in the FBX Flame emits; our parser discards it.

## 1. Conversion strategy

| Option | Description | Selected |
|--------|-------------|----------|
| A. Toggle `cam.target_mode=False` in Flame before export | Let Flame bake aim→Euler at rig-switch time, then export. Match the manual workaround. | |
| A′. Compute Euler from (position, aim, up, roll) ourselves, write to `cam.rotation` before export | Flame-side pre-bake via direct rotation assignment. | |
| B. Fix the FBX reader — extract aim-rig semantics from the FBX Flame already emits, resolve to Euler in v5 JSON | Parser change, no Flame-state mutation, no hook change. | ✓ |
| C. Preserve aim as Blender Track-To constraint | Full semantic round-trip with target Empty in Blender. | |

**User's choice:** B — "I think B is the way"

**Notes:**
- Approach A was ruled out by the live probe: `target_mode` is not in `cam.attributes` on Flame 2026.2.1 (39-entry list checked), and `PyCoNode` has no rig-toggle method in its public dir. Any existing `cam.target_mode.set_value(False)` in the codebase is a latent bug (flagged as D-10 in CONTEXT.md).
- A′ remained theoretically viable but would require probing whether Flame's `export_fbx` honors explicit rotation values when the aim Null is still bound via LookAtProperty. Deferred in favour of B which avoids the question entirely.
- C was acknowledged as most complete semantically but rejected as over-complex for a problem that's really about reading what's already in the FBX.
- B matches user principle "rig type doesn't matter, transform does" (ROADMAP 4.2) — we resolve to Euler, return as Free-rig.

## 2. Animation scope

| Option | Description | Selected |
|--------|-------------|----------|
| Static aim only | Read first keyframe of aim/up/roll, apply one Euler to every frame | |
| Animated aim target, static up/roll | Per-frame aim, fixed up/roll | |
| Fully animated | Per-frame aim + per-frame up + per-frame roll | ✓ |

**User's choice:** Fully animated (accepted recommended default: "Nope. Let's roll")

**Notes:**
- All three of aim, up, roll are marked animatable (`"A"` or `"A+"` in the probe FBX). Supporting the full set costs modest extra parser work and removes the need for any future aim-rig animation phase.
- The Null's `Lcl Translation` uses identical AnimCurveNode structure as the camera's existing Lcl Translation curves — same reader code can be applied to both.

| Option | Description | Selected |

**User's choice:** Fail loud (accepted recommended default)

**Notes:**
- Core value: "never silently ship wrong orientation" — fidelity discipline established in phase 04.1 (d68b77c, 094c3b3).
- Thresholds keyed to existing gimbal-lock tolerance (`1e-6`) for consistency with the rest of the rotation pipeline.
