<objective>
Teach `forge_flame/fbx_ascii.py` to resolve Aim/Target-rig camera orientation out of the FBX Flame emits, so the v5 JSON carries the correct per-frame `rotation_flame_euler` instead of (0, 0, 0). This is the core of Phase 4.2 — without this, the ROADMAP acceptance criterion ("round-trip orientation within 0.1° on all three Euler axes") is impossible because the v5 JSON the Blender bake consumes is carrying zero rotation.

Purpose: Per D-01/D-02 the fix lives at the parser layer (not Flame-side mutation, not Blender-side preservation). Per D-04/D-05 the math pipeline is (position, aim, up, roll) → look-at 3×3 → compute_flame_euler_zyx → (rx, ry, rz). Per D-08/D-09 both static and animated aim-rig cameras must work. Per D-14/D-15 degenerate inputs fail loud with `ValueError`. Plan 01 has already landed the math primitive (rotation_matrix_from_look_at) — this plan wires it into the FBX read path and proves it works on the Camera1 live probe. The D-07 verification gate (case 1 — Camera1 known-answer) lives INSIDE the integration test in this plan; case 2 (free-rig cross-check) was already covered in Plan 01's unit tests.

Output: (1) `forge_flame/fbx_ascii.py` extended with aim-Null detection, LookAtProperty/UpVector/Roll connection walks, per-frame look-at composition in `_merge_curves`, and fail-loud on degenerate input or missing Null; (2) `tests/fixtures/forge_fbx_aimrig.fbx` — the Camera1 live probe copied from the planning directory into tests/fixtures under a stable name; (3) `tests/test_fbx_ascii.py` extended with `TestAimRigFixture` class covering the Camera1 round-trip and D-14 fail-loud cases.
