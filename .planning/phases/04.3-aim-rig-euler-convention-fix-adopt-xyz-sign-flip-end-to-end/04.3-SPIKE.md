## Why the Plan's Symbolic Derivation Fails

I verified the symbolic derivation independently against numpy and `scipy.spatial.transform.Rotation`. **The plan's derivation IS mathematically self-consistent** — composing `R = Rz(-rz)·Ry(-ry)·Rx(-rx)` from `(rx, ry, rz)` and then decomposing per the plan's atan2 indices does round-trip to within `1e-12°`. The closed form is correct.

**What's wrong is the assumption that decomposing the look-at output under that convention reproduces `(1.814, 1.058, 1.252)`.** Empirically, on the Camera1 inputs:

| Convention                               | Decomposed (rx, ry, rz)        | max delta from hand-target | max delta from Flame truth |
|------------------------------------------|--------------------------------|----------------------------|----------------------------|
| Current `_zyx` (`Rz(rz)·Ry(-ry)·Rx(-rx)`) | (+1.858515, +0.977378, +1.252) | 0.0806°                    | 0.0865°                    |
| Plan's `_xyz` (`Rz(-rz)·Ry(-ry)·Rx(-rx)`) | (+1.858515, +0.977378, **-1.252**) | **2.5041°**                | **2.5050°**                |
| `R = Rx(-rx)·Ry(-ry)·Rz(rz)` (XY-neg)    | (+1.837005, +1.017240, +1.220) | 0.0408°                    | 0.0467°                    |
| Brute-force scipy `xyz` (extrinsic)      | (-1.858515, -0.977378, +1.252) | n/a (signs don't match)    | n/a                        |
| Brute-force scipy `XYZ` (intrinsic)      | (-1.837005, -1.017240, +1.220) | n/a                        | n/a                        |

**No single decomposition convention applied to the current `rotation_matrix_from_look_at` output reproduces `(1.814°, 1.058°, 1.252°)` to better than 0.04°.**

## What Actually Reproduces `(1.814, 1.058, 1.252)`

By searching combinatorial sign/order space I found exactly **one** combination that matches the hand-decomposed value to 3 decimal places:

1. Build the look-at matrix with **positive** roll sign in the Rodrigues step:
   ```python
   theta = np.radians(+roll_deg)        # NOT -roll_deg as currently
   ```
2. Decompose with scipy's extrinsic `xyz` extraction (equivalently: `R = Rx(rx)·Ry(ry)·Rz(rz)`).
3. Negate all three results.

Equivalent to: **the camera matrix that matches Flame is `R_flame = Rx(-rx)·Ry(-ry)·Rz(-rz)`** but you have to build it from look-at with `+roll_deg` (not `-roll_deg`) in the Rodrigues step.

Numerically:

```
build_R(roll_sign=+1) → scipy 'xyz' → (-1.814039, -1.057661, -1.252091)
                       → negate all → (+1.814039, +1.057661, +1.252091)
```

vs Flame viewport truth `(1.8193, 1.0639, 1.2529)`:

| Axis | Recovered    | Truth     | Delta     |
|------|--------------|-----------|-----------|
| rx   | +1.814039    | +1.8193   | -0.005261 |
| ry   | +1.057661    | +1.0639   | -0.006239 |
| rz   | +1.252091    | +1.2529   | -0.000809 |

Max |delta| = **0.006°** — **well within the 0.01° Flame ground-truth gate.**

## Two Coupled Changes Required

To land this fix correctly, both of the following must change in lockstep:

### Change 1 — `rotation_matrix_from_look_at` roll sign

In `forge_core/math/rotations.py` L167:

```python
# Current (Phase 04.2 c05516e):
theta = np.radians(-float(roll_deg))

# Required for the new convention:
theta = np.radians(+float(roll_deg))
```

This is the **basis-construction adjustment** the CONTEXT.md §deferred §"Open question" anticipated. The 04.2 handoff note flagged: *"Worth a second look before committing to the full pipeline update."*

This is a non-trivial change because:
- The current `+roll_deg → rz≈+1.252` invariant is what the existing `tests/test_rotations.py::TestLookAtMatrix.test_camera1_known_answer` validates (`assert abs(rz - CAMERA1_ROLL_DEG) < 1.0`). With the new combination, the invariant becomes `+roll_deg → rz≈+1.252` again (sign preserved post-negation), so that test would still pass — but the underlying R is different.
- Free-rig anti-regression: `tests/test_rotations.py::TestLookAtMatrix.test_free_rig_cross_check` builds R via `flame_euler_to_cam_rot(*rot)` (free-rig convention) then feeds it through `rotation_matrix_from_look_at(position, aim, up, 0.0)` (zero roll). Roll sign flip has zero effect at `roll_deg=0`, so this test stays green.
- Other look-at callers: `forge_flame/fbx_ascii.py` aim-rig branch (the only production caller) — would need to flip its convention in lockstep.

### Change 2 — Decomposer convention

Per the empirical fit: `R = Rx(-rx)·Ry(-ry)·Rz(-rz)` (XYZ matrix order, all three signs negated, BUT the matrix product order is X-Y-Z left-to-right, NOT Z-Y-X as the plan derives).

The plan's name "XYZ-signflip" is correct in spirit (all three signs negated) but the **matrix-product order is reversed** vs what the plan derives:
- Plan: `R = Rz(-rz)·Ry(-ry)·Rx(-rx)` (Z·Y·X reading left to right)
- Empirical: `R = Rx(-rx)·Ry(-ry)·Rz(-rz)` (X·Y·Z reading left to right)

These are **different rotations** for non-trivial triples (matrix multiplication is non-commutative). A composition under one is not a composition under the other.

Re-derived decomposition for `R = Rx(-rx)·Ry(-ry)·Rz(-rz)`:

Substitute α=-rx, β=-ry, γ=-rz → `R = Rx(α)·Ry(β)·Rz(γ)` (standard XYZ-positive intrinsic; equivalent to extrinsic ZYX).

Closed form for `R = Rx·Ry·Rz` is:
```
R = | cβ·cγ                  -cβ·sγ                  sβ                 |
    | cα·sγ + sα·sβ·cγ        cα·cγ - sα·sβ·sγ        -sα·cβ            |
    | sα·sγ - cα·sβ·cγ        sα·cγ + cα·sβ·sγ         cα·cβ            |
```

Extract:
- `R[0,2] = sβ`             → β = arcsin(R[0,2])           → **rx = -atan2(-R[1,2], R[2,2])  ;  ry = -arcsin(R[0,2])  ;  rz = -atan2(-R[0,1], R[0,0])**

Or equivalently the simpler form:
```python
ry = -np.arcsin(R[0, 2])
rx =  np.arctan2(R[1, 2], R[2, 2])           # = -atan2(-R[1,2], R[2,2])
rz =  np.arctan2(R[0, 1], R[0, 0])           # = -atan2(-R[0,1], R[0,0])
# Gimbal at ry = ±90° (cos(β) ≈ 0):
# rx = 0; rz = atan2(R[1,0], R[1,1]) etc.
```

Plus the corresponding composer `flame_euler_xyz_to_cam_rot` builds `Rx(-rx)·Ry(-ry)·Rz(-rz)` (NOT `Rz(-rz)·Ry(-ry)·Rx(-rx)` as the plan specifies).

## Impact on Plan as Written

The Phase 04.3 plan's specifications need amendments:

1. **`<symbolic_derivation_for_xyz_decomposer>` is wrong about the matrix-product order.** The closed form for the decomposer it gives doesn't match the empirically-correct convention. Tasks 2-5 reference this derivation literally, including explicit grep-asserts in their acceptance criteria (e.g., Task 3 acceptance: `grep -c "Matrix.Rotation(-rz" tools/blender/bake_camera.py` returns 1). With the corrected convention, the grep targets are different (it's the X factor that gets the leading rz-style flip, the Z factor is just a matter of order).

2. **`rotation_matrix_from_look_at` requires a sign change** (Change 1 above). The plan's <critical_constraints> explicitly forbid touching this function — *"Do NOT touch ... the existing _zyx pair (compute_flame_euler_zyx / flame_euler_to_cam_rot) in forge_core/math/rotations.py."* The roll sign in look-at is technically separate from the `_zyx` pair, but it IS in the same file.

3. **The hand-decomposed Camera1 oracle `(1.814°, 1.058°, 1.252°)` is reachable only with both Change 1 and Change 2 applied together.** Tests that use it as an oracle (Task 2's `TestComputeFlameEulerXyz` Camera1 known-answer) must be predicated on the look-at change landing too.

4. **The aim-rig branches in `forge_flame/fbx_ascii.py`** still call `rotation_matrix_from_look_at(roll_deg=cam.static_roll)` (FBX-stored sign, +). With Change 1 landed, that call would now interpret the FBX roll under the new sign convention — semantically equivalent to today's behavior because the convention is coupled to the new decomposer, not separately observable.

5. **`tools/blender/bake_camera.py::_flame_euler_to_rot_matrix` and `tools/blender/forge_sender/flame_math.py::_rot3_to_flame_euler_deg`** must implement the **X·Y·Z** product order (per Change 2), not the Z·Y·X order the plan specifies.
