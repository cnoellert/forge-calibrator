# Phase 04.3 — Plan Amendment (post-spike correction)

## Why This Amendment Exists

Task 1's gating spike returned **Branch B** with rz delta = 2.504° (max |delta| from CONTEXT.md target). The plan's `<symbolic_derivation_for_xyz_decomposer>` block specified the convention as `R = Rz(-rz)·Ry(-ry)·Rx(-rx)` — Z·Y·X matrix product order with all three Euler signs negated. That product is mathematically self-consistent (composer↔decomposer round-trips to 1e-12°), but **it does not match Flame's actual aim-rig camera math** when applied to `rotation_matrix_from_look_at` output.

The empirical search documented in `04.3-SPIKE.md` (and reproduced by `spike_xyz_final.py`) found exactly one combination that recovers the CONTEXT.md target `(1.814°, 1.058°, 1.252°)` to ≤ 0.006° from Flame viewport ground truth `(1.8193°, 1.0639°, 1.2529°)`:

| Component | Original plan | Corrected (this amendment) |
|---|---|---|
| Decomposer convention | `R = Rz(-rz) · Ry(-ry) · Rx(-rx)` (Z·Y·X order) | **`R = Rx(-rx) · Ry(-ry) · Rz(-rz)`** (X·Y·Z order) |
| `rotation_matrix_from_look_at` roll Rodrigues | `theta = np.radians(-roll_deg)` (untouched) | **`theta = np.radians(+roll_deg)`** (sign flipped) |
| `<critical_constraints>` (executor prompt) | Forbids touching `rotation_matrix_from_look_at` | **Carve-out: roll-sign flip at L167 ONLY is permitted** (the `_zyx` pair stays byte-identical) |

Both changes are coupled — flipping one without the other does **not** reproduce the target. Therefore both must land **inside the same atomic Task 3 commit** (the existing pipeline-symmetric guard now covers six files instead of five).

The convention's NAME stays `_xyz` — it's still "all three signs negated, XYZ axis ordering." Only the matrix-product chaining order corrects from Z·Y·X to X·Y·Z.

## §A — Replacement for `<symbolic_derivation_for_xyz_decomposer>`

The original derivation block in `04.3-01-PLAN.md` (lines 239-293) is **superseded by this section**. Use the math here, not the math in the plan body.

### Corrected convention

Flame's aim-rig camera composes its world rotation as:

```
R = Rx(-rx) · Ry(-ry) · Rz(-rz)         # X·Y·Z product (left-to-right reading order)
```

with all three Euler signs negated. Substitute α = -rx, β = -ry, γ = -rz; this reduces to a standard XYZ-positive intrinsic product `R = Rx(α) · Ry(β) · Rz(γ)`. Closed form:

```
R = | cβ·cγ                    -cβ·sγ                    sβ      |
    | cα·sγ + sα·sβ·cγ          cα·cγ - sα·sβ·sγ          -sα·cβ |
    | sα·sγ - cα·sβ·cγ          cα·sβ·sγ + sα·cγ           cα·cβ |
```

### Decomposition (non-gimbal, |R[0,2]| < 1)

- `R[0,2] = sβ`                     → `β = arcsin(R[0,2])`
- `tan(α) = -R[1,2] / R[2,2]`       → `α = atan2(-R[1,2], R[2,2])`
- `tan(γ) = -R[0,1] / R[0,0]`       → `γ = atan2(-R[0,1], R[0,0])`

In `(rx, ry, rz)` (note `atan2(-y, x) = -atan2(y, x)` identically):

- `ry = -β = -arcsin(R[0,2])`
- `rx = -α = -atan2(-R[1,2], R[2,2])  = atan2(R[1,2], R[2,2])`
- `rz = -γ = -atan2(-R[0,1], R[0,0])  = atan2(R[0,1], R[0,0])`

### Decomposition (gimbal, cβ ≈ 0 → |R[0,2]| ≈ 1)

Pin `α = 0` (rx = 0). With `α = 0`, both R[1,0] and R[1,1] reduce to functions of γ alone:

- `R[1,0] = cα·sγ + sα·sβ·cγ → sγ` (when α = 0; sα = 0, cα = 1)
- `R[1,1] = cα·cγ - sα·sβ·sγ → cγ`
- `γ = atan2(R[1,0], R[1,1])`
- `rz = -γ = -atan2(R[1,0], R[1,1])`

(Same formula for both β = +π/2 and β = −π/2, because the sα·sβ terms vanish when α = 0.)

### Gimbal indicator

`cβ` cancels out of R[0,0] and R[0,1] only as a magnitude: `R[0,0]² + R[0,1]² = cβ²·(cγ² + sγ²) = cβ²`. So:

```
cb = sqrt(R[0,0]² + R[0,1]²)        # |cos(β)|; gimbal when cb ≤ 1e-6
```

(Note this is **different** from `_zyx`'s `sqrt(R[0,0]² + R[1,0]²)` — the row/col indices change with the product order. Do not copy-paste from `_zyx`.)

### Final Python form (numpy)

```python
def compute_flame_euler_xyz(cam_rot: np.ndarray) -> Tuple[float, float, float]:
    """Decompose a 3x3 cam-to-world rotation into Flame's XYZ-signflip Euler.

    Flame's aim-rig camera composes R as ``Rx(-rx) · Ry(-ry) · Rz(-rz)``:
    X·Y·Z matrix-product order with all three Euler signs negated.
    Substituting α=-rx, β=-ry, γ=-rz reduces to the standard
    XYZ-positive intrinsic product R = Rx(α)·Ry(β)·Rz(γ); extraction
    follows the standard closed-form decomposition.

    Verified 2026-04-25 by:
      - forge-bridge live probe of Camera1 (position/aim/up/roll/fov)
      - Flame viewport manual-match to (1.8193°, 1.0639°, 1.2529°)
      - empirical sign/order search in 04.3-SPIKE.md / spike_xyz_final.py
    Hand-decomposing rotation_matrix_from_look_at(Camera1) under this
    convention reproduces (1.814°, 1.058°, 1.252°) — within 0.006° of
    Flame viewport truth on every axis. Closes the Phase 04.2 ~0.087°
    ry residual on the aim-rig fixture.

    Does NOT modify or replace ``compute_flame_euler_zyx``; the Free-rig
    solve path depends on that pair's symmetric pass-through behaviour
    and is out of scope for this phase.

    Returns (rx_deg, ry_deg, rz_deg). Handles gimbal lock (ry = ±90°)
    by pinning rx=0 and recovering rz from the upper-left 2x2 block."""
    cb = np.sqrt(cam_rot[0, 0] ** 2 + cam_rot[0, 1] ** 2)
    gimbal = cb <= 1e-6
    if not gimbal:
        rx =  np.arctan2(cam_rot[1, 2], cam_rot[2, 2])
        ry = -np.arcsin(cam_rot[0, 2])
        rz =  np.arctan2(cam_rot[0, 1], cam_rot[0, 0])
    else:
        rx = 0.0
        ry = -np.arcsin(cam_rot[0, 2])
        rz = -np.arctan2(cam_rot[1, 0], cam_rot[1, 1])
    deg = np.degrees(np.array([rx, ry, rz]))
    return (float(deg[0]), float(deg[1]), float(deg[2]))


def flame_euler_xyz_to_cam_rot(rx_deg: float, ry_deg: float, rz_deg: float) -> np.ndarray:
    """Compose a 3x3 cam-to-world rotation from Flame's XYZ-signflip Euler.

    Exact inverse of ``compute_flame_euler_xyz``: builds R using
    Flame's aim-rig composition ``R = Rx(-rx) · Ry(-ry) · Rz(-rz)``.
    Given any (rx, ry, rz) that came out of the decompose, this
    returns the same rotation matrix that went in.

    Used by the aim-rig Flame → Blender bake path (rebuild world
    matrix from stored JSON) and by tests that need to generate
    known-answer rotations under the XYZ-signflip convention.

    Does NOT modify or replace ``flame_euler_to_cam_rot``; the
    Free-rig pipeline keeps the ``_zyx`` pair."""
    rx, ry, rz = np.radians([rx_deg, ry_deg, rz_deg])
    cx, sx = np.cos(-rx), np.sin(-rx)
    cy, sy = np.cos(-ry), np.sin(-ry)
    cz, sz = np.cos(-rz), np.sin(-rz)
    Rx = np.array([[1, 0, 0], [0, cx, -sx], [0, sx, cx]])
    Ry = np.array([[cy, 0, sy], [0, 1, 0], [-sy, 0, cy]])
    Rz = np.array([[cz, -sz, 0], [sz, cz, 0], [0, 0, 1]])
    return Rx @ Ry @ Rz                     # X·Y·Z product order — NOT Z·Y·X
```

The Blender-side mathutils version of the decomposer (`_rot3_to_flame_euler_deg` in `flame_math.py`) MUST mirror this verbatim (swap `np.arctan2`→`math.atan2`, `np.arcsin`→`math.asin`, `R[i, j]`→`R[i][j]`, `np.degrees`→`math.degrees`).

The Blender-side mathutils composer (`_flame_euler_to_rot_matrix` in `bake_camera.py`) flips both the angle signs AND the multiplication order:

```python
def _flame_euler_to_rot_matrix(rx_deg, ry_deg, rz_deg) -> Matrix:
    rx, ry, rz = (math.radians(a) for a in (rx_deg, ry_deg, rz_deg))
    return (Matrix.Rotation(-rx, 4, 'X')
            @ Matrix.Rotation(-ry, 4, 'Y')
            @ Matrix.Rotation(-rz, 4, 'Z'))
```

The X factor is now the **leftmost** in the product, and all three angles have leading minuses.

```python
def test_gimbal_lock_ry_plus_90(self):
    """Hand-built gimbal test under the X·Y·Z product convention.

    Target: rx=0, ry=+90°, rz=+30°. Under R = Rx(-rx)·Ry(-ry)·Rz(-rz)
    this is R = Rx(0) @ Ry(-90°) @ Rz(-30°). Matrix entries computed
    BY HAND from the closed form (cz=cos(-30°)=√3/2, sz=sin(-30°)=-0.5;
    cy=cos(-90°)=0, sy=sin(-90°)=-1):

        Ry(-90°) = [[ 0, 0,-1], [ 0, 1, 0], [ 1, 0, 0]]
        Rz(-30°) = [[ √3/2, +0.5, 0], [-0.5, √3/2, 0], [ 0, 0, 1]]
        R = Ry(-90°) @ Rz(-30°)
          = [[ 0,    0,    -1   ],
             [-0.5,  √3/2,  0   ],
             [ √3/2, 0.5,   0   ]]

    Anti-regression: this test guards against a copy-paste from
    _zyx's gimbal branch. _zyx uses sqrt(R[0,0]^2 + R[1,0]^2) as the
    gimbal indicator (check R[2,0]=±1) AND atan2(-R[0,1], R[1,1]) for
    rz; both would give wrong answers on this matrix because R[0,0]
    and R[1,0] aren't simultaneously zero here, and the row/col
    indices are different under X·Y·Z. The correct _xyz gimbal
    indicator is sqrt(R[0,0]^2 + R[0,1]^2)=0 (R[0,2]=±1)."""
    sqrt3_over_2 = math.sqrt(3) / 2
    R = np.array([
        [ 0.0,        0.0,         -1.0       ],
        [-0.5,        sqrt3_over_2, 0.0       ],
        [ sqrt3_over_2, 0.5,        0.0       ],
    ], dtype=float)
    rx, ry, rz = compute_flame_euler_xyz(R)
    assert math.isclose(rx,  0.0, abs_tol=1e-3)
    assert math.isclose(ry, 90.0, abs_tol=1e-3)
    assert math.isclose(rz, 30.0, abs_tol=1e-3)
```

The pure-axis tests (rx, ry, rz) in the original plan stay valid under the X·Y·Z convention — pure-axis matrices like `Rx(-30°)` decompose identically under both Z·Y·X and X·Y·Z products (only one factor is non-trivial). Don't change those.
