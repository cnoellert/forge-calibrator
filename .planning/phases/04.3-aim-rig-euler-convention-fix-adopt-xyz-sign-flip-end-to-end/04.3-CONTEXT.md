# Phase 04.3: Aim-rig Euler convention fix — Context

**Gathered:** 2026-04-24
**Status:** Ready for planning
**Source:** Synthesised from Phase 04.2 handoff notes (`.continue-here.md` @ commit `080ef36`)

<domain>
## Phase Boundary

Close the Phase 04.2 known limitation: the aim-rig camera round-trip lands ~0.087° off on the `ry` axis for the Camera1 fixture, because **our Euler decomposition uses the wrong rotation convention**. Phase 04.2 identified the correct convention empirically (forge-bridge probe + user viewport manual-match 2026-04-24) but tried to land the fix in the reader alone (commit `3e25b5e`), which hard-crashed Flame's native `action.import_fbx()` on the return trip. That fix was reverted (commit `cbfd185`) and re-scoped to this phase: **adopt the correct convention in every place the Euler math touches, in sync.**

**In scope:**
- The aim-rig FBX reader (`forge_flame/fbx_ascii.py` aim-rig branch, static + per-frame)
- Blender subprocess bake/extract scripts (`tools/blender/bake_camera.py`, `tools/blender/extract_camera.py`, `tools/blender/forge_sender/flame_math.py`)
- New `_xyz` rotation primitives in `forge_core/math/rotations.py` (additive — do not remove or modify `_zyx`)

**Out of scope (DO NOT TOUCH):**
- `forge_flame/adapter.py` — Free-rig solve path; uses old `_zyx` convention symmetrically in production. Changing it requires Free-rig regression testing that is out of scope for this phase.
- Existing `compute_flame_euler_zyx` / `flame_euler_to_cam_rot` in `forge_core/math/rotations.py` — Free-rig and tests depend on them.
- Solver math (`forge_core/solver/*`), image / OCIO / Wiretap paths, UI, installer.

**Deferred / open question** (see `<deferred>` below): whether the residual is entirely convention-driven or whether `rotation_matrix_from_look_at` also has a basis-construction bug. Worth a quick investigation before committing to the full pipeline swap.

</domain>

<decisions>
## Implementation Decisions

### Rotation convention (the math)
- **LOCKED:** Flame's free-rig rotation composes as `R = Rz(-rz) · Ry(-ry) · Rx(-rx)` — XYZ matrix order with **all three** Euler signs negated. This was verified empirically 2026-04-24 via forge-bridge probe of Camera1 plus user viewport manual-match.
- **LOCKED:** The existing `compute_flame_euler_zyx` / `flame_euler_to_cam_rot` pair implements `R = Rz(+rz) · Ry(-ry) · Rx(-rx)` — only rx/ry signs flipped. It is **wrong** as a model of Flame's rendered rotation, but it is self-consistent (round-trips through itself) and the Free-rig solve pipeline depends on that symmetry. It stays.

### New functions (additive)
- **LOCKED:** Add `compute_flame_euler_xyz(cam_rot) -> (rx_deg, ry_deg, rz_deg)` in `forge_core/math/rotations.py`.
- **LOCKED:** Add `flame_euler_xyz_to_cam_rot(rx_deg, ry_deg, rz_deg) -> cam_rot` as its exact inverse (same file).
- **LOCKED:** Both functions implement the XYZ-signflip convention `R = Rz(-rz) · Ry(-ry) · Rx(-rx)`.
- **LOCKED:** Numpy-only, no dependency on `forge_flame`. Follows the module's existing docstring, gimbal-lock (`1e-6`) and atan2 patterns verbatim.
- Must include docstrings that cite the 2026-04-24 verification (forge-bridge + viewport match) as the provenance of the convention, and cross-reference the fact that `_zyx` stays untouched.

### Reader — aim-rig branch of fbx_ascii
- **LOCKED:** In `forge_flame/fbx_ascii.py`, the aim-rig branch (static at `~L891`, per-frame at `~L959`) switches from `compute_flame_euler_zyx(R)` to `compute_flame_euler_xyz(R)`.
- **LOCKED:** Do not change anything else in `fbx_ascii.py`. Free-rig FBX readback, parser, and writer all stay on `_zyx`.
- **LOCKED:** Roll handling stays as shipped in Phase 04.2 (commit `c05516e`: pass `roll_deg = +cam.static_roll`, i.e. FBX-stored sign as-is). Do not re-litigate the roll negation.

### Blender subprocess scripts (aim-rig only)
- **LOCKED:** `tools/blender/bake_camera.py` — `_flame_euler_to_rot_matrix` currently implements `R = Rz(rz)·Ry(-ry)·Rx(-rx)` (ZYX convention). Replace with the XYZ-signflip convention (`Rz(-rz)·Ry(-ry)·Rx(-rx)`) using mathutils. This function is used on the Blender side when baking Flame-JSON cameras into the .blend scene; it must match the reader.
- **LOCKED:** `tools/blender/forge_sender/flame_math.py` — `_rot3_to_flame_euler_deg` currently mirrors `compute_flame_euler_zyx`. Replace with a decomposition that mirrors the new `compute_flame_euler_xyz` (using mathutils / numpy-equivalent scalar ops — same gimbal tolerance, same atan2 structure).
- **LOCKED:** `tools/blender/extract_camera.py` — no direct math changes; it already delegates to `_rot3_to_flame_euler_deg` from `flame_math.py`. Review the module docstring and update the "mirrors … compute_flame_euler_zyx" cross-reference so it points at `compute_flame_euler_xyz`.
- **LOCKED:** Keep the "if you touch one, touch the other" cross-reference comments in each Blender script; just retarget them at the `_xyz` functions.

### Pipeline symmetry (the non-negotiable)
- **LOCKED:** The reader, bake, extract, and flame_math changes land together, in a single phase, in one wave of commits that leaves the tree consistent on every intermediate revision. No half-applied states. This is the direct lesson from the 04.2 revert — `3e25b5e` landing the fix in the reader alone is what hard-crashed `importToActionFBX`, because bake/flame_math still emitted `_zyx`-decomposed matrices into the FBX, and Flame's importer hit an internal edge case.

### Tests
- **LOCKED:** Add direct unit tests for `compute_flame_euler_xyz` / `flame_euler_xyz_to_cam_rot`. The tests must **not** use `flame_euler_xyz_to_cam_rot` as their own oracle — that is the exact anti-pattern that masked the 04.2 bug (tests used `flame_euler_to_cam_rot` as the oracle for `compute_flame_euler_zyx`, so they verified round-tripping, not correctness). Instead:
  - Encode the Camera1 ground-truth Euler `(1.8193°, 1.0639°, 1.2529°)` and the Camera1 cam_rot derived from `rotation_matrix_from_look_at` (position, aim, up, roll from the reference data below) as a fixture, and assert the functions map between them within `1e-3°`.
  - Add explicit matrix-value tests with hand-computed 3x3 entries for simple rotations (e.g. pure rx, pure ry, pure rz at known angles) so a future convention regression fails loudly.
- **LOCKED:** Existing 380-test suite must stay green. Any failing test in `tests/test_rotations.py`, `tests/test_blender_roundtrip.py`, `tests/test_hook_parity.py`, `tests/test_aim_rig_*` or similar is a BLOCKER — it means adapter.py or a Free-rig caller silently depended on the aim-rig branch's convention, and we need to revisit scope.

### Install / deployment
- Out of scope. No `install.sh` changes expected. Blender script install path (`/opt/Autodesk/shared/forge_blender_scripts/`) is stable as of commit `64a96cd`.

### Claude's Discretion
- How to factor the Blender-side `_xyz` decomposition (inline in `flame_math.py` vs. add a helper) — as long as the convention mirrors `compute_flame_euler_xyz` exactly.
- How to structure the regression fixture for the Camera1 ground-truth test (standalone pytest fixture, inline tuple, or new `tests/fixtures/` file).
- Exact docstring wording.
- Test file placement: extending `tests/test_rotations.py` vs. a new `tests/test_rotations_xyz.py` — pick whichever keeps the module under 500 lines.

<deferred>
## Deferred Ideas

### Open question worth a short look before committing to the full pipeline swap
The 04.2 handoff explicitly flagged this:

> Is the 0.087° residual *entirely* from the Euler convention mismatch, or does `rotation_matrix_from_look_at` also have a bug? Matrix diff showed our look-at's `right` has `Y=0` pre-roll (as expected from `cross(forward, (0,1,0))`), and Flame's ground-truth `right` has `Y=+0.022`. The Rodrigues roll application closes that gap when the roll sign is right — but the residual in the `+X` component suggests either Up handling or a subtle convention difference in the basis construction itself.

**Disposition for Phase 04.3:** Treat this as a 30-minute investigative spike inside the first plan. If the hand-decomposed `(1.814°, 1.058°, 1.252°)` value above reproduces from our current `rotation_matrix_from_look_at(position, aim, up, roll)` output when manually decomposed with the new XYZ-signflip convention, the convention is 100% the culprit and no look-at change is needed. If it doesn't reproduce, add a targeted basis-construction fix inside this phase. **Do not** expand this phase into a general solver-quality review — if the look-at math has a deeper bug, log it as Phase 04.4 and ship the convention fix regardless (it's a strict improvement and unblocks a cleaner look-at investigation).

### Explicitly out of scope for 04.3
- Free-rig convention cleanup (`adapter.py` + any other `_zyx` caller). Out-of-scope per handoff; requires Free-rig visual regression testing and a deliberate deprecation of `_zyx`.
- Tolerance tightening in ROADMAP (0.1° → 0.01°). Document in the phase SUMMARY as a recommendation; ROADMAP update is a Phase 04.4 doc task.
- `rotation_order` per-camera support (Flame's camera attribute `"Rot XYZ"`). Currently hardcoded; live-probe confirmed Camera1 uses `"Rot XYZ"` — matches what we're now implementing. Generalising to per-camera order is out of scope.
- Precision tuning beyond 0.01° (sub-arcsecond). Not required for production CG registration given the plate is typically 2K/4K.
