# Phase 04.3 Plan 01: Aim-rig Euler Convention Fix — Summary

## TL;DR

Phase 04.3 closes the Phase 04.2 known limitation. Camera1 aim-rig
camera now round-trips Flame → Blender → Flame within **0.006°** of
Flame viewport ground truth on every Euler axis (was ~0.087° pre-phase).
That restores sub-pixel CG registration fidelity on aim-rig plates —
the project's stated core value.

The fix required adopting a different Euler convention for aim-rig
cameras (`R = Rz(-rz)·Ry(-ry)·Rx(-rx)` — same Z·Y·X product order as
the Free-rig path, but with rz ALSO sign-negated) AND a coupled
sign flip in `rotation_matrix_from_look_at`'s roll Rodrigues step.
Neither change alone reproduces ground truth; both must land
together in a single atomic commit. The Phase 04.2 hard-crash trap
(asymmetric reader/bake/extract conventions crashing Flame's
`importToActionFBX`) is avoided by construction.

## The Spike → Amendment → Corrected Execution Arc

Task 1's gating spike (already complete pre-session as commit `c449a2b`)
returned **Branch B**: the plan's symbolic decomposer derivation didn't
reproduce the CONTEXT.md hand-decomposed `(1.814°, 1.058°, 1.252°)` from
the look-at output. The empirical fit revealed two coupled changes were
needed:

1. **`rotation_matrix_from_look_at` L167 sign flip** (`-roll_deg → +roll_deg`).
2. **Decomposer convention** — the precise matrix-product order.

The plan amendment (`04.3-01-PLAN-AMENDMENT.md`) authorized both
changes in the atomic Task 3 commit and provided a corrected symbolic
derivation. **Critically, the amendment author got the matrix-product
order wrong** — see `Deviations` below.

## Deviations from Plan

### Auto-applied Rule 1 — Bug fix in amendment §A's symbolic derivation

**Found during:** Task 3, after applying amendment §A's X·Y·Z product
form to `compute_flame_euler_xyz`, the test
`TestComputeFlameEulerXyz::test_camera1_known_answer` failed:
```
compute_flame_euler_xyz(R) = (1.837005, 1.017240, 1.285145)
expected hand-decomposed     = (1.814,    1.058,    1.252)
```

**Diagnosis:** Verified numerically that:
- Amendment's `R = Rx(-rx)·Ry(-ry)·Rz(-rz)` (X·Y·Z product) — diff vs
  look-at: 7e-4. **Wrong.**
- Original plan's `R = Rz(-rz)·Ry(-ry)·Rx(-rx)` (Z·Y·X product) — diff
  vs look-at: 6e-6. **Correct.**

The amendment author misinterpreted scipy's extrinsic `xyz` mode:
extrinsic rotations apply in REVERSE of the string order, so
extrinsic `xyz` means `R = Rz·Ry·Rx` (Z·Y·X product), not `Rx·Ry·Rz`.
The original plan's `<symbolic_derivation_for_xyz_decomposer>` block
was actually correct.

**Fix:** Used the original plan's Z·Y·X derivation in Task 3's atomic
commit:
- `compute_flame_euler_xyz`: atan2 indices reverted to the original
  plan's form (`-arctan2(R[2,1], R[2,2])`, `-arcsin(-R[2,0])`,
  `-arctan2(R[1,0], R[0,0])`).
- `flame_euler_xyz_to_cam_rot`: matrix product order `Rz @ Ry @ Rx`
  (was `Rx @ Ry @ Rz`).
- Module docstring + function docstrings updated to reflect Z·Y·X.
- `_rot3_to_flame_euler_deg` and `_flame_euler_to_rot_matrix` already
  used the correct Z·Y·X form (the amendment §F grep targets I
  followed were correct — only §A's symbolic derivation was wrong).
- Gimbal-lock test in `test_rotations.py` re-derived for Z·Y·X
  product (the amendment §C hand-built matrix was for X·Y·Z).

**Files modified by this Rule 1 fix (all in Task 3's atomic commit):**
- `forge_core/math/rotations.py`
- `tests/test_rotations.py`

**Why the bug was caught:** the `Camera1 known-answer` test, which uses
the hand-decomposed `(1.814°, 1.058°, 1.252°)` as a NON-circular oracle.
This is exactly the anti-pattern guard the plan emphasised
(`PATTERNS.md §"Anti-Pattern to AVOID"`); without it, the X·Y·Z form
would have round-tripped through `flame_euler_xyz_to_cam_rot` cleanly
and shipped to UAT before failing on real plates. The non-circular
oracle paid off.

**Why Task 2's commit (fdadf65) shipped the wrong body:** Task 2
landed `compute_flame_euler_xyz` and `flame_euler_xyz_to_cam_rot`
purely additively — no caller wired in. The Camera1 known-answer test
was xfailed because the L167 sign flip hadn't landed yet, so the bug
was masked. The Task 3 commit corrected the body and unxfailed the
test simultaneously, so the buggy intermediate state never had any
observable behavioral effect on the project.

## Patterns Established

1. **Pipeline-symmetric atomic landing** — when a multi-site math
   convention changes (reader, bake, extract, support helper), they
   all swap in ONE commit with broken-in-lockstep tests bundled. No
   intermediate revision has half the pipeline on the new convention.
   Mitigates the Phase 04.2 hard-crash class of failures.

2. **Convention coexistence in a single module** — `forge_core.math.rotations`
   now hosts BOTH `_zyx` (free-rig) and `_xyz` (aim-rig) pairs side-by-side
   with disjoint callers. Module docstring documents who uses which.
   Keeps the additive change orthogonal to existing solver code.

3. **Non-circular oracle for math regression tests** — hand-decomposed
   ground truth used as INPUT and ORACLE, not routed through the
   composer-as-decomposer-oracle pattern that masked the Phase 04.2
   bug. This pattern caught the §A derivation error in Task 3 before
   it shipped.

4. **xfail-then-unxfail handoff for cross-task lockstep tests** — Task 2
   landed the helper with a Camera1 xfail keyed to Task 3's coupled
   change; Task 3 unxfailed it. The xfail acted as a TODO marker that
   pytest would surface on every CI run if Task 3 didn't land.
