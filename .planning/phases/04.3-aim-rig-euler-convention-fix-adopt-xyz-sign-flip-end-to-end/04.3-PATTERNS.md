**Composition pattern (the inverse — `flame_euler_to_cam_rot`)** (lines 62-80) — copy structure, flip signs:

```python
def flame_euler_to_cam_rot(rx_deg: float, ry_deg: float, rz_deg: float) -> np.ndarray:
    """Compose a 3x3 cam-to-world rotation from Flame's Euler triple.
    ...
    Used by the Flame → Blender bake path (rebuild world matrix from
    stored JSON) and by tests that need to generate known-answer
    rotations without round-tripping through solve_2vp."""
    rx, ry, rz = np.radians([rx_deg, ry_deg, rz_deg])
    cx, sx = np.cos(-rx), np.sin(-rx)
    cy, sy = np.cos(-ry), np.sin(-ry)
    cz, sz = np.cos(rz),  np.sin(rz)            # <-- ONLY THIS LINE DIFFERS:
                                                 #     new `_xyz` uses np.cos(-rz), np.sin(-rz)
    Rx = np.array([[1, 0, 0], [0, cx, -sx], [0, sx, cx]])
    Ry = np.array([[cy, 0, sy], [0, 1, 0], [-sy, 0, cy]])
    Rz = np.array([[cz, -sz, 0], [sz, cz, 0], [0, 0, 1]])
    return Rz @ Ry @ Rx
```

**Decomposition pattern (`compute_flame_euler_zyx`)** (lines 33-59) — copy structure. The atan2 indices change because a pure sign flip on rz moves the non-zero terms into different matrix cells.

```python
def compute_flame_euler_zyx(cam_rot: np.ndarray) -> Tuple[float, float, float]:
    """Decompose a 3x3 cam-to-world rotation into Flame's Euler convention.
    ...
    Handles gimbal lock (ry = ±90°) by pinning rx=0 and deriving rz from
    the upper-left 2x2 block."""
    cb = np.sqrt(cam_rot[0, 0] ** 2 + cam_rot[1, 0] ** 2)
    gimbal = cb <= 1e-6
    if not gimbal:
        rx = -np.arctan2(cam_rot[2, 1], cam_rot[2, 2])
        ry = -np.arcsin(-cam_rot[2, 0])
        rz = np.arctan2(cam_rot[1, 0], cam_rot[0, 0])
    else:
        rx = 0.0
        ry = -np.arcsin(-cam_rot[2, 0])
        rz = np.arctan2(-cam_rot[0, 1], cam_rot[1, 1])
    deg = np.degrees(np.array([rx, ry, rz]))
    return (float(deg[0]), float(deg[1]), float(deg[2]))
```

**Shape new code must take:**
- `compute_flame_euler_xyz(cam_rot) -> (rx_deg, ry_deg, rz_deg)` — same signature, same `cb <= 1e-6` gimbal tolerance, same `deg = np.degrees(np.array([rx, ry, rz])); return (float(deg[0]), ...)` return block.
- `flame_euler_xyz_to_cam_rot(rx_deg, ry_deg, rz_deg) -> np.ndarray` — same signature, same `np.radians([...])` → `cx, sx = np.cos(-rx), np.sin(-rx)` pattern, same `Rz @ Ry @ Rx` return.
- Docstring style: **Google-style with explicit provenance**. Must cite `forge-bridge probe + viewport manual-match 2026-04-24` and cross-reference `compute_flame_euler_zyx` with a one-line warning: "Does NOT modify or replace `_zyx`; Free-rig solve path depends on that pair's symmetric-pass-through behaviour."
- `from __future__ import annotations` already at top of file — no change needed.

**Gimbal tolerance:** `1e-6` on `cb = sqrt(R[0,0]**2 + R[1,0]**2)`. **Keep this identical** to `_zyx`.

**Signs to flip (the exact delta from `_zyx`):**
- `compute_flame_euler_xyz`: the atan2 row/col indices must be re-derived from `R = Rz(-rz)·Ry(-ry)·Rx(-rx)`. **Do not reuse the `_zyx` atan2 expressions** — expand the matrix product symbolically, then extract. A pure sign flip on `rz` (same indices) will round-trip through `flame_euler_xyz_to_cam_rot` but will not correctly decompose an arbitrary `cam_rot` built by `rotation_matrix_from_look_at`. This is the exact trap CONTEXT.md flags under "Tests must not use the composer as oracle for the decomposer."
- `flame_euler_xyz_to_cam_rot`: flip the `cz, sz = np.cos(rz), np.sin(rz)` line to `cz, sz = np.cos(-rz), np.sin(-rz)`. All three angles now feed in as their negated forms.

```python
def _rot3_to_flame_euler_deg(R) -> tuple:
    """Decompose a 3x3 cam-to-world rotation into Flame's Euler triple.

    Flame composes rotations as R = Rz(rz) · Ry(-ry) · Rx(-rx). This is
    the matching inverse decomposition. Must stay numerically identical
    to forge_core.math.rotations.compute_flame_euler_zyx — if you change
    one, change both and run tests/test_blender_roundtrip.py.

    Handles gimbal lock (ry ≈ ±90°) by pinning rx=0 and recovering rz
    from the remaining 2x2 block."""
    # mathutils 3x3 Matrix indexes as [row][col]. Keep aligned with numpy.
    cb = math.sqrt(R[0][0] ** 2 + R[1][0] ** 2)
    gimbal = cb <= 1e-6
    if not gimbal:
        rx = -math.atan2(R[2][1], R[2][2])
        ry = -math.asin(-R[2][0])
        rz =  math.atan2(R[1][0], R[0][0])
    else:
        rx = 0.0
        ry = -math.asin(-R[2][0])
        rz =  math.atan2(-R[0][1], R[1][1])
    return (math.degrees(rx), math.degrees(ry), math.degrees(rz))
```

## Anti-Pattern to AVOID

**From CONTEXT.md (the 04.2 bug root-cause):**

> Plan 01's tests used `flame_euler_to_cam_rot` as their own oracle — that is the exact anti-pattern that masked the 04.2 bug (tests used `flame_euler_to_cam_rot` as the oracle for `compute_flame_euler_zyx`, so they verified round-tripping, not correctness).

**Concrete example of the anti-pattern** — from `tests/test_blender_roundtrip.py` L177-L185, `TestEulerHelperInverse`:

```python
@pytest.mark.parametrize("rot", [c[2] for c in CAMERAS], ids=[c[0] for c in CAMERAS])
def test_euler_matrix_roundtrip(self, rot):
    R = flame_euler_to_cam_rot(*rot)      # composer
    rot_recovered = compute_flame_euler_zyx(R)  # decomposer
    np.testing.assert_allclose(rot_recovered, rot, atol=1e-9)  # only proves inverse, not correctness
```

And in `tests/test_forge_sender_flame_math.py` L97-L108, `test_rot3_to_flame_euler_deg_roundtrip`:

```python
R_np = _compose_flame_rotation_np(rx_in, ry_in, rz_in)   # uses flame_euler_to_cam_rot
R_m = _np_to_mathutils_matrix(R_np)
rx_out, ry_out, rz_out = _rot3_to_flame_euler_deg(R_m)
assert math.isclose(rx_out, rx_in, abs_tol=1e-6)
```

**For `_xyz` tests, the planner must instead:**
1. Pin an input `cam_rot` from either (a) the Camera1 `rotation_matrix_from_look_at` output (external oracle — the look-at helper is independently proven) OR (b) a hand-built 3x3 matrix with known entries.
2. Assert the output triple matches Flame viewport truth `(1.8193°, 1.0639°, 1.2529°)` within `0.01°` on the Camera1 case, and matches `(1.814°, 1.058°, 1.252°)` within `1e-3°` on the hand-decomposed case.
3. Add at least one pure-axis case per axis (e.g. `compute_flame_euler_xyz(Rz_of_30deg_hand_built_matrix)` must return `(0, 0, 30)` — or `(0, 0, -30)` depending on sign — pin the answer by hand).
