expected: When right-clicking a solved **static** Camera Match camera → Export Camera to Blender, the hook should route through the new static JSON path (`camera_io.export_flame_camera_to_json`), not the FBX bake path. If instead the FBX path fires (observed via a `forge_bake_*` tempdir in `$TMPDIR` and Blender's bake subprocess logs), then WR-01 is confirmed: `_is_animated_camera` misclassifies static cameras as animated, but the output is still correct post-rotation-hotfix. The rationale for unification: Flame's `export_fbx(bake_animation=True)` handles static cameras correctly (produces 2 identical keyframes) given all the downstream hotfixes (empty-block, TimeSpanStop, LocalTime, KeyAttr expansion, rotation negation, pixel_to_units=1.0, filmback preservation) landed in the same phase.

## Gaps

### G-01: Animated camera collapses to single frame on Export to Blender

discovered: 2026-04-23 during UAT item 1 investigation
severity: blocking for animated-camera workflows (core-value violation)
debug_session: .planning/debug/resolved/animated-camera-single-frame.md
status: resolved
description: When user right-clicks an animated Action camera → Camera Match → Export Camera to Blender, the resulting `.blend` contains only a SINGLE frame of animation instead of the camera's full N-frame motion. Root cause: `_is_animated_camera()` called `action.export_fbx(path=..., cameras=[cam], ...)` — Flame's C-extension rejects the unknown `cameras=` kwarg with TypeError, silently caught by the outer except, making every camera misclassify as static.
resolution: Commit `3d9e8b2` corrected the API call (use `action.selected_nodes.set_value([cam])` + `only_selected_nodes=True`). A cascade of subsequent hotfixes then closed downstream issues surfaced once animated data actually reached the FBX pipeline: rotation negation (`49bbe43`), Takes LocalTime (`45fb7fd`), GlobalSettings TimeSpanStop (`e6f8243`), empty-block `{ }` preservation + KeyAttr expansion (`895f097`), pixel_to_units=1.0 on return (`bbe5f43`), filmback preservation (`228387a`), and finally unification of static + animated paths (`9b7bdfc`). Full live round-trip verified correct for Free-rig animated cameras.

### G-02: Aim/Target-rig orientation lost in round-trip (deferred to next phase)

discovered: 2026-04-23 during final live UAT after all core hotfixes landed
severity: documented limitation; DOES ship in v6.3 — aim-rig cameras round-trip with orientation lost
status: deferred — scheduled for phase 04.2 as REQUIRED scope (per user 2026-04-23)
description: Flame's `action.export_fbx(bake_animation=True)` on an Aim/Target-rig camera writes `rotation=(0,0,0)` for every frame, discarding the aim+up+roll that defines the camera's real orientation. Translation keyframes are preserved correctly, but when the returned camera lands in Flame (as a Free-rig camera with rotation=0) it looks down -Z from its position regardless of where the original camera was actually pointing. Free-rig cameras (Camera Match solves) are unaffected — they use Euler rotation which FBX export preserves natively.
workaround: users with Aim/Target-rig cameras can switch the rig type to Free in Flame's UI before exporting. Flame will internally bake aim+up+roll into the equivalent Euler rotation at the moment of rig conversion, after which the FBX export carries the correct orientation.
rationale_for_deferral: aim-rig conversion wasn't in phase 04.1's original scope (stereo-rig filter, detect-and-route, fps stamp, crash instrumentation). The volume of downstream FBX-contract fixes delivered in 04.1 already exceeded plan. The aim→Euler math + Blender-side Track-To or rotation-bake handling is a self-contained workstream of its own.
