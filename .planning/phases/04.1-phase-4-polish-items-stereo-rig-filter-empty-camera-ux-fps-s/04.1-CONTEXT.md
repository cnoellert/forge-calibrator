<decisions>
## Implementation Decisions

### Scope (GA-1)
- **D-01:** All 5 ROADMAP items are in scope. Items 1, 2 are smoke-test-surfaced; items 3, 4, 5 ride along because the user chose full-scope over minimal-ship.

### Empty-camera bake (item 2, GA-2)
- **D-02:** Fix strategy is **detect-and-route**. The hook (`flame/camera_match_hook.py` around line 2099 where the temp dir is created and the FBX bake is kicked off) inspects the target camera and branches:
  - **Static camera** → call `forge_flame.camera_io.export_flame_camera_to_json` with the current `cam.position`/`cam.rotation`/`cam.fov`/`cam.focal` values at `flame.batch.start_frame`. Blender's `bake_camera.py` already handles single-frame JSON.
  - **Animated camera** → existing FBX bake path (`fbx_io.export_action_cameras_to_fbx(..., bake_animation=True)` → `fbx_ascii.fbx_to_v5_json`). Unchanged.
- **D-03:** Detection mechanism is **left to researcher/planner** — verify what Flame exposes for "does this camera have keyframes?" (candidates: `cam.position.values` length, `cam.keyframable_attributes`-style API, or a delta-read of `cam.position.get_value()` across two frames). Pick the most reliable per the Flame 2026.2.1 API and the rotation-convention memory notes.
- **D-04:** No fallback to fail-clearer popup required for the common case — the detect-and-route path covers both static and animated without user education. Keep existing error popups (`Failed to write FBX`, `Failed to convert FBX to JSON`) for genuine failures.

### Plural-popup filter (item 1, GA-3)
- **D-05:** Filter site is **Flame-side in `tools/blender/forge_sender/transport.py:_FLAME_SIDE_TEMPLATE`** — specifically the `"created": [...]` list comprehension. Applying the filter before the bridge returns means only accurate names cross the wire; the Blender-side popup renders the filtered list verbatim.
- **D-06:** Filter rule is **duck-type check** — keep only names whose corresponding `action.nodes` entry has `position`+`rotation`+`fov`+`focal` attributes (same pattern as `forge_flame/fbx_io.py:51-70 iter_keyframable_cameras`). Rejected options were name-pattern exclusion (brittle) and explicit-inclusion-list (narrow; drops legitimate new user-visible cameras).
- **D-07 (caveat):** The user flagged that the three GA-3b options were not ideal — the duck-type approach was chosen as the least-bad of three, not a clear winner. **If the planner or implementer identifies a better filter pattern during research or execution (e.g., a Flame API that exposes "is user-visible camera" directly), flag it back for a decision revisit before committing to D-06.**

### Task-5 crash investigation (item 3, GA-4)
- **D-08:** Do **both**: defensive instrumentation first, then one repro attempt.
  - **Instrumentation:** add logging to the Send-to-Flame path around `tools/blender/forge_sender/transport.py` (pre-import state, `fbx_ascii` parse, `import_fbx` call, post-import state). Logs land in `$TMPDIR/forge_send_*/` next to the existing `incoming.fbx` artifact. Instrumentation stays shipped — cheap insurance for future crashes, not just this one.
  - **Repro attempt:** one focused session against fresh batch + clean state + known test plate. N Send-to-Flame roundtrips until either the crash fires (capture diagnostic report, tempdir, stderr) or N is exhausted. If crash fires: route to `/gsd-debug` for fix in this phase or as an immediate follow-up. If N exhausts without crash: ship with instrumentation; next real-world occurrence self-captures.
- **D-09:** The crash is **not a v6.3 ship gate** — if the repro attempt fails, Phase 4.1 still closes on DOC-01 via the other 4 items. Crash investigation is tracked as a known-unknown with instrumentation in place.

### Phase 2 code review warnings (item 4, GA-6)
- **D-10:** Run `/gsd-code-review-fix 02` **before** Phase 4.1 plans start — this lands the two warning fixes as `fix(02-*)` commits that cross-reference the original `02-REVIEW.md`. Phase 4.1 plans stay focused on items 1, 2, 3, 5; item 4 is resolved as a prerequisite, not a plan.

### fps stamp in bake_camera.py (item 5, GA-5)
- **D-11:** **Always stamp** `forge_bake_frame_rate` on every baked Blender camera (no conditional logic). Makes the camera self-describing for future debugging and matches D-19's "caller-owned frame rate" intent end-to-end.
- **D-12:** fps source is the **Flame project fps** (`flame.batch.frame_rate`), **NOT** Blender's default `bpy.context.scene.render.fps`. Blender's scene fps on a fresh subprocess-launched Blender defaults to 24.0 unless the bake script explicitly sets it from the JSON, which makes blind-stamping of `scene.render.fps` unreliable.
- **D-13 (propagation mechanism — left to planner):** the Flame hook knows `flame.batch.frame_rate`; it must propagate that to the Blender bake subprocess via one of:
  - CLI arg passed to `bake_camera.py` (e.g., `--frame-rate "24 fps"`)
  - A `frame_rate` field added to the v5 JSON schema (preferred — keeps the camera data self-describing in the JSON and survives future pipeline changes)
  - Set `bpy.context.scene.render.fps` at the top of `bake_camera.py` before any stamping happens
  Planner picks the mechanism during research; the D-19 Blender-side ladder in `forge_sender/__init__.py:_resolve_frame_rate` is the consumer of the stamped value and should remain unchanged.
- **D-14 (defense-in-depth):** If the Flame-side propagation somehow fails, `bake_camera.py` falls back to `bpy.context.scene.render.fps / fps_base` with a stderr warning (`forge_bake_frame_rate: falling back to Blender scene fps — Flame-side propagation failed`). Never silently stamp 24.0 when the truth is unknown.

### Success criterion (GA-7)
- **D-15:** Phase 4.1 closes DOC-01 when the `tools/smoke-test/seamless-bridge-smoke.sh` smoke test exits 0 on **both** round-trip paths:
  1. **Fresh Camera Match solve path** (primary v6.3 headline): draw VP lines, solve, Export Camera to Blender → edit → Send to Flame → verify. This exercises items 1 + 2 + 5 end-to-end.
  2. **Animated Action path**: pre-existing animated camera → Export Camera to Blender → edit → Send to Flame → verify. Confirms the FBX path isn't regressed by the detect-and-route change.
- **D-16:** Task-5 crash repro is a separate quality gate (per D-09) — does not block DOC-01 closure.

### Claude's Discretion
- **Detection mechanism for "is this camera animated?"** (D-03) — researcher/planner picks per Flame 2026.2.1 API reality.
- **Instrumentation log format** (D-08) — plain key=value pairs or JSON; pick the most debuggable.
- **Repro attempt N-value** (D-08) — start with N=5; escalate if crash seems close.
- **fps propagation mechanism** (D-13) — JSON schema field preferred, but CLI arg or scene-fps-set are acceptable alternatives.
- **Where the instrumented logs land** (D-08) — default is `$TMPDIR/forge_send_*/` next to `incoming.fbx`; pick a cleaner home if warranted.

</decisions>

## Specific Ideas

- **Plural-popup filter revisit clause (D-07):** user explicitly flagged that the GA-3b options (name pattern, duck-type, explicit inclusion) "are not ideal". Duck-type was picked as least-worst. If a better Flame API surfaces during research, flag it — don't silently commit to the least-worst pick.

- **Detect-and-route precedent (D-02):** matches the existing architecture — `forge_flame/fbx_io.py` already has Perspective camera exclusion by duck-typing, and `camera_io.py` vs `fbx_io.py` is already a static-vs-animated split at the module boundary. Extending that split into the hook's export handler is a natural next step.
