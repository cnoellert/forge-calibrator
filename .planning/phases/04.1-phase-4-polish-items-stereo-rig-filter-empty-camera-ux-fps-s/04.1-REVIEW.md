## Warnings

### WR-01: `_is_animated_camera` misclassifies static-but-bake-sampled cameras as animated

**File:** `flame/camera_match_hook.py:1754-1772`

The probe calls `action.export_fbx(path=scratch_fbx, cameras=[cam], bake_animation=True)` and then classifies by `len(frames) >= 2`. Per the existing `TestBakedFixture` comment in `tests/test_fbx_ascii.py:378-381` ("For a static source camera, Flame's baker still emits a two-keyframe endpoint curve — so we expect two frames in the JSON, both at the same values"), a **static** camera emitted with `bake_animation=True` produces 2 frames, not 0-1 as the docstring claims.

Result: any static camera will be routed into the animated (FBX) branch. The JSON will have duplicate identical keyframes, which the Blender bake path then handles, but this defeats the stated point of the detect-and-route change (the plan called it "the bug this closes"). The docstring at lines 1719-1722 is wrong about the frame-count signature.

Additionally, the probe uses `fbx_to_v5_json` with its default `frame_rate="23.976 fps"`, not the project fps — also a small concern because post-260421-c1w clipping kwargs are not passed, so the probe will always count pre-roll + trailing frames, not true distinct per-frame keys.

**Fix:** Either change the threshold to treat 2 identical frames as static (`len(frames) > 2` *and* check that frames differ), or replace the FBX-count probe with a cheaper, direct probe (e.g., `hasattr(cam.position, "get_keys") and len(cam.position.get_keys())`). At minimum, update the docstring and add a test that exposes the current misclassification.

```python
# Option A: adjust threshold with distinctness check
if len(frames) < 2:
    return False
first = frames[0]
return any(
    f["position"] != first["position"]
    or f["rotation_flame_euler"] != first["rotation_flame_euler"]
    or f["focal_mm"] != first["focal_mm"]
    for f in frames[1:]
)
```

### WR-02: `_is_animated_camera` performs a full FBX bake of the Action on every export

**File:** `flame/camera_match_hook.py:1754-1760`

The detect helper calls `action.export_fbx(path=scratch_fbx, cameras=[cam], bake_animation=True)` on every invocation of the export handler, *before* the real export runs. The docstring flags this as "expensive" — it literally does "the FBX bake it's trying to avoid as a detection step." For animated cameras, this doubles end-to-end export time. For static cameras it replaces a cheap JSON write with two FBX bakes + two FBX parses. User-observed latency will increase on every export.

**Fix:** Replace the scratch-FBX probe with a direct PyAttribute keyframe check once a live-Flame probe confirms the API shape. The plan already flagged this as "bridge-offline probe deferred"; recommend scheduling that probe before shipping to users who export animated cameras frequently.

### WR-03: Dead `elif` branch in `_resolve_flame_project_fps_label` can emit garbage fps labels

**File:** `flame/camera_match_hook.py:1814-1822`

If `flame.batch.frame_rate` is a slot wrapper without `.get_value()` but evaluates truthy, the `elif` fires and coerces the raw slot via `_numeric_to_label` — potentially a Flame PyAttribute wrapper that `str(raw)` renders as something like `"<flame.PyAttribute object at 0x...>"`, producing a label like `"<flame.PyAttribute object at 0x...> fps"` that gets stamped into JSON.

The docstring also claims "Attempts `flame.batch.frame_rate.get_value()` first" but per D-19 this is operationally a near-always fallback to `'24 fps'`.

**Fix:** Tighten the fallback path to only accept known-usable types:

```python
try:
    import flame as _flame
    raw = _flame.batch.frame_rate
    if raw is not None:
        if hasattr(raw, "get_value"):
            val = raw.get_value()
            if val is not None:
                return _numeric_to_label(val)
        elif isinstance(raw, (str, int, float)):
            return _numeric_to_label(raw)
except Exception:
    pass
```

### IN-03: Empty-frames case emits `"frames": []` silently

**File:** `forge_flame/fbx_ascii.py:674-678, 709-712`

When `frame_start`/`frame_end` clip all frames, `_merge_curves` returns `[]` and `fbx_to_v5_json` writes `"frames": []` to disk. The hook tolerates this (returns 0). `bake_camera.py:281-282` correctly rejects it with a loud SystemExit. Out of scope for this phase, but a defensive raise when `frame_start > frame_end` would fail earlier and clearer.

### IN-04: Test H3 does not exercise the animated branch

**File:** `tests/test_hook_export_camera_to_blender.py:251-279`

Fakes `fake_export_fbx`, `fake_fbx_io` and monkeypatches `_is_animated_camera` → True, but never invokes `_export_camera_to_blender`. The only assertion is `assert callable(getattr(_hook_module, "_is_animated_camera", None))`, which passes whether the branch dispatches correctly or not. Effectively a name-presence test, not a branch-dispatch test.

**Fix:** Invoke `_export_camera_to_blender` with a mocked selection and assert `fake_fbx_io.export_action_cameras_to_fbx.called`; or label the test honestly.

### IN-05: Test H4 similarly fails to exercise the static branch

Same structural issue as IN-04 — elaborate mocks (`fake_export_json`, `fake_camera_io`, monkeypatched helpers) are set up but the handler is never invoked. Only asserts helper callability.

**Fix:** Same as IN-04.

### IN-06: Test H5 is source-grep regression guard, not a behavior test

**File:** `tests/test_hook_export_camera_to_blender.py:347-359`

`inspect.getsource(_hook_module)` + substring check is the weakest possible guarantee. Whitespace reformatting or f-string line-splitting would fail without changing behavior; a buggy refactor preserving literal strings would pass.
