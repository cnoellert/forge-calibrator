### Human Verification Required

#### 1. Confirm _is_animated_camera routing for a freshly-solved static camera

**Test:** Open Flame 2026.2.1, create a fresh batch, run Camera Match on a plate (no keyframes on the Action camera — default fresh-solve state). Right-click the Action node, select "Camera Match → Export Camera to Blender". Watch the hook log output in the Flame terminal.

**Expected:** One of:
- (a) The hook log shows no "Blender bake failed" and exports quickly (sub-1s would suggest JSON path; 2-5s suggests FBX bake double-hit)
- (b) After export, check the .blend: if it contains a single keyframe at the solve frame, the JSON path ran. If it contains 2 identical keyframes (batch start + end), the FBX path ran.

**Why human:** WR-01 from code review confirms `_is_animated_camera` uses `len(frames) >= 2` threshold. Flame's `bake_animation=True` on a static camera always produces 2 identical endpoint keyframes (confirmed by `TestBakedFixture` in `tests/test_fbx_ascii.py:379-381`). This means the function always returns True and the static branch never fires. Only a live Flame session can confirm the actual routing behavior and whether the intended optimization is working.

**Resolution paths:**
- If FBX path fires for static: the detect-and-route architectural intent is unmet but correctness is preserved. File a follow-up to fix the threshold (see WR-01 "Option A" in REVIEW.md) or replace with a direct API probe.
- If JSON path fires for static: WR-01 is invalid — Flame's behavior differs from the test fixture assumption. The implementation is working as designed.

#### 2. Confirm fps label is correct for the live Flame project (not just the D-14 fallback "24 fps")

**Test:** In Flame 2026.2.1, set the project to a non-24fps rate (e.g. 25fps or 23.976fps). Run Camera Match on a plate, apply to Action, export to Blender. In Blender, select the baked camera. Open Properties > Object Data > Custom Properties. Check the value of `forge_bake_frame_rate`.

**Expected:** The custom property shows the correct project fps label (e.g. "25 fps" if the project is 25fps), not "24 fps" (which would indicate D-14 fallback fired).

**Why human:** `_resolve_flame_project_fps_label()` uses `flame.batch.frame_rate` as its source, which Phase 2 D-19 asserts was a NoneType slot on Flame 2026.2.1. If still a NoneType slot, the function falls through to the D-14 "24 fps" fallback regardless of actual project fps. Only a live Flame session at a non-24fps project rate can confirm whether the correct label is propagated or whether users always get "24 fps" from D-14.

## Overall Verdict

Phase 04.1 has successfully shipped code for all three blocking polish items (GA-2, GA-3, GA-4, GA-5). The test suite is clean: 352 passed, 2 skipped. The core-value invariant (geometric fidelity through the Flame-Blender roundtrip) is preserved — confirmed by live user validation after the rotation hotfix (commit 49bbe43).

Two advisory findings from the code review (WR-01 and WR-03) require live Flame confirmation before the phase can be marked fully passed:

1. **WR-01** (static camera routing): The detect-and-route optimization may not be firing for static cameras, meaning static exports use FBX-double-bake overhead. Correctness is preserved; efficiency is not. This is advisory, not a regression.

2. **WR-03** (fps label sourcing): The primary fps source (`flame.batch.frame_rate`) was a NoneType slot in Phase 2. If still so, every export silently gets "24 fps" regardless of project fps. Users would not see an error, but the stamp would be incorrect for non-24fps projects.

Neither gap prevents v6.3 shipping if the user consciously accepts them. If confirmed acceptable via live testing (or override), status can be promoted to `passed`.
