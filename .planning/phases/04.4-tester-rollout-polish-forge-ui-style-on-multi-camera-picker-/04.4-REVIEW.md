# Phase 04.4: Code Review Report

**Reviewed:** 2026-04-26
**Depth:** standard
**Files Reviewed:** 8
**Status:** issues_found

## Summary

Phase 04.4 lands the FORGE/Camera batch menu restructure, the FORGE-styled
multi-camera picker, the new action-schematic right-click handler, and the
Blender-side `FORGE_OT_send_to_flame_choose_action` operator that handles
cameras lacking forge-bake metadata. Security posture on the new
`make_create_code` is correct — `repr()` is used consistently to embed
user-provided strings, and the Wave 0 tests grep-anchor the contract.

The findings below cluster around three themes:

1. **Menu-path string drift**: the new batch menu is three levels deep
   (`FORGE → Camera → Export Camera to Blender`), but the preflight error,
   docs, and one stale Tier-1 popup either omit the `Camera` segment or
   still use the legacy "Camera Match hook" wording. A user troubleshooting
   from these strings will not find the menu Flame actually renders.
2. **Brittle bridge-error substring matching**: the collision-detection
   path keys on the literal Flame error text "Could not set Batch node
   name". The Wave 0 grep test guards against rename drift in our source,
   but it cannot detect a wording change in Flame itself — that path will
   fall through to the generic remote-tier popup with no warning.
3. **Minor robustness gaps in the new choose-Action operator**: an
   uncaught `TypeError` path on malformed bridge results, a couple of
   minor dead-code / unused-import items.

No critical security or correctness issues found. The math layer and
round-trip path are untouched in this phase.

### WR-03: Uncaught TypeError on malformed create-Action bridge result

**File:** `tools/blender/forge_sender/__init__.py:469-478`
**Issue:** After a successful create-Action POST, the code parses the
result via `ast.literal_eval(envelope["result"])` inside a try block that
catches `(ValueError, SyntaxError, KeyError)`. If the bridge ever returns
`{"result": None, "error": ""}` (or any non-string non-mapping result),
`ast.literal_eval(None)` raises `TypeError`, which propagates up as an
unhandled exception and surfaces in Blender as a red traceback in the
console rather than the intended Tier-1 popup.

The bridge's REPL contract (memory/flame_bridge_repl_contract.md) makes
the result a `repr()` string in the happy path, but a malformed envelope
or future bridge change could deliver `None`.

**Fix:**
```python
# tools/blender/forge_sender/__init__.py:471
except (ValueError, SyntaxError, KeyError, TypeError):
```

### WR-04: Collision-detection substring match is one Flame version away from breaking silently

**File:** `tools/blender/forge_sender/__init__.py:451-461`
**File:** `tools/blender/forge_sender/transport.py:347-365`
**Issue:** The choose-Action collision branch keys on the literal substring
`"Could not set Batch node name"` in the bridge envelope's `error` field.
The Wave 0 test `test_create_action_name_collision_error_string_match` is
a source-text canary — it grep-asserts the literal stays in transport.py.
Useful, but it cannot detect an upstream change: if Autodesk reworks the
RuntimeError text in Flame 2026.3, this path falls through to the generic
remote-tier popup with no warning, and users see a copy-pasted Flame
traceback instead of the friendly "An Action named 'X' already exists"
message.

This is the brittleness the phase context called out. Two mitigations
worth considering:
- Add a structured assertion in the bridge code body: catch RuntimeError
  in the create payload, inspect the message, and return a stable code
  the addon can match (e.g. JSON `{"error_code": "name_collision"}`).
- Pre-check name uniqueness before attempting `set_value`: list existing
  action names via the same `_CODE_LIST_ACTIONS` query and reject the
  name client-side. Avoids the round-trip on collision and decouples from
  Flame's error text.

Either mitigation removes the failure mode where the helpful message
silently regresses to a stack trace. This is a v6 hardening item, not a
correctness bug for this phase ship — flagging because the phase context
explicitly asked us to watch for it.

**Fix (suggested, pre-check approach in transport.py):**
```python
def make_create_code(name: str) -> str:
    ...
    return (
        "import flame\n"
        "_existing = [n.name.get_value() for n in flame.batch.nodes "
        "if hasattr(n.type, 'get_value') and n.type.get_value() == 'Action']\n"
        f"_name = {name!r}\n"
        "if _name in _existing:\n"
        "    raise RuntimeError('FORGE_NAME_COLLISION: ' + _name)\n"
        "_n = flame.batch.create_node('Action')\n"
        "_n.name.set_value(_name)\n"
        "_n.name.get_value()\n"
    )
```
Then match on the stable `FORGE_NAME_COLLISION:` prefix instead of
Flame's wording.

### IN-05: EnumProperty identifiers will fail on non-ASCII Flame action names

**File:** `tools/blender/forge_sender/__init__.py:365-376`
**Issue:** `_get_action_items` builds enum tuples as `(name, name, "")`
where `name` is the Action's Flame name straight from the bridge.
Blender's EnumProperty `identifier` (the first tuple element) requires
ASCII and uniqueness. An Action named "Caméra" or one containing path
separators will either crash Blender on dialog open or render a mangled
identifier that fails to round-trip back to `self.target_action` in
`execute()`.

The forge-calibrator project is internal VFX tooling and most Action
names are ASCII, so this is a low-likelihood failure mode. Worth a
defensive sanitize that maps the identifier to an ASCII-safe slug while
preserving the display name:

**Fix:**
```python
def _get_action_items(self, ctx):
    items = getattr(self, "_cached_actions", None) or []
    result = []
    for name in items:
        # Identifier must be ASCII; display name preserves original.
        ident = name.encode("ascii", "replace").decode("ascii")
        result.append((ident, name, ""))
    result.append(("__create_new__", "-- Create New --",
                   "Create a new Action in Flame"))
    return result
```
Then map `self.target_action` back to the original via a stored dict in
`invoke()` before posting. (Or accept the limitation and document it as
"Blender-side picker requires ASCII Action names; rename in Flame first
or use the right-click → FORGE → Camera → Export Camera to Blender entry
point which has no such constraint.")
