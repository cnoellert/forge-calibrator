## Decisions Made

- **Followed PATTERNS verbatim.** Every new symbol — `list_batch_actions`, `make_create_code`, `FORGE_OT_send_to_flame_choose_action`, the panel extension — matches the analog adaptation block in 04.4-PATTERNS.md exactly. RESEARCH P-03's correction to D-09 (RuntimeError on collision, not auto-suffix) is the single semantic change versus CONTEXT, and is fully baked into both the `make_create_code` docstring (canary substring) and the operator's `execute()` substring-match branch.
- **Top-level `import ast` for the operator.** Placed alongside `import json` rather than lazy-imported inside `execute()`. Rationale: bpy operators load once at addon-register-time; lazy import inside the hot path adds no measurable benefit and obscures the dependency. Mirrors the existing `requests` / `json` / `typing` top-level pattern in the file.
- **`{name!r}` f-string syntax over explicit `repr(name)` call.** They're identical per Python language spec, but `{name!r}` mirrors the existing pattern at `transport.py:237-240` (`json_str_repr=repr(v5_json_str)` is the build_payload pattern; the new make_create_code uses `{name!r}` inline because there's no `.format()` indirection — the f-string is the template). The Wave-0 test test_make_create_code_embeds_name_via_repr asserts on the rendered output (single-quoted form), which both approaches satisfy.
- **Operator registered before the panel in `_CLASSES`.** Blender resolves `bl_idname` references at register-time; if the panel registers before the operator, the operator's `bl_idname` reference in `layout.operator("forge.send_to_flame_choose_action", ...)` would still work at draw-time but would fail any register-time validation. Following the existing `(FORGE_OT_send_to_flame, VIEW3D_PT_forge_sender)` pattern.
- **Did NOT auto-delete the orphan auto-named "actionN" node Flame leaves behind on collision.** Flame's `create_node("Action")` succeeds (returns auto-named PyActionNode) BEFORE `name.set_value()` raises on collision. The user-perceived effect is "I tried to create 'BlenderCam' and got an error, but now there's an extra 'action7' node in my Batch." RESEARCH §P-03 explicitly identifies this as a known limitation; the plan's `<how-to-verify>` step 7 also flags it. The plan author left the auto-delete-on-failure path as a future Wave-3.5 follow-up if UAT shows it confuses users. For now the user-recover flow is "pick a different name from the dialog (or pick the existing one)."
- **No bridge-side Qt UI.** Per memory/flame_bridge_qt_main_thread.md, the addon's bridge calls (list_batch_actions / make_create_code) only touch `flame.batch.nodes` and `flame.batch.create_node` — no QMessageBox, no QInputDialog, no QDialog.exec(). All user-facing dialogs are Blender-side (wm.invoke_props_dialog) or Blender popups (wm.popup_menu via _popup helpers). This honors the bridge-safety mandate.
