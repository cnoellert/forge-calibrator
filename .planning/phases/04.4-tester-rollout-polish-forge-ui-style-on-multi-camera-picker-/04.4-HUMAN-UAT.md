## Closed

- **GAP-04.4-UAT-01 (`OQ-2 fallback unimplemented`):** The Camera-node-scope right-click
  path (Plan 04.4-03 / `_first_camera_in_action_selection`) trusts `cam.parent` to
  return the containing `PyActionNode`. In the actual Flame hook callback context
  (Flame 2026.2.1 arm64, macOS), `cam.parent` returns an object whose chained API
  (likely `.export_fbx` or one of its callees) resolves to `None`, so the call from
  `forge_flame.fbx_io.export_action_cameras_to_fbx` raises
  `'NoneType' object is not callable` and the user gets a Tier-1 "Failed to write
  FBX" dialog. Plan 04.4-03 already documented the fallback (scan
  `flame.batch.nodes` for the Action whose `.nodes` contains `cam_node`) but
  deferred implementation pending UAT. UAT has now confirmed the parent path fails.
  Scope: rewrite `_first_camera_in_action_selection` in
  `flame/camera_match_hook.py` to scan `flame.batch.nodes` (with `cam.parent`
  preserved as the fast path inside try/except) so the export pipeline sees the
  same PyActionNode that the Action-scope right-click path produces.
  - Closed: 2026-04-27 by Plan 04.4-07 / Task 1.
  - **Fix:** flame/camera_match_hook.py:_first_camera_in_action_selection
    rewritten with cam.parent fast path + flame.batch.nodes scan fallback
    (identity comparison `inode is item`). Mirrors _find_action_cameras's
    iteration shape.
  - **Tests:** tests/test_camera_match_hook.py — 5 new tests covering
    fast path, parent-raises fallback, parent-returns-None fallback,
    none-pair-on-empty-selection, perspective-skip-then-real-cam.
  - **Verified:** code-level (pytest); next live Flame-restart UAT will
    reconfirm sub-test (3) of Test 2 passes end-to-end.

- **GAP-04.4-UAT-02 (`forge_bake_camera_name not applied on import`):** The Blender →
  Flame send path (`FORGE_OT_send_to_flame_choose_action.execute` and the existing
  v1.2.0 stamped path both stamp `forge_bake_camera_name` into the v5 payload's
  `custom_properties`. The bridge-side `_forge_send` consumes
  `forge_bake_action_name` to find the target Action but DROPS
  `forge_bake_camera_name` — the imported camera takes whatever name the FBX
  template carries (observed: 'Camera1', not the Blender-side name). End-to-end
  effect: a tester who renames a camera in Blender and sends it to Flame sees
  the Flame-side camera ignore that rename and ship as the FBX default. Scope:
  read `forge_bake_camera_name` in `_forge_send`, then either rename the imported
  camera nodes via `created[i].name.set_value(...)` after `import_fbx_to_action`
  returns, or patch the FBX template's camera name field before write.
  - Closed: 2026-04-27 by Plan 04.4-07 / Tasks 3+4 (closes GAP-04.4-UAT-02).
  - **Fix:** tools/blender/forge_sender/transport.py:_FLAME_SIDE_TEMPLATE
    extended with rename block that reads
    `payload["custom_properties"]["forge_bake_camera_name"]` and calls
    `c.name.set_value(...)` on the first duck-typed camera in `created`
    (after import_fbx_to_action returns, before the success envelope).
    Stereo-rig siblings + RootNode_* + Perspective preserved unchanged.
  - **Tests:** tests/test_forge_sender_transport.py — new
    test_template_renames_imported_camera_via_forge_bake_camera_name
    in TestBuildPayload pins template-text invariants and JSON
    round-trip. Bridge-side runtime behaviour is Flame-only and
    out-of-scope-for-pytest (HUMAN-UAT GAP-2 fix scope verbatim).
  - **Verified:** code-level (pytest); next live UAT will reconfirm
    Blender-side rename round-trips end-to-end.

- **GAP-04.4-UAT-03 (`stamped-path send regression after Plan 04.4-07`):**
  Stamped-camera return-trip (Blender → Flame) crashed Flame after the
  Plan 04.4-07 rename block hit a sibling-name collision and called
  `c.name.set_value(forge_bake_camera_name)` on a Flame node already
  using that name. Per RESEARCH §P-03 the collision raises RuntimeError
  AND can destabilise Flame internal state (the SIGSEGV signature here
  matched the earlier 0x00001508 import races).
  - Closed: 2026-04-27 by hot-fix at 53e6163.
  - **Fix:** tools/blender/forge_sender/transport.py:_FLAME_SIDE_TEMPLATE
    rename block enumerates sibling node names from `action.nodes`
    BEFORE attempting `set_value`; skips the rename with a
    `step=rename_skipped reason=collision target=<name>` log line
    when the target name is already taken. The choose-Action create-new
    path is unaffected (fresh Action has no sibling cameras). The
    stamped path now preserves v1.2.0 behaviour (imported camera keeps
    Flame's auto-suffixed name alongside the original).
  - **Verified:** live 2026-04-27 PM — stamped-camera send round-tripped
    cleanly with no SIGSEGV; bridge debug log shows the
    `step=rename_skipped` line on collision path.

- **GAP-04.4-UAT-04 (`Camera-scope NoneType — stale pycache + family-wrapper proxy`):**
  Two distinct bugs surfaced under one error signature:

  (a) `bash install.sh` cleared `/opt/Autodesk/shared/python/camera_match/__pycache__`
  but NOT the sibling `forge_flame/__pycache__` or `forge_core/__pycache__`,
  so Flame loaded stale .pyc files for `forge_flame.fbx_io` and the
  Plan 04.4-07 source updates didn't take effect. Resolved by purging
  all sibling pycaches before Flame restart (saved as
  memory/flame_install_pycache_gap.md).

  (b) Plan 04.4-07's `_first_camera_in_action_selection` fast path
  filtered on `hasattr(parent, "nodes")` only — but in the hook
  callback context, `cam.parent` returned a `PyActionFamilyNode`
  base-class wrapper whose `.nodes` was populated but whose
  `.export_fbx` was None (the family-base class lacks the Action-
  specific export methods). The fast path returned the broken proxy,
  and `action.export_fbx(...)` then raised
  `'NoneType' object is not callable`.
  - Closed: 2026-04-27 by hardening the fast-path filter.
  - **Fix:** flame/camera_match_hook.py:_first_camera_in_action_selection
    fast path now also requires
    `callable(getattr(parent, "export_fbx", None))`. If the parent is
    a family-wrapper proxy, the filter rejects it and the fallback
    `flame.batch.nodes` scan runs (which returns the healthy
    `PyActionNode` outside the family-wrapper window).
  - **Tests:** tests/test_camera_match_hook.py — new
    test_first_camera_parent_with_broken_export_fbx_falls_back covers
    the broken-proxy case; _FakeActionNode constructor extended with
    a `_UNSET` sentinel + explicit `export_fbx=None` parameter so
    tests can simulate either healthy or broken parents.
  - **Verified:** live 2026-04-27 PM — both Camera-scope and
    Action-scope right-clicks complete cleanly post-restart; the
    PyActionFamilyNode wrapper for the active action also turned
    out to be a transient state cleared by Flame's full restart.

- **GAP-04.4-UAT-06 (`_merge_curves static-fallback emits empty frames`):**
  Surfaced during GAP-04 closure cycle: a static camera with no
  baked animation curves (Flame's `export_fbx(bake_animation=True)`
  occasionally produces zero curves on certain Action states) routed
  through `_merge_curves`'s static-fallback branch, which emitted a
  single frame at `frame = frame_offset` (= `start_frame - 1` in the
  standard hook call). The next line dropped any frame
  `< frame_start` (= `start_frame`), producing `frames=[]` and
  tripping the Blender bake script's "no frames in JSON" guard. The
  pre-fix comment claimed "rarely trips" but the case fires
  deterministically for any static camera in a batch with
  `start_frame > 0`.
  - Closed: 2026-04-27 inline.
  - **Fix:** forge_flame/fbx_ascii.py:_merge_curves static-fallback
    branch now sets
    `frame = frame_start if frame_start is not None else frame_offset`,
    so the static frame lands on the first batch frame instead of one
    pre-roll frame before it.
  - **Tests:** tests/test_fbx_ascii.py — new
    TestMergeCurvesStaticFallbackFrameStart class with three tests
    covering the standard hook pairing, no-batch-info fallback, and
    frame_end clip interaction.
  - **Verified:** code-level (pytest 426 pass); live live UAT
    captured the bug end-to-end before the fix.
