## Open Gaps

Three open gaps surfaced during the post-Plan-04.4-07 live re-verification
on 2026-04-27 PM. Status of Test 2 reverted from `pass` to `issue`
pending these closures.

- **GAP-04.4-UAT-03 (`stamped-path send regression after Plan 04.4-07`):**
  After Plan 04.4-07's GAP-2 rename block landed in
  `tools/blender/forge_sender/transport.py:_FLAME_SIDE_TEMPLATE`,
  sending a stamped (Flame-baked, returning round-trip) camera back
  to Flame crashes Flame (SIGSEGV/SIGABRT — same signature as the
  earlier 0x00001508 import races). Root cause hypothesis: the
  rename block calls `c.name.set_value(forge_bake_camera_name)` on
  the imported camera; the original camera with the same name still
  lives in the target Action; Flame's import_fbx auto-suffixed the
  imported one (e.g. 'Camera2'); the subsequent set_value collides
  with the original; per RESEARCH §P-03 collision raises RuntimeError
  AND can destabilise Flame internal state.
  - **Hot-fix already committed (53e6163):** collision-guard added
    to the rename block — enumerate sibling node names from
    `action.nodes` BEFORE attempting `set_value`; skip the rename
    with a `step=rename_skipped` log line if `forge_bake_camera_name`
    is already taken by a sibling. The choose-Action create-new path
    is unaffected (fresh Action has no sibling cameras). The
    stamped path now preserves v1.2.0 behaviour (imported camera
    keeps Flame's auto-suffixed name alongside the original).
  - **Status as of handoff:** the hot-fix is committed but NOT yet
    verified in a live Flame run. The fix lives in the Blender-side
    addon source — the user's installed copy at
    `~/Library/Application Support/Blender/4.5/scripts/addons/forge_sender/transport.py`
    must be refreshed (either via zip rebuild + Prefs reinstall, or
    direct `cp` of the source file) and Blender restarted.
  - **Live re-verification owed before close:** stamped-camera send,
    confirm Flame survives, confirm the bridge debug log shows
    `step=rename_skipped reason=collision target='Camera1'` (or the
    relevant name).

- **GAP-04.4-UAT-05 (`hook fails on 3D cameras inside Action`):**
  User reports the Export Camera to Blender right-click works on
  "regular" cameras inside an Action's schematic but fails on "3D
  cameras". Flame distinguishes camera node variants (standard
  Camera, Camera 3D, aim-rig cameras with Cam>Camera1 nesting per
  memory/flame_target_ground_truth_anchor.md, etc.). The current
  duck-type filter
  `hasattr(c, a) for a in ("position","rotation","fov","focal")`
  may not cover 3D-camera variants, OR `_export_camera_pipeline`'s
  `action.export_fbx(only_selected_nodes=True)` may reject 3D
  cameras at the Flame API level, OR `cam.type == "Camera"` (plain
  str check on the action-schematic side) excludes a different
  type string used by 3D cameras.
  - **First diagnostic step:** probe the offending 3D camera via
    forge-bridge:
    ```python
    import flame
    cam = ...  # locate via flame.batch.nodes scan or parent
    print(repr(cam.type), type(cam).__name__,
          [a for a in ("position","rotation","fov","focal") if hasattr(cam, a)])
    ```

    'Camera 3D'), or one of the four attrs is missing.
  - **Fix scope:** depends on probe result. If it's a type-string

## Closed

- **GAP-04.4-UAT-01 (`OQ-2 fallback unimplemented`):** The Camera-node-scope right-click
  path (Plan 04.4-03 / `_first_camera_in_action_selection`) trusts `cam.parent` to
  return the containing `PyActionNode`. In the actual Flame hook callback context
  (Flame 2026.2.1 arm64, macOS), `cam.parent` returns an object whose chained API
  (likely `.export_fbx` or one of its callees) resolves to `None`, so the call from
  `forge_flame.fbx_io.export_action_cameras_to_fbx` raises
  `'NoneType' object is not callable` and the user gets a Tier-1 "Failed to write
  FBX" dialog. Plan 04.4-03 already documented the fallback (scan
  `flame.batch.nodes` for the Action whose `.nodes` contains `cam_node`) but
  deferred implementation pending UAT. UAT has now confirmed the parent path fails.
  Scope: rewrite `_first_camera_in_action_selection` in
  `flame/camera_match_hook.py` to scan `flame.batch.nodes` (with `cam.parent`
  preserved as the fast path inside try/except) so the export pipeline sees the
  same PyActionNode that the Action-scope right-click path produces.
  - Closed: 2026-04-27 by Plan 04.4-07 / Task 1.
  - **Fix:** flame/camera_match_hook.py:_first_camera_in_action_selection
    rewritten with cam.parent fast path + flame.batch.nodes scan fallback
    (identity comparison `inode is item`). Mirrors _find_action_cameras's
    iteration shape.
  - **Tests:** tests/test_camera_match_hook.py — 5 new tests covering
    fast path, parent-raises fallback, parent-returns-None fallback,
    none-pair-on-empty-selection, perspective-skip-then-real-cam.
  - **Verified:** code-level (pytest); next live Flame-restart UAT will
    reconfirm sub-test (3) of Test 2 passes end-to-end.

- **GAP-04.4-UAT-02 (`forge_bake_camera_name not applied on import`):** The Blender →
  Flame send path (`FORGE_OT_send_to_flame_choose_action.execute` and the existing
  v1.2.0 stamped path both stamp `forge_bake_camera_name` into the v5 payload's
  `custom_properties`. The bridge-side `_forge_send` consumes
  `forge_bake_action_name` to find the target Action but DROPS
  `forge_bake_camera_name` — the imported camera takes whatever name the FBX
  template carries (observed: 'Camera1', not the Blender-side name). End-to-end
  effect: a tester who renames a camera in Blender and sends it to Flame sees
  the Flame-side camera ignore that rename and ship as the FBX default. Scope:
  read `forge_bake_camera_name` in `_forge_send`, then either rename the imported
  camera nodes via `created[i].name.set_value(...)` after `import_fbx_to_action`
  returns, or patch the FBX template's camera name field before write.
  - Closed: 2026-04-27 by Plan 04.4-07 / Tasks 3+4 (closes GAP-04.4-UAT-02).
  - **Fix:** tools/blender/forge_sender/transport.py:_FLAME_SIDE_TEMPLATE
    extended with rename block that reads
    `payload["custom_properties"]["forge_bake_camera_name"]` and calls
    `c.name.set_value(...)` on the first duck-typed camera in `created`
    (after import_fbx_to_action returns, before the success envelope).
    Stereo-rig siblings + RootNode_* + Perspective preserved unchanged.
  - **Tests:** tests/test_forge_sender_transport.py — new
