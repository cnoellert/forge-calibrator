## Summary

This phase delivers four cohesive UX improvements before tester rollout: (1) a FORGE-palette restyle of the multi-camera picker dialog, (2) a second right-click entry point that scopes to camera nodes inside an Action's schematic, (3) removal of the Flame "Import Camera from Blender" menu and extension of the Blender forge_sender addon to support cameras without forge bake metadata (push-only inbound path), and (4) menu reorganization to `FORGE / Camera / {Open Camera Calibrator, Export Camera to Blender}`.

All three blocking probes (P-01, P-02, P-03) were resolved via live forge-bridge probes against Flame 2026.2.1. The single most important finding: **the camera-node right-click entry point (D-03) requires adding a `get_action_custom_ui_actions()` function** to the hook — it cannot reuse `get_batch_custom_ui_actions`. This is a structural addition, not a cosmetic tweak. The second critical finding: **Flame raises `RuntimeError` on Action name collision** when creating via `flame.batch.create_node("Action")` — D-09's assumption of auto-suffix behavior is wrong; the addon must handle this error explicitly.

For menu nesting (D-12): `get_batch_custom_ui_actions` supports single-level grouping via `{"name": "FORGE", "actions": [...]}` but does NOT support two-level nesting (`FORGE / Camera / ...`) in any observed Flame 2026.x tool. A flat workaround is required: two top-level groups, or a single group with descriptive leaf names. The planner must choose between the flat group approach and researching whether `hierarchy: ["FORGE"]` (observed in `get_timeline_custom_ui_actions` only) works in batch menus.

**Primary recommendation:** Ship the camera-node scope as `get_action_custom_ui_actions()` additions, use a single `FORGE` group in `get_batch_custom_ui_actions` with leaf items named "Open Camera Calibrator" and "Export Camera to Blender", and handle name collision in the addon with a user-facing error rather than silent auto-suffix.

## P-01 Answer: Nested Submenus in `get_batch_custom_ui_actions`

### Finding

**Two-level hierarchy (`FORGE / Camera / leaf`) is NOT supported in `get_batch_custom_ui_actions` on Flame 2026.2.1.**

[VERIFIED: live inspection of /opt/Autodesk/shared/python/ on Flame 2026.2.1 host]
[CITED: forum.logik.tv/t/submenus-via-python/2924 — Autodesk's Fred Warren confirmed nested submenus "not possible," FR FI-02123]

Evidence:
- Every installed tool using `get_batch_custom_ui_actions` uses either `{"name": "GroupName", "actions": [...]}` (single-level grouping, no `hierarchy` key) or `{"hierarchy": [], "actions": [...]}` (root-level items).
- The `hierarchy: ["FORGE"]` pattern exists only in `forge_cv_align`'s `get_timeline_custom_ui_actions` — not in any `get_batch_custom_ui_actions` implementation in the installed tools.
- No installed tool uses `{"name": "FORGE", "hierarchy": [], "actions": [], ...}` followed by `{"name": "Camera", "hierarchy": ["FORGE"], "actions": [...]}` in the batch context.
- The Logik forums confirm this limitation was reported as a feature request (FI-02123) with no resolution confirmed for Flame 2026.x.

### Workaround Options for D-12

**Option A (Recommended — Low Risk):** A single `FORGE` group in `get_batch_custom_ui_actions` with leaf items directly under it:

```python
def get_batch_custom_ui_actions():
    return [
        {
            "name": "FORGE",
            "actions": [
                {"name": "Open Camera Calibrator", "isVisible": _scope_batch_clip, "execute": _launch_camera_match},
                {"name": "Export Camera to Blender", "isVisible": _scope_batch_action, "execute": _export_camera_to_blender},
            ],
        }
    ]
```

This produces the right-click menu: `FORGE > Open Camera Calibrator` and `FORGE > Export Camera to Blender`. The "Camera" sub-tier is dropped. D-15 camera-node items land here too if the planner wants them in the Batch menu (they can't appear in the Batch menu anyway — see P-02).

**Option B (Speculative — Higher Risk):** Test `hierarchy: ["FORGE"]` in the batch context at install time. If it renders correctly, produce a two-tier `FORGE / Camera` hierarchy. This is UNVERIFIED for batch menus — the logik forum thread says it does not work, and no installed tool demonstrates it. [ASSUMED: if it works, the dict structure would mirror forge_cv_align's timeline pattern]

**Recommendation:** Use Option A. The `FORGE > leaf` structure is verified working in batch menus via analogous tools (`reveal_path`, `camera_match` itself). Two-level hierarchy is a nice-to-have, not a requirement per D-11/D-12.

## P-02 Answer: Camera-Node Selection Inside Action Schematic

### Finding

**The camera-node right-click requires `get_action_custom_ui_actions()`, NOT `get_batch_custom_ui_actions()`.**

[VERIFIED: live probe + inspection of axis_to_point_locators and encompass_selected_nodes on Flame 2026.2.1]

When a user right-clicks a **camera node inside an Action's schematic**, Flame fires `get_action_custom_ui_actions` and passes the selected PyCoNode objects in `selection`. When a user right-clicks the **Action node in the Batch schematic**, Flame fires `get_batch_custom_ui_actions` and passes the PyActionNode in `selection`.

These are different selection surfaces and different hook functions. The current `camera_match_hook.py` has no `get_action_custom_ui_actions` function — this is a new function that must be added.

### Selection Object Shape (Action Schematic)

Inside `get_action_custom_ui_actions`, the `selection` list contains `flame.PyCoNode` objects. Key properties verified live:

| Property | Type | Example Value | Notes |
|----------|------|---------------|-------|
| `item.type` | `str` (plain Python string) | `"Camera"` | NOT a PyAttribute — no `.get_value()` needed. Compare directly: `item.type == "Camera"` |
| `item.name` | PyAttribute | `"Default"` | Has `.get_value()` / `.set_value()` |
| `item.parent` | `flame.PyActionNode` | `<action1>` | Contains the Action that owns this camera. Has `.name.get_value()`, `.type.get_value()` == `"Action"` |
| `isinstance(item, flame.PyCoNode)` | `bool` | `True` | Works — `flame.PyCoNode` is exposed as a class |

The Perspective filter applies identically: `item.name.get_value() == "Perspective"` (or `item.type == "Camera" and item.name.get_value() != "Perspective"`).

### Scope Helper Pattern

```python
# For get_action_custom_ui_actions — isVisible callback
def _scope_action_camera(selection):
    """True when any selected item is a non-Perspective Camera node inside an Action schematic."""
    for item in selection:
        try:
            if (isinstance(item, flame.PyCoNode)
                    and item.type == "Camera"
                    and item.name.get_value() != "Perspective"):
                return True
        except Exception:
            continue
    return False
```

### Resolving Action from Camera Selection

Because `PyCoNode.parent` returns the containing `PyActionNode` [VERIFIED: live probe]:

```python
# In the camera-node-scope handler
def _first_camera_in_action_selection(selection):
    """Return (action_node, cam_node) from an Action-schematic selection, or (None, None)."""
    for item in selection:
        try:
            if (isinstance(item, flame.PyCoNode)
                    and item.type == "Camera"
                    and item.name.get_value() != "Perspective"):
                return item.parent, item  # parent is the PyActionNode
        except Exception:
            continue
    return None, None
```

This gives the handler the Action and Camera without needing to search `flame.batch.nodes`.

### New Hook Function Shape

```python
def get_action_custom_ui_actions():
    return [
        {
            "hierarchy": [],
            "actions": [
                {
                    "name": "Export Camera to Blender",
                    "isVisible": _scope_action_camera,
                    "execute": _export_camera_from_action_selection,
                },
            ],
        }
    ]
```

The function must be added at the bottom of `camera_match_hook.py` alongside `get_batch_custom_ui_actions`. D-04 says both items remain visible (isVisible naturally separates them by hook type). D-15 says these leaves live under `FORGE / Camera /` — with the two-tier nesting being unverified, the safest shape is a root-level item in the Action schematic's right-click, with a name like "Export Camera to Blender" that's clear by itself.

## P-03 Answer: Programmatic Action Node Creation

### Finding

**`flame.batch.create_node("Action")` works and is the correct API.** [VERIFIED: live probe]

```python
# Creates an Action node in the current batch
new_action = flame.batch.create_node("Action")
# Returns: PyActionNode, auto-named "action4" (sequential suffix)
```

**Critical correction to D-09:** Flame does NOT auto-suffix on name collision when calling `name.set_value()`. It raises `RuntimeError`:

```
RuntimeError: Could not set Batch node name, name is already being used.
```

[VERIFIED: live probe — explicit `set_value("TestAction_forge")` on second node with same name raised this error]

### Correct Create-and-Name Pattern

```python
def _create_action_in_batch(name: str) -> "flame.PyActionNode":
    """Create a new Action node in the current batch with the given name.
    
    Raises:
        RuntimeError: if name is already in use (Flame refuses duplicates).
    """
    import flame
    # 1. Create with auto-name
    node = flame.batch.create_node("Action")
    # 2. Rename — raises RuntimeError on collision
    node.name.set_value(name)
    return node
```

### Addon-Side Error Handling (Required)

The addon MUST catch `RuntimeError` from the bridge response when the Action name is already in use, and surface it via the existing `parse_envelope` → `report({'ERROR'}, ...)` path. The user must either choose a different name or pick the existing Action from the dropdown.

Bridge payload shape for create-and-name:

```python
# In transport.py, new payload template for "create Action" path
_CREATE_ACTION_TEMPLATE = '''\
import flame
new_node = flame.batch.create_node("Action")
new_node.name.set_value({name_repr})
new_node.name.get_value()
'''
```

The result field will be the new Action's name (confirmed via the auto-rename rename which returned the set value correctly). If the name is taken, the bridge response will have `error: "RuntimeError: Could not set Batch node name..."`.

```python
def _pick_camera(cameras, dialog_title):
    """FORGE-styled camera picker. Returns (action, cam, label) or None."""
    if len(cameras) == 1:
        return cameras[0]
    from PySide6.QtWidgets import (
        QDialog, QVBoxLayout, QHBoxLayout,
        QLabel, QListWidget, QPushButton, QFrame,
    )
    from PySide6.QtCore import Qt

    dialog = QDialog()
    dialog.setWindowTitle(f"FORGE — {dialog_title}")  # em dash
    dialog.setMinimumWidth(360)
    dialog.setStyleSheet(_FORGE_SS)  # existing module-level constant

    layout = QVBoxLayout(dialog)
    layout.setContentsMargins(16, 14, 16, 14)
    layout.setSpacing(12)

    header = QLabel(dialog_title)
    header.setStyleSheet("color: #E87E24; font-weight: bold; font-size: 14px;")
    layout.addWidget(header)

    sep = QFrame(); sep.setFrameShape(QFrame.HLine)
    sep.setStyleSheet("color: #3a3f4f;"); layout.addWidget(sep)

    lst = QListWidget()
    lst.setStyleSheet(
        "QListWidget { background: #1e2028; color: #ccc; border: 1px solid #555; "
        "  border-radius: 3px; font-size: 12px; }"
        "QListWidget::item:selected { background: #E87E24; color: #fff; }"
    )
    choices = [c[2] for c in cameras]
    lst.addItems(choices)
    lst.setCurrentRow(0)
    layout.addWidget(lst)

    sep2 = QFrame(); sep2.setFrameShape(QFrame.HLine)
    sep2.setStyleSheet("color: #3a3f4f;"); layout.addWidget(sep2)

    btn_row = QHBoxLayout()
    btn_row.addStretch()
    cancel_btn = QPushButton("Cancel")
    cancel_btn.clicked.connect(dialog.reject)
    btn_row.addWidget(cancel_btn)
    ok_btn = QPushButton("Select")
    ok_btn.setObjectName("primary")
    ok_btn.clicked.connect(dialog.accept)
    ok_btn.setDefault(True)
    btn_row.addWidget(ok_btn)
    layout.addLayout(btn_row)
    lst.itemDoubleClicked.connect(lambda _: dialog.accept())

    if dialog.exec() != QDialog.Accepted:
        return None
    return cameras[lst.currentRow()]
```

**Why QListWidget over QComboBox:** The existing `_FORGE_SS` constant already has full `QListWidget`-compatible styling via its `QComboBox QAbstractItemView` rules. A `QListWidget` shows all options at once (no click to expand), which is better UX for 2-6 cameras. Planner can override to `QComboBox` but `QListWidget` is the recommendation.

**_FORGE_SS factoring:** The stylesheet at `camera_match_hook.py:300-336` is defined inside the body of `_open_camera_match`. It must be hoisted to module-level as `_FORGE_SS` so `_pick_camera` can reference it without being nested inside `_open_camera_match`. This is a required refactor step.

### Pattern 2: Blender Operator with invoke() for Action Picker

The existing `FORGE_OT_send_to_flame` operator uses `execute()` only. The metadata-absent path needs a dialog before execute runs. Blender's mechanism: `invoke()` + `wm.invoke_props_dialog()` or a custom `draw()` method.

```python
# Blender 4.5+ pattern for pre-execute dialog with dynamic EnumProperty
import ast, requests
import bpy
from bpy.props import EnumProperty, StringProperty

class FORGE_OT_send_to_flame_new_action(bpy.types.Operator):
    """Send camera to Flame — choose or create target Action."""
    bl_idname = "forge.send_to_flame_new_action"
    bl_label = "Send to Flame (choose Action)"
    bl_options = {'REGISTER'}

    target_action: EnumProperty(
        name="Target Action",
        description="Existing Flame Action or '-- Create New --'",
        items=lambda self, ctx: self._get_action_items(ctx),
    )
    new_action_name: StringProperty(
        name="New Action Name",
        description="Name for the new Action to create in Flame",
        default="",
    )

    def _get_action_items(self, ctx):
        # Called by Blender when the EnumProperty is evaluated.
        # Returns cached result from invoke() so we don't re-probe on every draw.
        items = getattr(self, "_cached_actions", [])
        result = [(n, n, "") for n in items]
        result.append(("__create_new__", "-- Create New --", "Create a new Action in Flame"))
        return result

    def invoke(self, context, event):
        # Probe Flame for current Actions via bridge
        try:
            resp = requests.post(
                "http://127.0.0.1:9999/exec",
                json={"code": (
                    "[n.name.get_value() for n in flame.batch.nodes "
                    "if hasattr(n.type, 'get_value') and n.type.get_value() == 'Action']"
                )},
                timeout=5.0,
            )
            envelope = resp.json()
            raw = envelope.get("result", "[]")
            self._cached_actions = ast.literal_eval(raw) if raw else []
        except Exception as e:
            self.report({'ERROR'}, f"forge-bridge not reachable: {e}")
            return {'CANCELLED'}
        return context.window_manager.invoke_props_dialog(self, width=360)

    def draw(self, context):
        layout = self.layout
        layout.prop(self, "target_action")
        if self.target_action == "__create_new__":
            layout.prop(self, "new_action_name")

    def execute(self, context):
        # ... build payload and POST to bridge
        pass
```

**Key points:**
- `invoke()` runs the bridge probe once and caches results in `self._cached_actions`
- The `EnumProperty` `items` lambda is called on every draw, but the cached list is static after invoke
- `wm.invoke_props_dialog()` produces a small Blender-native dialog (no custom Qt needed on Blender side)
- The operator's `draw()` method conditionally shows `new_action_name` only when "Create New" is selected

**Alternative:** If a single operator is preferred, add `invoke()` and the new properties to the existing `FORGE_OT_send_to_flame` and gate the dialog path on whether metadata is absent. This keeps one `bl_idname`. The tradeoff is increased complexity in the existing operator's `execute()` path. Planner decides.

```python
_new.name.set_value({name_repr})
_new.name.get_value()
"""
# name_repr = repr("MyAction")  — safe against any embedded quotes/special chars
# Returns: repr of the new name string e.g. "'MyAction'"
# On name collision: error = "RuntimeError: Could not set Batch node name..."
```

### Anti-Patterns to Avoid

- **Don't use `.get_value()` on `PyCoNode.type` in action hook scope helpers.** Inside `get_action_custom_ui_actions`, `item.type` is a plain Python `str` — calling `.get_value()` will raise `AttributeError`. Use `item.type == "Camera"` directly.
- **Don't assume `name.set_value()` returns the new value.** Live probe shows `set_value()` returns `None`. Verify by calling `name.get_value()` after.
- **Don't share `_FORGE_SS` between `_open_camera_match` body and `_pick_camera` while the constant is defined inside the former.** Hoist `_FORGE_SS` to module level (line ~299, just before `class ImageWidget`) as the first task of any QDialog work.
- **Don't add `get_action_custom_ui_actions` items that call handlers that `import flame` at module top level.** Keep lazy imports inside the handler function body — same pattern as existing handlers.

## Common Pitfalls

### Pitfall 1: `PyCoNode.type` is a str in Action context but a PyAttribute in Batch context

**What goes wrong:** Writing `item.type.get_value() == "Camera"` in a `get_action_custom_ui_actions` scope helper. In the batch hook, `item.type` is a PyAttribute with `.get_value()`. In the action hook, `item.type` is a plain Python `str`. Calling `.get_value()` on a `str` raises `AttributeError`, which suppresses the item silently (the `try/except` in scope helpers swallows it and the menu never appears).

**Root cause:** Flame's Python API exposes batch nodes and action nodes via different Python classes with different attribute types.

**How to avoid:** In `get_action_custom_ui_actions` scope helpers, use `item.type == "Camera"` directly (string equality). In `get_batch_custom_ui_actions` scope helpers, use the existing pattern: `t.get_value() if hasattr(t, "get_value") else str(t)`.

**Warning signs:** Menu item is registered but never appears when right-clicking camera node.

### Pitfall 2: Flame raises `RuntimeError` on Action name collision, not auto-suffix

**What goes wrong:** The addon sends `name.set_value("existing_action_name")` inside the bridge payload and assumes Flame auto-renames. Instead, the bridge response contains `error: "RuntimeError: Could not set Batch node name, name is already being used."`.

**Root cause:** D-09 in CONTEXT.md states "Flame's auto-suffix on collision applies" — this is WRONG per live probe. Flame does auto-name on `create_node("Action")` (creates "action4", "action5", etc.) but it does NOT auto-suffix when you then call `name.set_value()` with an already-used name.

**How to avoid:** The addon's "create new Action" flow must: (a) validate the user-provided name is non-empty before POSTing, and (b) in `parse_envelope`, detect the `RuntimeError: Could not set Batch node name` error string and surface it as a clear user message ("An Action named '{name}' already exists. Choose a different name.").

**Warning signs:** Bridge response has `error` field set with "Could not set Batch node name" substring; addon silently fails or shows a confusing generic error.

### Pitfall 3: `get_action_custom_ui_actions` is a NEW function — forgetting it means no camera-node right-click

**What goes wrong:** The planner scopes the camera-node entry point as changes only to `get_batch_custom_ui_actions`. The camera-node right-click never fires because `get_batch_custom_ui_actions` only fires on Batch-schematic right-clicks, not Action-schematic right-clicks.

**Root cause:** Two different right-click surfaces, two different hook functions. The distinction is documented in the Flame Python hook API but easy to miss.

**How to avoid:** The plan must include a task that adds `get_action_custom_ui_actions()` to `camera_match_hook.py`. Verifying it works requires a Flame restart + open the Action's schematic + right-click a camera node (not the Action node in Batch).

**Warning signs:** After install, right-clicking an Action node in Batch shows the new menu; right-clicking a camera node inside the Action schematic shows no forge menu.

### Pitfall 4: Menu changes require a full Flame restart

**What goes wrong:** Using the gc/exec live-reload workaround to test menu structure changes. This reloads module globals but NOT Flame's internal menu dispatch table (`get_batch_custom_ui_actions` is called once at startup and cached).

**Root cause:** Memory note `flame_module_reload.md` — confirmed on Flame 2026.2.1.

**How to avoid:** Development loop must be: edit → `install.sh` → Flame restart. For handler logic (not menu structure), the bridge can invoke handlers directly: `sys.modules['camera_match']._fn_name(fake_selection)`.

**Warning signs:** Code changes don't appear in menu after module reload but do appear after restart.

### Pitfall 5: `_FORGE_SS` currently defined inside `_open_camera_match` body

**What goes wrong:** `_pick_camera` is at module level (line 1863) but `_FORGE_SS` is defined inside `_open_camera_match` (inner function scope). If the styled picker tries to reference `_FORGE_SS`, it gets `NameError`.

**Root cause:** The palette was added when only the line-drawing window needed it; it was scoped inside that function.

**How to avoid:** First task for any FORGE-styled dialog work: hoist `_FORGE_SS` (currently at line ~300 inside `_open_camera_match`) to module level (between the `_SANITIZE_NAME_RE` constant block and the `_open_camera_match` function definition).

**Warning signs:** `NameError: name '_FORGE_SS' is not defined` when the picker is invoked.

```python
def list_batch_actions(timeout: float = 5.0) -> list:
    """Return list of Action names in current Flame batch. Raises on bridge error."""
    import requests, ast
    resp = requests.post(
        "http://127.0.0.1:9999/exec",
        json={"code": (
            "import flame\n"
            "[n.name.get_value() for n in flame.batch.nodes "
            "if hasattr(n.type, 'get_value') and n.type.get_value() == 'Action']"
        )},
        timeout=timeout,
    )
    resp.raise_for_status()
    envelope = resp.json()
    if envelope.get("error"):
        raise RuntimeError(envelope["error"])
    return ast.literal_eval(envelope.get("result") or "[]")
```

## R-06: forge-bridge Payload Discipline (Summary)

**Bridge contract (verified against memory/flame_bridge_repl_contract.md):**
- POST `{"code": "<python>"}` to `http://127.0.0.1:9999/exec`
- Response: `{"result": repr_of_last_expr, "stdout": ..., "stderr": ..., "error": ..., "traceback": ...}`
- `result` is `repr()` of the last expression — parse with `ast.literal_eval()`
- NO `_result` magic variable — end code with bare expression
- Persistent namespace across calls (monkey-patch pattern works)

**Safe payload shapes for new operations:**

```python
# List Actions — safe, no user input
CODE_LIST = (
    "import flame\n"
    "[n.name.get_value() for n in flame.batch.nodes "
    "if hasattr(n.type, 'get_value') and n.type.get_value() == 'Action']"
)

# Create Action — user input embedded via repr()
def make_create_code(name: str) -> str:
    return (
        f"import flame\n"
        f"_n = flame.batch.create_node('Action')\n"
        f"_n.name.set_value({repr(name)})\n"  # repr() escapes any quotes/special chars
        f"_n.name.get_value()"
    )
```

**Probing discipline (memory/flame_bridge_probing.md):**
- One probe per request; never bundle multiple risky calls
- Ping first (`print(2+2)`) if Flame hasn't been touched recently
- Avoid `help()` — crashes bridge; prefer `repr(x.__doc__)`
- If two sequential probes crash Flame, stop probing and pivot
