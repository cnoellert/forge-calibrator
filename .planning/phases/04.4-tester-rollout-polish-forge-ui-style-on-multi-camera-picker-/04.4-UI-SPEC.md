# Phase 04.4 — UI Design Contract

> Visual and interaction contract for Phase 04.4 tester-rollout polish. This phase spans two
> distinct UI surfaces:
>
> **Surface A — Flame / PySide6:** Multi-camera picker `QDialog` restyle, the new Batch
> right-click menu (`get_batch_custom_ui_actions`), and the new Action-schematic right-click
> menu (`get_action_custom_ui_actions`). All governed by the locked FORGE palette from
> `memory/forge_ui_style.md` and the `forge_cv_align` reference implementation.
>
> **Surface B — Blender / bpy:** Extension of the `forge_sender` addon to handle cameras
> without forge bake metadata. Uses `bpy.types.Operator` with `invoke_props_dialog` and
> native Blender layout primitives. Inherits the conventions locked in `02-UI-SPEC.md`.
>
> Web-UI template dimensions (responsive breakpoints, design tokens, CSS) are not applicable.
> Where a standard template section is not applicable, it is marked **N/A** with rationale.
>
> Authority: CONTEXT.md D-01..D-15, RESEARCH.md P-01..P-03 and R-06..R-08 are the primary
> inputs. RESEARCH.md corrections to CONTEXT.md (flat menu hierarchy, RuntimeError on name
> collision) are the binding authoritative source where they conflict.

## Spacing Scale

**Surface A (Flame / PySide6):** The FORGE palette uses a pixel-based system, not a CSS token
system. The values below are taken directly from `memory/forge_ui_style.md` and the
`forge_cv_align` reference. They are presented as named tokens so the executor can reference
them symbolically.

| Token | Value | Usage |
|-------|-------|-------|
| dialog-margin | 16px top/bottom, 14px left/right | QDialog `setContentsMargins(16, 14, 16, 14)` — matches FORGE reference |
| section-gap | 12px | `setSpacing(12)` on QVBoxLayout — between logical rows |
| inline-gap | 8px | Within a button row, between widgets |
| separator | QFrame HLine color `#3a3f4f` | Visual divider between logical groups |
| min-dialog-width | 360px | `setMinimumWidth(360)` on picker QDialog — matches RESEARCH Pattern 1 |
| button-padding | `6px 12px` (secondary), `8px 12px` (primary) | `_FORGE_SS` QPushButton rules |

**Exceptions:** `14px` horizontal dialog margin and `6px` button vertical padding are
non-4px-multiple values inherited from the locked FORGE palette (`memory/forge_ui_style.md`,
sourced from the `forge_cv_align` reference at
`/opt/Autodesk/shared/python/forge_cv_align/forge_cv_align.py:227-371`). These values are
part of the established FORGE design system across all FORGE tools and are approved exceptions
to the 4px grid rule for this project.

**Surface B (Blender / bpy):** N/A — native Blender widgets; spacing controlled by
`layout.row()` / `layout.separator()` per active theme. Matches 02-UI-SPEC.md.

## Surface A: Flame / PySide6 Contracts

### A-1: Multi-Camera Picker QDialog

**Purpose:** Replace the bare `QInputDialog.getItem` at `camera_match_hook.py:1863` with a
FORGE-styled `QDialog`. Called when `_export_camera_to_blender` detects 2+ non-Perspective
cameras in the target Action.

**Prerequisite refactor:** Hoist `_FORGE_SS` from inside `_open_camera_match` body to
module-level constant (between the `_SANITIZE_NAME_RE` block and the `_open_camera_match`
function). This unblocks `_pick_camera` referencing the constant without nesting.

**Function signature (unchanged):**
```python
def _pick_camera(
    cameras: list[tuple],  # List[Tuple[action_node, cam_node, label_str]]
    dialog_title: str = "Select Camera",
) -> tuple | None:          # Returns (action, cam, label) or None on Don't Export
```

**Early-exit rule:** If `len(cameras) == 1`, return `cameras[0]` immediately — no dialog shown.
If `len(cameras) == 0`, return `None` immediately. Dialog is only shown for 2+ cameras.

**Widget composition:**

| Layer | Widget | Style |
|-------|--------|-------|
| Container | `QDialog` | `setStyleSheet(_FORGE_SS)`; `setMinimumWidth(360)`; `setWindowTitle("FORGE — Select Camera")` |
| Layout | `QVBoxLayout` | `setContentsMargins(16, 14, 16, 14)`; `setSpacing(12)` |
| Header | `QLabel(dialog_title)` | `color: #E87E24; font-weight: bold; font-size: 14px;` |
| Separator 1 | `QFrame(frameShape=HLine)` | `setObjectName("sep")` — `_FORGE_SS` `QFrame#sep` rule applies |
| List | `QListWidget` | bg `#1e2028`, text `#ccc`, border `#555`, border-radius 3px, font-size 12px; selected item bg `#E87E24`, text `#fff`; `setCurrentRow(0)` |
| Separator 2 | `QFrame(frameShape=HLine)` | Same as Separator 1 |
| Button row | `QHBoxLayout` | `addStretch()` on left; "Don't Export" left of "Select Camera" |
| Don't Export button | `QPushButton("Don't Export")` | Secondary style from `_FORGE_SS` (default `QPushButton` rule) |
| Select Camera button | `QPushButton("Select Camera")` | `setObjectName("primary")`; `setDefault(True)` |

**Choice: QListWidget over QComboBox.** All options visible at once; no click-to-expand needed
for the expected 2-6 camera count. `_FORGE_SS` already has full list-widget styling via its
`QComboBox QAbstractItemView` ancestry.

**List item label format:** Use the existing `label_str` from the `cameras` tuple. Format is
already `"{action_name} — {cam_name}"` (em-dash separator). Do not reformat.

**Keyboard semantics:**

| Key | Behavior |
|-----|----------|
| Enter / Return | Accepts dialog (same as clicking Select Camera) |
| Escape | Dismisses dialog without exporting (same as clicking Don't Export) |
| Arrow Up / Down | Moves QListWidget selection |
| Double-click item | Accepts dialog immediately |

Wire: `lst.itemDoubleClicked.connect(lambda _: dialog.accept())`; `ok_btn.clicked.connect(dialog.accept)`; `cancel_btn.clicked.connect(dialog.reject)`.

**Return contract:**
- `dialog.exec() == QDialog.Accepted` → return `cameras[lst.currentRow()]`
- `dialog.exec() != QDialog.Accepted` ("Don't Export" or Escape) → return `None`
- Caller treats `None` as user abort — handler exits without modifying the Action.

**Empty state:** Not reachable via picker (0-camera case returns `None` before dialog opens).

### A-2: Batch Right-Click Menu (`get_batch_custom_ui_actions`)

**Flat FORGE group — two-level nesting (`FORGE / Camera / leaf`) is NOT supported on Flame
2026.2.1.** (RESEARCH P-01: verified via code inspection + logik.tv FR FI-02123.) The "Camera"
sub-tier from CONTEXT D-12 is dropped. Menu is flat under the `FORGE` group name.

**Menu structure:**

```python
def get_batch_custom_ui_actions():
    return [
        {
            "name": "FORGE",
            "actions": [
                {
                    "name": "Open Camera Calibrator",
                    "isVisible": _scope_batch_clip,
                    "execute": _launch_camera_match,
                },
                {
                    "name": "Export Camera to Blender",
                    "isVisible": _scope_batch_action,
                    "execute": _export_camera_to_blender,
                },
            ],
        }
    ]
```

**User-visible result:** Right-clicking a Clip node in Batch shows:
```
FORGE
  └─ Open Camera Calibrator
```

Right-clicking an Action node in Batch shows:
```
FORGE
  └─ Export Camera to Blender
```

**Removed item:** "Import Camera from Blender" is deleted (CONTEXT D-06). No backwards-compat
shim. Hard cut on install.

**Scope helpers (unchanged behavior, new label context):**
- `_scope_batch_clip`: existing helper — returns `True` when selection contains a Clip-type
  node. Governs "Open Camera Calibrator" visibility.
- `_scope_batch_action`: existing helper at `camera_match_hook.py:1806` — `type.get_value() ==
  "Action"`. Governs "Export Camera to Blender" visibility.

### A-3: Action-Schematic Right-Click Menu (`get_action_custom_ui_actions`)

**This is a new function that does NOT exist in the current hook.** It is required because
right-clicking a camera node inside an Action's schematic fires `get_action_custom_ui_actions`,
not `get_batch_custom_ui_actions`. The two hook functions serve different selection surfaces and
cannot be unified. (RESEARCH P-02, verified live.)

**Menu structure:**

```python
def get_action_custom_ui_actions():
    return [
        {
            "hierarchy": [],   # root level — no subgroup; two-level nesting unverified
            "actions": [
                {
                    "name": "Export Camera to Blender",
                    "isVisible": _scope_action_camera,
                    "execute": _export_camera_from_action_selection,
                },
            ],
        }
    ]
```

**User-visible result:** Right-clicking a non-Perspective camera node inside an Action schematic
shows:
```
Export Camera to Blender
```
(No FORGE group header — `hierarchy: []` produces a root-level item per verified installed-tool
pattern.)

**Scope helper — NEW:**

| Function | Signature | Logic |
|----------|-----------|-------|
| `_scope_action_camera` | `(selection) -> bool` | Returns `True` when any item in `selection` satisfies: `isinstance(item, flame.PyCoNode)` AND `item.type == "Camera"` (plain string — NOT `.get_value()`) AND `item.name.get_value() != "Perspective"` |

**Critical: `item.type` in `get_action_custom_ui_actions` is a plain Python `str`, not a
PyAttribute.** Calling `.get_value()` raises `AttributeError` and silently suppresses the menu
item. Use `item.type == "Camera"` (direct string equality). See RESEARCH Pitfall 1.

**Resolver helper — NEW:**

| Function | Signature | Returns |
|----------|-----------|---------|
| `_first_camera_in_action_selection` | `(selection) -> tuple[PyActionNode, PyCoNode] \| tuple[None, None]` | First non-Perspective PyCoNode Camera from selection; resolves Action via `cam.parent` (verified: returns `PyActionNode`) |

**Handler — NEW:**

| Function | Behavior |
|----------|----------|
| `_export_camera_from_action_selection` | Calls `_first_camera_in_action_selection(selection)`; if `(None, None)` returned, surfaces error and exits; otherwise calls into the same export pipeline as `_export_camera_to_blender` with the resolved `(action_node, cam_node)`, skipping `_pick_camera` entirely |

**Picker bypass:** When the selection is a specific camera node, the picker is not shown. The
resolved camera is used directly. This is the key UX improvement over the Action-scope path.

### B-1: Panel Extension — New "Send to Flame (choose Action)" Button

**When shown:** When the active camera is a camera object (`context.active_object.type ==
'CAMERA'`) but does NOT have forge bake metadata (`forge_bake_action_name` absent OR
`forge_bake_camera_name` absent OR `forge_bake_source != "flame"`).

**Existing panel row order is unchanged for the stamped-camera (happy) path.** Only the
disabled-state layout gains a second button row.

**New row order for no-metadata state:**

| # | Element | Blender call | Icon | Enabled |
|---|---------|--------------|------|---------|
| 1 | Warning row | `row = layout.row(); row.alert = True; row.label(text="No Flame metadata — choose a target Action", icon='ERROR')` | `ERROR` | n/a (label) |
| 2 | Separator | `layout.separator()` | — | n/a |
| 3 | Send to Flame (disabled) | `layout.operator("forge.send_to_flame", icon='EXPORT')` | `EXPORT` | disabled (poll returns False — no metadata) |

**Both buttons are visible simultaneously in the no-metadata state.** The top button is
disabled (existing path gate), the bottom button is enabled (new path). This avoids hiding the

**Operator `poll()` for new operator (`forge.send_to_flame_choose_action`):**
1. `context.active_object is not None`
2. `context.active_object.type == 'CAMERA'`

No metadata check — the whole point is to handle cameras without metadata.

### B-2: "Send to Flame (choose Action)" Operator Dialog

**Operator identity:**

| Attribute | Value |
|-----------|-------|
| `bl_idname` | `forge.send_to_flame_choose_action` |
| `bl_label` | `Send to Flame (choose Action)` |
| `bl_options` | `{'REGISTER'}` |

**Dialog mechanism:** `wm.invoke_props_dialog(self, width=360)` — Blender's native modal dialog.
No custom Qt or Tkinter.

**Dialog layout (via `draw()` method):**

| # | Element | Call | Condition |
|---|---------|------|-----------|
| 1 | Target Action dropdown | `layout.prop(self, "target_action")` | Always shown |
| 2 | New Action name field | `layout.prop(self, "new_action_name")` | Only when `self.target_action == "__create_new__"` |

**Properties:**

| Property | Type | Items / Default | Description |
|----------|------|-----------------|-------------|
| `target_action` | `EnumProperty` | Dynamically populated in `invoke()` from bridge; always appends `("__create_new__", "-- Create New --", "Create a new Action in Flame")` | Dropdown of existing Actions in current Flame Batch |
| `new_action_name` | `StringProperty` | default `""` | Name for the new Action; only rendered when `__create_new__` selected |

**`invoke()` behavior:**
1. POST to `http://127.0.0.1:9999/exec` with `CODE_LIST_ACTIONS` payload.
2. Parse response with `ast.literal_eval(envelope["result"])` to get `List[str]`.
3. Cache result in `self._cached_actions` (list of existing Action names).
4. If bridge is unreachable → `self.report({'ERROR'}, ...)` + `return {'CANCELLED'}` (no
   dialog shown).
5. If bridge returns an error envelope → `self.report({'ERROR'}, ...)` + `return {'CANCELLED'}`.
6. If bridge returns an empty list → proceed to dialog; dropdown will show only
   `"-- Create New --"`. Do NOT suppress the dialog in this case.
7. Return `context.window_manager.invoke_props_dialog(self, width=360)`.

**`execute()` behavior — pick existing Action path:**
1. `self.target_action` is an existing Action name (not `"__create_new__"`).
2. Build existing-path payload targeting the named Action.
3. POST and handle envelope via existing `parse_envelope` → success / Remote Tier error.

**`execute()` behavior — create new Action path:**
1. `self.target_action == "__create_new__"`.
2. Validate `self.new_action_name` is non-empty. If empty → `report({'ERROR'}, "New Action name
   cannot be empty")` + `return {'CANCELLED'}`.
3. POST `make_create_code(self.new_action_name)` to bridge.
4. If envelope contains `error` with substring `"Could not set Batch node name"` → surface
   name-collision error (see Copywriting B below).
5. On success → continue with camera push into the newly created Action.

## Navigation Map

This phase adds and reorganizes two distinct menu surfaces. Both are right-click context menus
in Flame — there is no persistent navigation chrome.

### Batch Schematic Right-Click

```
[right-click Clip node]
  FORGE
    Open Camera Calibrator      → _launch_camera_match(selection)

[right-click Action node]
  FORGE
    Export Camera to Blender    → _export_camera_to_blender(selection)
                                  ├─ _first_action_in_selection(selection)
                                  ├─ _find_action_cameras(action)
                                  ├─ _pick_camera(cameras)    ← RESTYLED
                                  └─ [export pipeline]
```

### Action Schematic Right-Click (NEW)

```
[right-click Camera node inside Action schematic]
  Export Camera to Blender      → _export_camera_from_action_selection(selection)
                                  ├─ _first_camera_in_action_selection(selection)
                                  │    └─ (action_node, cam_node) via cam.parent
                                  └─ [export pipeline — NO picker]
```

**Picker bypass condition:** The Action-schematic path always bypasses `_pick_camera` because
the camera is resolved directly from the selection object. The user has already indicated which
camera they want by right-clicking it.

### Blender N-Panel (Forge tab)

```
[Forge > Send to Flame panel]

Stamped-camera state (forge_bake_action_name present):
  Target Action: {action_name}
  Target Camera: {camera_name}
  [separator]
  [Send to Flame]                    → forge.send_to_flame (existing operator)

No-metadata state (metadata absent):
  [!] No Flame metadata — choose a target Action
  [separator]
  [Send to Flame] (disabled)         → forge.send_to_flame (poll() returns False)
  [Send to Flame (choose Action)]    → forge.send_to_flame_choose_action (new operator)
                                          └─ invoke_props_dialog:
                                               Target Action: [dropdown / Create New]
                                               New Action Name: [text field, conditional]
```
