# Phase 04.4: Tester-rollout polish — Context

**Gathered:** 2026-04-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Pre-tester UX polish on the Flame Batch right-click menu surface, plus a coupled Blender-addon scope expansion. Four cohesive items:

1. Apply the FORGE UI palette (`memory/forge_ui_style.md`) to the multi-camera picker that today is a bare `QInputDialog.getItem` (`flame/camera_match_hook.py:1863`).
2. Add a second right-click entry point for export/import that scopes to the camera node *inside* an Action, bypassing the picker when the selection is unambiguous.
3. Drop Flame's "Import Camera from Blender" menu item entirely. Extend the Blender-side `forge_sender` addon so cameras without `forge_bake_action_name` metadata can be pushed into a Flame Action of the user's choice (existing Action picked from a dropdown, or a freshly created one).
4. Reorganize the Batch menu into `FORGE / Camera / {Open Camera Calibrator, Export Camera to Blender}` — `FORGE` is the future top-level for all FORGE tools.

This is polish + a clean menu surface. New capabilities (multi-cam round-trip, lights, viewport color matching, etc.) belong in v2 — out of scope.

</domain>

<decisions>
## Implementation Decisions

### Multi-camera picker (folds 999.1)

- **D-01:** Palette-only restyle. Replace the bare `QInputDialog.getItem` at `flame/camera_match_hook.py:1863` with a custom `QDialog` styled per `memory/forge_ui_style.md` (background `#282c34`, accent `#E87E24`, em-dash window title `FORGE — <title>`). Same data shape: takes the existing `cameras: List[Tuple[action, cam, label]]` and returns `(action, cam, label) | None`.
- **D-02:** 999.1 polish items deferred to v2: thumbnail previews, sort-order choice, active-camera indicator, cross-Action name disambiguation, last-picked memory. None of these are required for tester rollout. The phase 999.1 backlog entry should be marked partially-folded with the styling bullet closed.

### Right-click on camera nodes inside Action

- **D-03:** Two entry points, one handler. The existing Action-scope menu items keep their current behavior (selection has no implicit camera → picker shown when ≥2 cameras). New camera-node-scope menu items resolve the camera directly from selection and skip the picker.
- **D-04:** Both menu items remain visible. No "hide one when the other applies" logic — `isVisible` callbacks naturally separate them by selection type.
- **D-05:** Hook handlers (`_export_camera_to_blender`, the export side of `_import_camera_from_blender` is removed per D-08) refactor to accept either `(action_node, cam_node)` resolved from a camera-node selection, or fall through to `_first_action_in_selection` + `_pick_camera` when the selection is an Action. Single resolution helper.

### Bidirectional Blender direction pivot

- **D-06:** REMOVE the Flame-side "Import Camera from Blender" menu item and the `_import_camera_from_blender` handler. Delete the `QFileDialog`-based import path entirely. The Blender-side push (forge_sender addon) becomes the sole inbound path.
- **D-07:** EXTEND the `forge_sender` addon (`tools/blender/forge_sender/`) to accept cameras without forge bake metadata. On send:
  - If `forge_bake_action_name` is present on the active camera → existing path (target the named Action).
  - If absent → show a dialog offering: (a) dropdown of existing Actions in the currently-loaded Flame Batch, queried via forge-bridge, (b) a "Create new Action" option with a text field for the new Action's name.
- **D-08:** The "list Actions in current Batch" lookup runs via forge-bridge `/exec` payload that iterates `flame.batch.nodes` and filters by `type == "Action"` (per existing `_scope_batch_action` pattern at `flame/camera_match_hook.py:1806`). Scope is the **currently-loaded Batch only** — not Actions across other Desktops or Reels.
- **D-09:** When the user picks "Create new Action," the addon's bridge call creates the Action inside `flame.batch` (the currently-open Batch) with the user-provided name and pushes the camera into it. If an Action with that name already exists, Flame's auto-suffix on collision applies (consistent with existing import behavior).
- **D-10:** Error UX matches the existing `forge_sender` patterns (Tier-1 popups via Blender `report({'ERROR'}, …)`). New failure modes to surface:
  - forge-bridge unreachable when listing Actions → existing "bridge not reachable" path.
  - No Actions in the current Batch and user did not pick "Create new" → addon prompts again or surfaces a clear empty-state message.
  - Action creation failed in Flame → bridge response surfaces the traceback.

### Menu reorganization

- **D-11:** Top-level menu becomes `FORGE`. Intended as the future home for *all* FORGE tools (forge_cv_align future home, future tools land here too). This phase only registers the camera-calibrator items under it.
- **D-12:** Submenu structure: `FORGE / Camera / {Open Camera Calibrator, Export Camera to Blender}`. Two leaf actions only — "Import Camera from Blender" is removed per D-06.
- **D-13:** User-facing label rename only. Internal names stay unchanged: `camera_match_hook.py`, `forge_flame/`, install path `/opt/Autodesk/shared/python/camera_match/`. The label "Camera Calibrator" replaces "Camera Match" in dialog titles, window titles, and docs.
- **D-14:** Hard cut on the old "Camera Match" submenu — no backwards-compat shim. Justification: no testers have used the tool yet; this rollout is the introduction. Old submenu disappears on next install.
- **D-15:** Camera-node-scope menu items (D-03) live under the same `FORGE / Camera /` hierarchy. Whether they're additional leaf items (e.g. "Export Selected Camera to Blender") or the same leaves with broader scope is a planner decision based on Flame's submenu API behavior.

### Probes the planner needs

- **P-01:** Does Flame 2026.2.1's `get_batch_custom_ui_actions()` support nested submenus (e.g. `[{name: "FORGE", actions: [{name: "Camera", actions: [...]}]}]`)? If not, planner needs a flat workaround (e.g. `FORGE — Camera` flat group as a single label).
- **P-02:** Does `isVisible` fire on a camera-node selection inside an Action's schematic? If not, the camera-node-scope entry point may need to attach to a different selection target (e.g. the Action containing the selected camera, with selection state passed through). Existing `_scope_batch_action` pattern only matches `type == "Action"` — a parallel `_scope_action_camera` helper is the expected shape.
- **P-03:** What's the Flame API to create a new Action node inside `flame.batch` programmatically (used by the addon's "Create new Action" path)? Existing code only reads/iterates Actions; new-Action creation is unverified.

### Claude's Discretion

- Exact widget composition of the styled `QDialog` picker (single `QListWidget` vs `QComboBox` + button row). Constrained by `memory/forge_ui_style.md` palette and FORGE em-dash title — implementation shape is open.
- Whether the "list Actions in Batch" query runs as inline Python via forge-bridge `/exec` or as a structured RPC method. Both are equivalent given forge-bridge's REPL contract; planner picks the lower-friction one.
- Naming of the camera-node-scope menu leaves (e.g. "Export Camera to Blender" vs "Export Selected Camera to Blender"). Planner picks based on whether nested submenus collapse or not.

### Folded Todos

- **999.1 (Improve multi-camera picker UX)** — partially folded. Bullet 0 (palette restyle, implicit) closes. Bullets 1-5 (thumbnails, sort order, active-cam indicator, name disambiguation, last-picked memory) stay deferred to v2. Update the 999.1 backlog entry to reflect partial closure.

</decisions>

## Existing Code Insights

### Reusable Assets

- **FORGE palette in `camera_match_hook.py:300`** — already in use for the line-drawing window. Restyling the picker is a matter of factoring this palette into a small helper (or copy-paste plus a styled `QDialog` subclass) rather than inventing colors.
- **`_scope_batch_action` (`:1806`) + `_first_action_in_selection` (`:1824`)** — selection-type-matching templates. The new camera-node scope helper follows the same `type.get_value()` duck-typing pattern.
- **`_find_action_cameras` (used at `:2141`, `:2408`)** — already filters out Perspective per `memory/flame_perspective_camera.md`. Camera-node-scope path can reuse it to validate the selected camera isn't Perspective before bypassing the picker.
- **`forge_sender.transport.send` (`tools/blender/forge_sender/transport.py:54`)** — POSTs to `http://127.0.0.1:9999/exec`. The new "list Actions" / "create Action" payloads ride on the same transport.
- **`forge_sender.preflight`** — already runs the bridge-reachable + metadata checks. New metadata-absent branch lands inside the existing preflight result path.

### Established Patterns

- **Bridge calls return `repr(last_expression)`** — see `memory/flame_bridge_repl_contract.md`. Action-list query returns a Python list-of-strings via `repr()`; addon parses with `ast.literal_eval`.
- **Tier-1 / Tier-2 / Tier-3 error popups** — addon already classifies (see Phase 02 SUMMARY). New "no Actions in Batch" / "Action creation failed" failures slot into the existing tier model.
- **Menu changes need a Flame restart** — D-13 keeps internal names unchanged so `install.sh` re-deploys land cleanly without import-path churn.
- **Hard-cut menu replacement on install** — D-14 mirrors how prior menu changes shipped (no transition shim).

### Integration Points

- **`get_batch_custom_ui_actions()` (`flame/camera_match_hook.py:2498`)** — the single menu-registration return. New `FORGE / Camera / X` structure replaces the existing `Camera Match / X` group in one edit.
- **`forge_sender.__init__.py` Operator class** — new "Send to new Action" branch lands inside the existing Operator's `execute` (no new operator needed unless the dialog flow requires modal state).
- **forge-bridge `/exec` payload** — gains two new payload shapes: (a) `[a.name.get_value() for a in flame.batch.nodes if a.type.get_value() == "Action"]`, (b) the Action-creation Python invoked after the user picks "Create new" + name.
- **`docs/seamless-bridge.md` + README** — both updated to reflect the new menu hierarchy + the dropped "Import Camera from Blender" path.
