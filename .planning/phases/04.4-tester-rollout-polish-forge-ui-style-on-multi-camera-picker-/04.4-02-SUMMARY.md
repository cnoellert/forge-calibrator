## Decisions Made

- **QListWidget over QComboBox** for `_pick_camera` per RESEARCH Pattern 1 — all 2-6 camera options visible at once is better UX than a click-to-expand combo. The existing `_FORGE_SS` stylesheet already covers `QListWidget`-compatible styling via its `QComboBox QAbstractItemView` rules; the inline `setStyleSheet` on `lst` adds the selection-color styling specific to this dialog.
- **Label format `f"{action} > {cam}"`** for the Camera-scope path. The plan body specified em-dash `f"{action} — {cam}"`, but `_find_action_cameras` (line 1812) actually uses `>`. Followed the live code's format so downstream filename construction (`_safe_filename(label)`) and the info-dialog text in `_export_camera_pipeline` produce identical output regardless of entry point. Tracked as a Rule 1 deviation below.
- **PATTERNS Shape A (extract shared body)** over Shape B (branching head). Shape A keeps the two entry points' selection-resolution flows independently testable and audit-able; the explicit pipeline boundary makes the D-05 contract obvious in the diff.
- **Wave 0 test-infra fix** is a Plan 04.4-02-scope edit. The lazy `import flame` inside the helper is required (top-level `import flame` would crash Flame at hook load), so the helper at test runtime needs a `flame` in `sys.modules`. Plan 04.4-01's test removed it for "pollution" prevention; the prevention is unnecessary for `flame` (not a real installed package on dev machines). Tracked as a Rule 3 deviation below.
- **Did NOT restyle the Apply Camera flow's QInputDialog** at line ~1502 inside `_open_camera_match`. That picker is a separate UX surface tied to the calibration window's Apply button, not the multi-camera picker in scope here. The prompt's scope boundary explicitly forbids out-of-scope changes; restyling that picker is a candidate for a future polish phase.
- **Did NOT touch `get_batch_custom_ui_actions`.** Plan 04.4-03 owns the menu surface; this plan only lands the symbols Wave 2's new menu entries will call into.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Plan body specified em-dash `—` separator for the Camera-scope label format; live code uses `>`**

- **Found during:** Task 3 (`_export_camera_from_action_selection` label construction)
- **Issue:** Plan 04.4-02-PLAN.md line 478 says `label = f"{_val(action_node.name)} — {_val(cam_node.name)}"` (em-dash separator). The actual `_find_action_cameras` at line 1812 uses `f"{action_name} > {cam_name}"` (`>` separator). The plan's stated rationale ("the same shape `_find_action_cameras` would emit") was correct in spirit; the literal character was wrong. Following the plan literally would have produced a different label between the Action-scope and Camera-scope paths, breaking the downstream invariant.
- **Fix:** Used `f"{_val(action_node.name)} > {_val(cam_node.name)}"` to match the live `_find_action_cameras` output. Added an inline comment at the call site referencing line 1812 of `_find_action_cameras` so the rationale is grep-able.
- **Files modified:** `flame/camera_match_hook.py` (around line 2287, inside `_export_camera_from_action_selection`)
- **Verification:** Source-grep — both labels (Action-scope from `_find_action_cameras` and Camera-scope from the new handler) use `>` as the separator. The pipeline body's info-dialog text (`f"Exported '{label}'"`) is now format-identical between entry points.
- **Committed in:** `da79709`

**2. [Rule 3 - Blocking] Wave 0 test-harness lazy-import contract was unsatisfiable as written**

- **Found during:** Task 2 (running `pytest tests/test_camera_match_hook.py -v` after adding `_scope_action_camera`)
- **Issue:** The Wave 0 test (Plan 04.4-01, commit `e5cf2b2`) installs a fake `flame` module with `_PyCoNode` so that `isinstance(item, flame.PyCoNode)` works inside the helper. After importing `camera_match_hook`, the test then deletes the fake `flame` from `sys.modules` "to prevent pollution." But the helper uses lazy `import flame` (RESEARCH-mandated pattern; module-level `import flame` would crash Flame at startup). At test runtime, that lazy import fails (no `flame` in `sys.modules`), the helper's defensive `try/except Exception` swallows the `ModuleNotFoundError`, and every `_scope_action_camera_*` test fails with `False is not True`. The Plan 04.4-01 SUMMARY explicitly predicted these tests would flip from SKIP to PASS — but they would have stayed RED.
- **Fix:** Added `_KEEP_INSTALLED = {"flame"}` to `tests/test_camera_match_hook.py` and skipped any module name in that set during the post-load `sys.modules` restore loop. `flame` is not a real importable package on dev machines, so leaving the fake in `sys.modules` cannot pollute any other test module. Comment block added above the loop documenting why the exception exists.
- **Files modified:** `tests/test_camera_match_hook.py` (lines 98-114 region — the restore loop)
- **Verification:** `pytest tests/test_camera_match_hook.py -v` — all 5 `_scope_action_camera_*` tests PASS. Full suite (`pytest tests/ -q`) — 406 passed / 10 skipped (no regression in any other test module).
- **Committed in:** `04f4e08`
