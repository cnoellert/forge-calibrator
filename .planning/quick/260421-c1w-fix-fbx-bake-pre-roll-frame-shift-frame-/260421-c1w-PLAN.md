<objective>
Drop the implicit PRE-ROLL keyframe that `action.export_fbx(bake_animation=True)` emits at FBX KTime 0, and correct the +1 shift in the real per-frame samples, by threading a symmetric `frame_start` clip (INCLUSIVE lower bound) through the same wiring that quick tasks 260420-uzv and 260421-bhg built for `frame_offset` / `frame_end`.

Purpose: UAT on the 260421-bhg deploy (which correctly dropped the errant 1101 trailing key) surfaced a deeper bake-shape issue that was previously hidden BEHIND the trailing-frame bug. The actual Flame bake shape is:

- KTime 0 = implicit PRE-ROLL (the camera's initial state — Flame holds the value before the first "real" key)
- KTime 1 = source frame `start_frame` (the first real sample)
- KTime 2 = source frame `start_frame + 1`
- ...
- KTime N = source frame `start_frame + N - 1` (the last real sample)

260420-uzv used `frame_offset = start_frame`, which was correct under the *assumption* that KTime 0 == source `start_frame`. That assumption was wrong. After 260421-bhg clipped the tail, the hidden +1 shift became visible:

- User sees Blender frame 1003 showing the pose they expect at source frame 1002 (Xpos=1552 at 1003 instead of 1002 — matched to the user's verbatim UAT report).
- The KTime 0 pre-roll lands on Blender frame 1001 (the user's desired first frame), so the viewer sees a duplicate / static-hold frame at 1001 that isn't a real sample.
- The KTime N real sample for source frame 1100 gets dropped by `frame_end=1100` because it's landing on 1101.

Fix: make `frame_offset = start_frame - 1` (so KTime 1 -> 1001, KTime 2 -> 1002, ..., KTime 100 -> 1100 — the real samples now land on their correct plate frames), AND add a symmetric `frame_start` clip (so KTime 0 -> 1000 is dropped before emit — the pre-roll is removed).

Output: A `frame_start: Optional[int] = None` kwarg on `fbx_to_v5_json` (threaded to `_merge_curves`) that, when non-None, drops any frame whose POST-OFFSET value is STRICTLY less than `frame_start`. The Flame-side handler modifies the existing `start_frame` defensive read in place: one read drives both `frame_offset = int(float(sf)) - 1` (shifted by -1 to account for KTime 0 being pre-roll) AND `frame_start = int(float(sf))` (the INCLUSIVE lower bound — drop the now-offset-shifted KTime 0). Four new tests cover the UAT scenario, the None-default regression, the inclusive-boundary case, and the empty-result edge.

Post-fix behavior for a 1001..1100 batch:
- KTime 0 -> offset-adjusted frame 1000 -> DROPPED by `frame_start=1001` (pre-roll eliminated)
- KTime 1 -> offset-adjusted frame 1001 -> KEPT (source 1001's real pose)
- KTime 2 -> offset-adjusted frame 1002 -> KEPT (source 1002's real pose — Xpos=1552 now on the expected frame)
- ...
- KTime 100 -> offset-adjusted frame 1100 -> KEPT (source 1100's real pose — previously DROPPED, now preserved)
- Output: exactly 100 frames spanning 1001..1100.

`frame_start` completes the symmetric pair with `frame_end` — both are INCLUSIVE bounds on the offset-adjusted frame number. The drop conditions `frame < frame_start` and `frame > frame_end` are mirror images. Together they define a closed range `[frame_start, frame_end]` in post-offset plate-frame space.

Non-goals (per user, explicit):
- Do NOT change `bake_animation=True`. The 101-KTime emit (including the pre-roll) is Flame's baseline behavior; the fix is to interpret and clip it correctly, not to change what Flame bakes.
- Do NOT modify `tools/blender/bake_camera.py`. It already derives scene range from JSON via `min/max` of `frames[*].frame`; remove the pre-roll in the v5 JSON and the Blender scene tightens automatically.
- Do NOT include `./install.sh` deploy as a task. User owns deployment; plan ends at "tests pass".
- Do NOT rewire `frame_end` — that landed in 260421-bhg and is out of scope here. Build on top of it.
- Do NOT duplicate the defensive try/except for `start_frame`. The existing block already reads `flame.batch.start_frame.get_value()`; modify that block in place so one read drives both `frame_offset` and `frame_start`.

Semantic decision (locked): `frame_start` is INCLUSIVE. For a batch of 1001..1100 the output must contain frame 1001 (i.e., the boundary frame is KEPT). The drop condition is `frame < frame_start`, NOT `frame <= frame_start`. This is the exact symmetric counterpart of 260421-bhg's `frame > frame_end` (not `>=`). The clip applies AFTER `frame_offset` is added, so the comparison is against the user-facing (plate) frame number, not the raw FBX KTime.
