## Verdict: **PARTIAL — DIDN'T-PARK (no parking API found)**

There is no PyClip-level "park on frame N" API in Flame 2026.2.1. The
`TODO(next-spike)` in 260430-e5y assumed one would exist; **it does not.**

The export itself IS per-frame faithful — a full multi-frame export produces
N distinct PNGs, one per frame. But to extract a single arbitrary frame from
a multi-frame clip, the recipe must take one of two routes (neither is the
clean "park then export" the original recipe sketched):

**Route A — Export full clip, pick the right file (deterministic, slow):**
- Trade: ~0.36s/frame at 4K-16bit; for a 49-frame plate that's ~17.5s, then
  the recipe picks `output_dir/<clip>.<TC + offset>.png` by frame index.
- Pro: zero clip mutation; pure-read-only.
- Con: 17.5s for a single frame is ~40x slower than wiretap_rw_frame's
  ~1.5s/frame — net regression for the calibrator hot-path.

**Route B — Mark-bracketed export (mark_in/mark_out + restore):**
- Trade: set `seg.record_in == seg.record_out`, `e.export_between_marks = True`,
  export, restore originals — fast (<1s) but mutates clip edit-data.
- Pro: single-frame extraction at ~0.5s.
- Con: violates the spirit of R8 (persistent edit attributes on the
  production clip). If export crashes between set + restore, clip is left
  with corrupt marks.

**Route C — Throwaway-clone-and-mark (untested this spike):**
- Approach: `flame.duplicate(pc)` (or PyClipNode.duplicate()) into a throwaway
  batch, set marks on the clone, export, delete clone.
- Pro: zero mutation on production clip; fast per-frame (~0.5s).
- Con: requires creating a throwaway PyClipNode → violates R8's "no node
  creation" against gen_0460. Would need its own throwaway batch group
  (`spike_*` style) to be test-driven, and integration into the calibrator
  hot-path needs a follow-up spike.

**The original recipe's `TODO(next-spike): park clip on target_frame before
export` cannot be satisfied with a single-line API call.** Implementation
planner should pick Route A (slow but safe), Route B (fast but mutating), or
Route C (fast + safe but requires clone management) and treat that as a
real plan task, not a TODO.

## Implementation-blocker note for the next planner

> **Frame parking is NOT a one-line PyClip attribute.** PyClip exposes no
> `current_time`/`go_to_frame`/setter API in Flame 2026.2.1 — verified
> empirically against `gen_0460_graded_L01` (4448x3096, 16-bit, 49 frames).
> The implementation plan must explicitly pick Route A (full-export +
> filename pick), Route B (mark-bracket + restore), or Route C (throwaway
> clone) and budget the work. The 25-LOC recipe from 260430-e5y is therefore
> **incomplete for multi-frame clips** — the `target_frame` parameter is
> currently ignored. For Camera Match's typical use (single chosen reference
> frame from a multi-frame plate), this matters.
>
> **Performance ceiling:** Full-clip export at 4K-16bit is ~0.36s/frame
> amortized → ~17.5s for a 49-frame plate. Compare to wiretap_rw_frame's
> ~1.5s for a single 4K-32bit MXF frame. Route A is a 10x+ regression on
> the calibrator hot-path; Routes B/C should be preferred even with their
> implementation cost.
>
> **Crash signal:** Today's session had 3 Flame crashes total (2 before this
> spike; 1 during Phase 4 segment introspection). Bridge calls that touch
> segment time-data (`record_in`, `source_in` PyTime conversion) on
> high-resolution production clips appear to be a SIGSEGV trigger. Future
> probes should test on a smaller throwaway clip (`spike_*` batch with a
> tiny imported PNG) before touching production data.
