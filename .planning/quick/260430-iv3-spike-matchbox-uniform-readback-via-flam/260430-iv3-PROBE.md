## Probe 1 — Channel Introspection on Un-animated Matchbox Uniform

### Question
When a CameraMatch matchbox is in a batch and its `vp1_l1_start` uniform is set to a known non-default value via the matchbox UI (e.g., x=0.42, y=0.69), is that value reachable as a Flame Channel via the matchbox PyNode wrapper, the batch graph, or any flame.* module-level accessor?

### Approach
- (Pre-step, user keyboard) User drops a CameraMatch matchbox into `spike_260430_iv3`, opens its UI, sets `VP1 Lines → Line 1 → L1 Start` to (0.42, 0.69) by typing the values. User confirms ready.
- (Bridge /exec, single call) Resolve `mb` from `flame.batch.nodes` by matching `name == "Camera Match"`. Probe candidate channel containers and module-level surfaces; aggregate findings into one dict.

### Result

**Bridge call returned cleanly (no crash).** Resolved `mb` as `('CameraMatch', 'PyNode')` (NOTE: node name is `CameraMatch` single word, not "Camera Match" — plan's resolver string was corrected at probe time).

Key facts from the response (full envelope in transcript):

- **`dir_filtered`** (channel/anim/curve/expr/link/param/uniform/input/attr filter applied to `dir(mb)`):
  `['__delattr__', '__getattr__', '__getattribute__', '__setattr__', 'attributes', 'input_sockets']`
  → only `attributes` and `input_sockets` are non-dunder. The permissive `__getattr__` is what makes everything else return None silently.
- **All 11 candidate channel containers return `NoneType`:** `channels`, `animation_data`, `animation`, `anim_data`, `channel`, `anim`, `animations`, `parameters`, `params`, `uniforms`, `inputs` — every one is `is_none: True`. This is not "attribute missing" (the permissive `__getattr__` would still return None for missing names) — but the consistent `NoneType` across all candidates means there is no live channel container on this PyNode at all in the un-animated state.
- **`mb.attributes` is a `list`** (not a dict / not a channel registry). Its keys are schematic-node metadata only: `['pos_x', 'pos_y', 'name', 'collapsed', 'note', 'note_collapsed', 'selected', 'type', 'resolution', 'schematic_colour', 'schematic_colour_label', 'bypass', 'shader_name', 'resolution_mode', 'scaling_presets_value', 'adaptive_mode']` — none of our shader uniforms (`vp1_l1_start`, etc.) appear. This corroborates the 260430-ddi Path B finding.
- **`flame.<chan|anim>*` and `flame.batch.<chan|anim>*` are both empty lists.** No module-level or batch-level channel registry exists at this moment.
- **One new lead:** `input_sockets` surfaced in `dir_filtered` and was NOT in this probe's candidate list. Worth re-probing at Probe 2 with the post-Animate state.

### Verdict

**NOT-FOUND** — un-animated matchbox uniforms have no channel-shaped readback surface on the PyNode wrapper, on the batch graph, or at module level.

Per plan stop rules: NOT-FOUND → continue to Checkpoint B (do not skip — Probe 2 may reveal channels that materialize only after Animate).

## Probe 2 — Channel Materialization After Animate

### Question
If the user right-clicks `vp1_l1_start` in the matchbox UI → Animate (sets one keyframe at the current frame), does the channel now materialize on the node? If so, can `get_value()` / `get_value(time)` recover the scalar value?

### Approach
- (Pre-step, user keyboard) User right-clicks `VP1 Lines → Line 1 → L1 Start` in the matchbox panel and selects Animate (or sets a keyframe). User confirms ready.
- (Bridge /exec, single call) Re-run Probe 1 introspection PLUS attempt `get_value()` on any newly-materialized channel; capture exact channel name format.

### Result

**Bridge call returned cleanly (no crash).** Re-introspected the matchbox PyNode after Animate.

- **All 11 channel-shaped accessors STILL return `NoneType`** — `channels`, `animation_data`, `animation`, `anim_data`, `channel`, `anim`, `animations`, `parameters`, `params`, `uniforms`, `inputs`. Animate did not change a single thing visible at the PyNode wrapper level.
- **`input_sockets` / `output_sockets` / `sockets`** exist as `list/list/dict` but are **schematic-node sockets**, not shader uniforms. `input_sockets` items are plain Python `str` (`['Input 1']` — the matchbox's "Front" texture-input renamed to socket label "Input 1"). `first_item_type: str`. Dead lead.
- **`mb.attributes` unchanged** — same 16 schematic-node metadata items. No items match `vp/l1/l2/start/end/axis` filter.
- **`get_attribute()` is not a method on PyNode** — all 6 name-format attempts (`vp1_l1_start`, `VP1_L1_Start`, `L1 Start`, `L1Start`, `vp1_l1_start_x`, `vp1_l1_start.x`) failed with `TypeError: 'NoneType' object is not callable` (permissive `__getattr__` returned None, then we tried to call None).
- **No sibling Channel-bearing node was created by Animate.** Batch graph is unchanged: `[('testImage', 'PyClipNode'), ('CameraMatch', 'PyNode')]`.
- **NEW LEAD: `flame.batch.mimic_link`** appeared post-Animate (was not present in Probe 1's `flame_batch_chan: []`). This is Flame's "Mimic Link" channel-linking surface — adjacent to the expression-link path that Probe 3 will test. Pulled forward into Probe 3.

### Verdict

**DEAD** — animated matchbox uniforms are NOT readable via the matchbox PyNode wrapper, the schematic socket lists, or any per-attribute getter. The `Animate` operation has zero effect on the surface area we can introspect from Python on the matchbox node itself.

This kills the "matchbox uniforms become Channels on Animate" hypothesis. The shader uniform values clearly live somewhere — Flame ships them to the GPU each frame — but that "somewhere" is not on the matchbox PyNode wrapper.

Per plan stop rules: DEAD → continue to Checkpoint C → Probe 3 (expression-link picker symmetry, with the `mimic_link` lead pulled forward).

## Probe 3 — Expression-Link Picker Symmetry

### Question
Does Flame's expression-link mechanism — when an Action axis channel is being linked to another node's channel — expose the matchbox uniform's channel-path string? This is read-only inspection: no link is created.

### Approach
- (Pre-step, user keyboard) Ensure `spike_260430_iv3` has both the matchbox node AND a sibling Action node.
- (Bridge /exec, single call) Inspect Action's axis channels, module-level link/target/source/expr surfaces, and per-node publish surfaces.

### Result

**SKIPPED.** No /exec call was made. R6 honored — no state mutation. R1 honored — no probe issued.

**Why skipped:** Probes 1 and 2 conclusively established that matchbox shader uniforms are unreachable via the matchbox PyNode wrapper, schematic socket lists, or any per-attribute getter — before AND after Animate. The lone post-Animate lead (`flame.batch.mimic_link`) was the last 1% argument for running this probe: maybe the link/mimic mechanism would enumerate matchbox uniforms as available targets and reveal the addressing scheme.

User confirmed (domain knowledge): **Mimic Link is type-restricted — only links between equal node types.** An Action axis cannot mimic_link to a matchbox uniform regardless of how the channel registry is structured. The link picker therefore cannot enumerate matchbox uniforms as Action-axis targets, and `mimic_link` itself is dead for cross-type linking.

This is consistent with prior probes' finding that `flame.batch.chan*` was empty pre-Animate and only added `mimic_link` post-Animate — link targets ARE the channel registry, the registry has no matchbox-uniform entries, and the lone added surface is type-incompatible with the use case.

### Verdict

**SKIPPED — provably dead** (not "untested"). Even the optimistic interpretation requires (a) a registry that prior probes confirmed is empty, plus (b) a cross-type link mechanism that the user confirmed Flame does not provide.

## Probe 4 — Sidecar Comment-Node JSON Round-Trip (FALLBACK ONLY)

**Originally: RUN ONLY IF Probes 1, 2, 3 are all DEAD.** Re-evaluated at synthesis; SKIPPED — see below.

### Question
Can a Comment node next to the matchbox carry a JSON blob of VP values that the headless calibrator reads via PyAttribute?

### Approach
- (Bridge /exec, single call, R6 throwaway) Create a Comment-shaped node, find its content attribute, set/get a sentinel JSON string, byte-compare.

### Result

**SKIPPED.** No /exec call was made. R6 honored — no state mutation. R1 honored — no probe issued.

**Why skipped:** the READBACK side of the Comment-node sidecar would almost certainly work (text attributes are basic PyAttribute surfaces). But during synthesis, the architecture hit a deeper kill: **there is no plausible WRITE path**. For a Comment-node sidecar to be useful, something has to populate it with the current VP values whenever the artist edits the matchbox UI. The candidates are:

1. **Artist manually populates it** — terrible UX; defeats the entire point of the matchbox UI.
2. **The matchbox shader writes to it** — impossible. GLSL can render pixels; it cannot mutate Flame's batch graph or Python state.
3. **A right-click hook that runs at solve-time** — but the hook's job is to READ the values; if it can't read them off the matchbox in the first place (Probes 1-2), it can't write them to a sidecar either.

The Comment-node round-trip itself is trivially testable, but proving it works does not unblock anything. The bottleneck is upstream: **no Python read path exists for matchbox uniforms**, period — and that breaks the sidecar's writer side, not its reader side.

### Verdict

**SKIPPED — fallback collapsed.** The Comment-node sidecar is not a viable architecture because the matchbox cannot programmatically write to it. The viable architectural pivot identified during synthesis is pixel-encoded inputs + snapshot decode (see Recommendation Matrix and Next Planning Step below).

**Pivot to pixel-encoded inputs + snapshot decode.** Spike-then-plan, with the spike kept narrow.

The next planner's first task should be a **pixel-encoding feasibility spike** (~45-60 min) that:

1. **Modifies a copy of `matchbox/CameraMatch.1.glsl`** to encode the 8 VP `vec2` uniforms (vp1_l1_start, vp1_l1_end, vp1_l2_start, vp1_l2_end, vp2_l1_start, vp2_l1_end, vp2_l2_start, vp2_l2_end) into a deterministic pixel region — e.g., the top-left 4×4 pixels at full intensity, one component per pixel, with a sentinel "magic value" pixel for round-trip verification.
2. **Snapshots the matchbox output** via `flame.PyExporter().export()` to a temp PNG using the on-disk preset path that 260430-ddi flagged as needing concrete location work.
3. **Reads the PNG back in Python** (numpy / PIL — already available in the forge env) and decodes the encoded pixels.
4. **Verifies round-trip fidelity** — the decoded values match what the shader received as uniforms (i.e., the artist-visible UI state) within a known tolerance (PNG-8bit gives ~1/255 ≈ 0.004 precision; if VP coordinates need more precision, escalate to a 16-bit preset).

**Two known unknowns the spike must resolve before a real plan:**

- **Preset path resolution.** 260430-ddi's bridge crashed on `PyExporter().get_presets_base_dir(PresetVisibility.Autodesk)`. The new spike must locate the on-disk PNG-8bit Image_Sequence preset path manually (probably under `/opt/Autodesk/presets/2026.2.1/export/...`), since `get_presets_base_dir` cannot be called from /exec safely.
- **Bit depth + colour-space round-trip.** PNG-8bit raw colour space output (BakeViewTransform=false) was identified by 260430-ddi as the recommended snapshot config. The new spike must verify the encoded uniform values survive the GLSL → framebuffer → PNG → numpy round-trip without a colour management transform mangling them. If 8-bit is too lossy for VP precision, fall back to a higher-bit-depth Image_Sequence preset (TIFF-16, EXR — verify availability).

**Fallback if the pixel-encoding spike surfaces a blocker:** Keep the existing PySide2 line-drawing UI (v5.x architecture). The matchbox can stay as a *display-only* visual feedback layer alongside it (uniforms set programmatically by the PySide2 dialog, no readback needed), but the artist's primary interaction returns to Qt. This loses the "lives in batch graph" advantage but is proven to work.

**Out of scope for the next planner:** Re-running any of Probes 1-4 from this spike. Their dead-ends are documented with primary-source evidence.

## Memory Crumb Candidates

Two findings from this spike are durable enough to warrant memory crumbs (per `CLAUDE.md` memory-crumb discipline — "would I want this if I forgot it" bar). Final crumb authoring is deferred to the user.

1. **`flame_matchbox_uniforms_not_python_readable.md`** — Matchbox shader uniforms are not Python-readable in Flame 2026.2.1. Tested at PyNode level (`channels`, `animation_data`, `parameters`, `uniforms`, `inputs`, `attributes`, `input_sockets`, `output_sockets`, `sockets` — all NoneType or schematic-only), at module level (`flame.<chan|anim>*` empty), and at batch level (`flame.batch.<chan|anim>*` empty pre-Animate; only `mimic_link` post-Animate, which is type-restricted). Animate gesture has zero observable effect on the PyNode surface. The values clearly live in Flame's GPU pipeline state and never surface as enumerable Python objects. **Implication:** any architecture that needs Python to read a matchbox uniform value must use an out-of-band transport (e.g., pixel encoding into the rendered output, then snapshot+decode). This builds on but is more emphatic than `flame_matchbox_pynode_no_uniform_api.md` (the 260430-ddi crumb candidate, never written) — that crumb covered the PyNode wrapper specifically; this one covers ALL Python surfaces.

2. **`flame_mimic_link_type_restricted.md`** — Flame's `mimic_link` channel-linking mechanism only links between equal node types (e.g., Action→Action, Camera→Camera). An Action axis cannot mimic_link to a matchbox uniform. Surfaces post-Animate as `flame.batch.mimic_link`. Combined with the empty `flame.batch.<chan|anim>*` registries, this kills the "discover channel paths via the link picker" inspection pattern for cross-type targets. User-supplied domain knowledge — not derivable from the bridge introspection alone.
